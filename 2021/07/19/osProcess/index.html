2.1 进程与线程

2.1.1 进程

2.1.1.1 进程模型

  1. 多道程序设计：允许多个程序同时进入内存并运行（目的是提高系统效率）。

  2. 并发环境：①一段时间间隔内由多个程序运行；②看上去同时；③次序不是事先确定的。

  3. 多道程序设计模型：利用多进程来提高CPU利用率

  4. CPU的利用率=$\frac{CPU运行时间}{总时间}$.

  5. CPU利用率评价公式：$\rho=\frac{等待I/0操作的时间}{停留在内存中的时间}$。当内存中同时有n个进程时，则所有n个进程都在等待I/0的概率为$\rho^n$，此时CPU利用率为$1-\rho^n$。

2.1.1.2 进程的状态

  1. 进程的状态：运行态、就绪态、阻塞态。

  2. （1：OS发现进程不能继续运行如等待I/O；2：操作系统调度；3：操作系统调度；4：当进程等待的一个事件发生时如一些输入到达）

2.1.1.3 进程控制块

  1. PCB（进程控制块/进程描述符/进程属性）：OS用于管理控制进程的数据结构，记录进程属性、描述动态变化过程；是系统感知进程存在的唯一标志（进程与PCB一一对应）。
  2. 进程表：所有进程PCB的集合；一般由链表实现（所有进程、就绪态进程…）
  3. PCB包含的信息：
    • 进程标识符
    • 进程名
    • 用户标识符
    • 进程组关系
    • CPU现场信息：寄存器值（通用寄存器、程序计数器、程序状态字、栈指针）、指向该进程页表的指针
    • 所拥有的资源和使用情况：虚拟地址空间的状况、打开文件列表
    • 进程控制信息：当前状态、优先级、代码执行入口地址、运行统计信息（执行时间、页面调度）、进程间同步与通信、进程的队列指针、进程的消息队列指针

2.1.1.4 进程的控制

进程的控制由特定功能的原语（完成某种特定功能的一段程序，具有不可分割性和不可中断性，源于的执行必须是连续的，在执行过程中不允许被中断）完成以防止出现不可预知的问题。

  1. 进程的创建（驱动：系统初始化、正在运行的程序执行了创建进程的系统调用、用户请求创建一个新进程、一个批处理作业的初始化）

    • 分配标识和PCB
    • 分配地址空间
    • 初始化PCB
    • 设置PCB默认值
    • 把新进程添加到就绪队列（PCB添加到就绪队列链表）

    [例]UNIX : fork()系统调用

    • 为子进程分配一个空闲的进程描述符
    • 分配给子进程唯一标识 P ID
    • 复制父进程地址空间
    • 从父进程继承共享资源如打开文件和当前工作目录
    • 将子进程设为就绪态插入到就绪队列
    • 对子进程返回标识符0
    • 向父进程返回子进程的P ID

    UNIX : execve()

    • 对当前进程进行替换，替换者为一个制定的程序（参数：文件名称、参数列表、环境变量）
    • 系统把代码段替换成新的程序的代码
    • 废弃原有的数据段和堆栈段并为新程序分配新的数据段和堆栈段
    • 仅保留原有进程号

    WINDOWS下进程创建：CreateProcess()

  2. 进程的层次结构

    • Linux：进程树、子进程继承父进程很多内容、父进程关闭导致子进程ppid的改变、最初进程为init
    • WINDOWS：更加扁平化、父进程关闭不影响子进程、句柄

2.1.2 线程

线程是一种轻量级的进程，一个进程拥有一个或多个线程，这些线程拥有共享的资源如地址空间和文件等，线程具有自己的堆栈和寄存器环境以保证独立运行的最小要求。

2.1.2.1 引入线程的原因

  1. 应用的需要：仅具备进程的情况下用户体验较差，卡顿。
  2. 开销的考虑：进程操作时间/空间开销过大，限制了并发度的提高；而创建/撤销一个新线程和切换线程花费时间少，同一进程内的线程之间相互通信无需调用内核。
  3. 性能的考量：线程提高了并发的粒度以适应同一进程内处理多个并行任务的需求，从进程并发到线程并发；适应多核CPU。

2.1.2.2 线程模型

线程是操作系统能够运行调度的最小单位，含于进程之中，是进程中的实际运作单位；而进程是系统分配资源的基本单位，是资源拥有者。

2.1.2.3 线程机制的基本实现

  1. 用户空间中实现线程

    1. 实现方式

      • 在用户空间建立线程库用来提供一组管理线程的过程

      • 在用户空间实现运行时系统用来完成线程的管理工作

      • 内核管理进程，线程对内核透明

      • 线程切换不需要内核态的特权

      • 一个内核PCB对应多个线程（1 : N）

    2. 优点

      • 线程切换快
      • 调度算法是应用程序特定的
      • 用户级线程可以运行在任何实现线程库的操作系统上

    3. 缺点

      • 大多数系统调用是阻塞的。因此，由于内核阻塞进程从而进程中所有线程也被阻塞

  2. 内核空间中实现线程

    1. 实现方式

      • 内核管理所有线程，并且向应用程序提供API接口
      • 内核维护进程和线程的上下文
      • 线程切换需要内核支持
      • 以线程为基本单位进程调度
      • 一个控制单位BLOCK对应一个线程（1 : 1）

  3. 混合实现（N : M）

  4. ==调度程序激活技术==

    • 目的：改进内核级线程的速度；避免在用户空间和内核空间之间不必要的转换
    • 需要一个运行时系统：内核空间可以看到这个运行时系统并在需要时控制之（上行调用），运行时系统的效率更高

  5. ==弹出式线程==

    • 一个线程阻塞等待消息->消息到来之后再创建线程

2.2 调度

2.2.1 调度简介

  1. 进程行为：计算密集型、I/O密集型。由于I/O密集型进程数量的增加导致研究方向从提高CPU利用率到提高I/O利用率。

  2. 调度时机

    1. 非抢占式：进程可以一直执行直到进程结束或者发生阻塞。
    2. 抢占式：进程可以被操作系统打断；每个进程不能一直占用太长时间CPU，如有高优先级进程就绪也可以抢占CPU。

  3. 分类

    1. 批处理
    2. 交互式
    3. 实时

  4. 进程切换：一个进程让出处理器并由另一个进程占用处理器的过程。进程切换的频率要恰当。

    • 切换全局页目录以加载一个新的地址空间
    • 切换内核栈和上下文
      • 保存进程A的上下文环境（程序计数器、程序状态字）
      • 用新状态和其他相关信息更新过程A的PCB
      • 把进程A移动到合适的队列
      • 把进程B的状态设置为运行态
      • 从进程B的PCB中恢复上下文环境

  5. 目标

    1. 所有系统
      • 公平——给每个相似进程公平的CPU份额
      • 策略强制执行——保证规定的策略被执行人
      • 平衡——保持系统的所有部分都忙碌
    2. 批处理系统
      • 吞吐量——每小时最大作业数
      • 周转时间——从提交到终止间的平均时间
    3. 交互式系统
      • 响应时间——快速响应请求
      • 均衡性——满足用户期望
    4. 实时系统
      • 满足截止时间——避免丢失数据、出现事故
      • 可预测性——在多媒体系统中避免品质降低

2.2.2 批处理系统中的调度——吞吐量、周转时间

2.2.2.1 先来先服务(FCFS First Come First Serve) 非抢占式

  1. 算法：按照进程创建的次序运行
  2. 优点：最简单
  3. 缺点：长进程后面的短进程需要等待很长时间，不利于用户体验

2.2.2.2 最短作业优先(SJF Shortest Job First) 非抢占式

  1. 算法：（假设可以精确预测每个作业运行时间、作业之间相互独立、同时运行）选择用时最短的作业优先执行
  2. 优点：平均等待时间、平均周转时间最短
  3. 缺点：对长作业不利，会导致饥饿；估计时间不一定真的做到最短作业优先

2.2.2.3 最短剩余时间优先(SRTN Shortest Remaining Time Next) 抢占式

  1. 算法：当一个新就绪的进程比当前运行进程具有更短的完成时间时，系统抢占该进程，选择新就绪的进程执行，是最短作业优先的抢占式版本
  2. 优点：在所有进程同时可运行时，最短的平均周转时间
  3. 缺点：不公平，可能使长的任务长时间得不到运行产生饥饿现象；无法精确直到一个作业的运行时间；完全未考虑作业的紧迫程度因而不能保证紧迫性作业被及时处理

2.2.2.4 最高响应比优先(HRRN Highest Response Ratio Next)

  1. 算法：调度时，首先计算每个进程的响应比R。此后总是选择R最高的进程执行。

    $响应比R=\frac{周转时间}{处理时间}={处理时间+等待时间}{处理时间}=1+\frac{等待时间}{处理时间}$

  2. 优点：克服饥饿，兼顾长作业。（等待时间相同，处理时间越短的优先处理；处理时间相同，等待时间越长的优先处理。）

2.2.3 交互式系统中的调度——响应时间、均衡性

2.2.3.1 轮转调度(RR Round Robin)

  1. 算法：将所有就绪进程按先来先服务的原则排成一个队列，每次调度时把CPU分配给队首进程并令其执行一个时间片，当这个时间片结束时强迫这个进程让出处理器并排列到就绪队列的尾部等候下一次调度。

    ==时间片应略大于一次典型的交互所需要的时间==

  2. 优点：公平；有利于交互式计算，响应时间快

  3. 缺点：由于进程切换，时间片轮转算法花费较高开销（假设时间片 10ms，如果进程切换花费 0.1ms，CPU开销约占 1%）

2.2.3.2 最高优先级调度(HDF Highest Priority First)

  1. 算法：在进程控制块PCB中加入优先级属性，选择优先级最高的进程投入运行，优先级可以是静态不变的也可以是动态调整的。
  2. 优点：实现简单
  3. 缺点：不公平

[例]动态优先级调度

  • 每个进程的优先级设置为$1/f$

    • $f=\frac{历史占用CPU时间}{历史I/O阻塞时间+历史占用CPU时间}$
    • 如果存在时间片，$f$为上一次该进程时间片中CPU运行时间
    • $I/O$型进程优先级应该比较高

  • 优先级调度+时间片轮转

    • 把一组进程按照优先级分为若干类
    • 处于同一优先级类内部的进程按照时间片轮转方式调度
    • 不同优先级类的进程按照最高优先级调度

  • 特点

    • 对于优先级队列i，只有当高优先级的队列为空时，该队列的进程才有可能运行
    • 若该队列存在多个进程，这些进程以时间片轮转的方式平等运行
    • 高优先级进程会抢占当前进程

2.2.3.3 多级反馈队列(Multiple Feedback Queue)

  1. 算法：
    1. 设置多个就绪队列，第一级队列优先级最高（允许进程在队列间移动）
    2. 给不同就绪队列中的进程分配不同长度的时间片，第一级队列时间片最小；随着队列优先级的降低时间片增大
    3. 当第一级队列为空时在第二级队列调度，以此类推
    4. 各级队列按照时间片轮转的方式进行调度
    5. 当一个新创建进程就绪后，进入第一级队列
    6. 进程用完时间片而放弃CPU进入下一级队列
    7. ==由于阻塞而放弃CPU的进程进入相应的等待队列，一旦等待的时间发生该进程就返回原来一级就绪队列==
  2. 优点：
    • 终端型用户：终端型用户提交作业通常较小，系统只要能使这些进程在第一队列所规定的时间片内完成便可使终端型用户满意
    • 短进程用户：周转时间依然较短
    • 长进程用户：用户不必担心其进程长期得不到处理
  3. 缺点：
    • 对于需要长时间运行并且持续交互的进程不友好（进程出现交互即放入最高优先级；间歇敲击键盘可使进程优先级永远处于最高态）

2.2.3.4 最短进程优先(Shortest Process Next)

  1. 算法：选择运行时间最短的进程执行。

  2. 老化算法

    • 根据进程过去的行为进行推测

    • $aT_0+(1-a)T_1$

    • $T_0,T_0/2+T_1/2,T_0/4+T_1/4+T_2/2,T_0/8+T_1/8+T_2/4+T_3/2$

2.2.3.5 保证调度

算法：对用户作出明确的性能保证并实现

[例]

  • n个进程，每个进程使用CPU处理能力的1/n
  • 必须跟踪各个进程自创建以来已使用了多少CPU时间
  • 进程实际获得的CPU时间是已知的
  • 通过获得的CPU时间和应获得的CPU时间之比动态调整运行过程，保证每个进程平均使用CPU

2.2.3.6 彩票调度

  1. 算法：给每个进程发彩票，优先级越高则发的彩票越多，每隔一段时间举行一次彩票抽奖，抽出来的号是谁的，谁就能运行。
  2. 优点：不会出现某个进程一直等待的问题；轻量且高效
  3. 缺点：不能很好地适合I/O型进程；票数分配问题没有确定的解决方式

2.3 进程通信

目的：进程之间的同步工作，完成既定任务。
问题：

  • 一个进程如何把信息传递给另一个
  • 两个或更多的进程在关键活动中不会交叉
  • 进程顺序不能出错

处理进程之间的合作与竞争关系：

  • 竞争是合作的特殊情况
  • 合作又通过处理各种竞争问题来解决

互斥与原语：

  • 互斥：多个进程不能同时访问/执行同一段内容
  • 原语执行过程中不能被打断

2.3.1 竞争条件与临界区

  1. 竞争条件：两个或多个进程读取某些共享数据而最后结果取决于进程运行的精确时序（不一定出错但程序无法发现错误发生）

  2. 临界区：各个进程中对某个临界资源（共享变量）实时操作的程序片段

    临界资源：系统中某些一次只允许一个进程使用的资源

    互斥：各进程竞争使用临界资源

  3. 临界区的使用规则：

    1. 没有进程在临界区时，想进入临界区的进程可以进入
    2. 不允许两个进程同时处于临界区
    3. 临界区外运行的进程不得阻塞其他进程进入临界区
    4. 不得使进程无限期等待进入临界区

2.3.2 进程互斥

2.3.2.1 实现互斥的方案

1 屏蔽中断

算法：每个进程在刚刚进入临界区后立即屏蔽所有中断，并且在即将离开临界区前打开中断

原理：屏蔽中断后，时钟中断也被屏蔽，对中断时才会进行进程切换的情况有效

优点：简单

缺点：将屏蔽中断的权力交给了用户进程，不安全；只能屏蔽一个处理器的中断，不适用多处理器

适用范围：单核心、操作系统内核

2 锁变量

算法：定义一个全局变量，其初始值为0。当一个进程想进入临界区时首先测试该变量，若为0则该进程将其设置为1并进入临界区；若为1则该进程将等待直到其值变为0。

优点：直观

缺点：不可行，由于中断时机不定可能导致同时进入临界区的情况

核心思想：区分进入临界区的意图和优先权。

  • 意图：需要进入临界区但还没有真正进入
  • 优先权：出现冲突即多个进程都有进入临界区的意图时进入临界区的优先权

    • 只有进程自己可以修改自己的意图
    • 进程可以把优先权修改成自己优先，且只能在进入临界区之前修改

2.3.3.5 管程

  1. 定义：一个管程是一个由过程、变及数据结构等组成的一个集合，它们组成一个特殊的模块或软件包。

    注：可以理解为一个类；进程可在任何需要的时候调用管程中的过程，但不能在管程之外声明的过程中直接访问管程内的数据结构。

    • 管程内部的条件变量

    1. 互斥
      • 管程中的过程是互斥进入的，保证管程中数据结构的完整性

    2. 同步
      • 管程中设置条件变量以及等待/唤醒操作以解决同步问题

    2. signal(c)：被一个进程调用，以指出事件（断言）现在为真，从而可以唤醒等待该事件被满足的进程继续执行。

    1. 当一个进入管程的进程执行wait操作时应当释放管程的互斥权，

2.3.3.6 信号量与管程辨析

  1. 信号量和管程都能解决编程中的同步互斥问题。二者本质上互通，可以用信号量实现管程、也可以用管程实现信号量。
  2. 信号量本质上是可共享的资源数量；管程是一种抽象数据结构用来限制同一时刻只有一个线程进入临界区。
  3. 信号量是可以并发的，并发量取决于信号量中count的初始值；而管程内部同一时刻最多只能有一个线程执行

  6. 信号量的V操作如果唤醒其他线程，当前线程和被唤醒线程并发执行；对于管程的signal操作，要么当前线程继续执行，要么被唤醒线程继续执行，二者不可并发

2.4 死锁

2.4.1 基本概念

  1. 定义：一组进程中，每个进程都无限等待被该组进程中另一个进程所占有的资源，因而永远无法得资源，这种现象称为进程死锁，这一组进程称为死锁进程。（参与死锁的所有进程都在等待资源）
  2. 产生死锁的==必要条件==：
    1. 互斥使用（资源独占）：一个资源每次只能给一个进程使用
    2. 占有且等待（请求和保持；部分分配）：进程在申请新的资源的同时保持对原有资源的占有
    3. 不可抢占（不可剥夺）：资源申请者不能强行从资源占有者手中夺取资源，资源只能由占有者自愿释放
    4. 循环等待：形成一个等待环路

2.4.2 死锁建模

二元组G=(V,E)

V：结点的集合，分为P（进程）、R（资源）两部分

E：有向边的集合($P_i,R_j$)、($R_i,P_j$)

分配（占有）边：资源->进程

申请（请求）边：进程->资源

2.4.3 死锁定理

如果资源分配图中没有环路，则系统中没有死锁；如果图中存在环路，则系统中可能存在死锁。

如果每个资源类中只包含一个资源，则环路是死锁存在的充分必要条件。

2.4.4 处理死锁的策略

  1. 忽略：假装不存在死锁/鸵鸟算法/系统因为其他原因崩溃的频率远远大于死锁

  2. 检测并恢复

    1. 利用资源分配图：

      1. 首先利用资源分配图来检测死锁（资源分配图中的环路是死锁的必要条件/如果每个资源类中只包含一个资源，则环路是死锁存在的充分必要条件）
      2. 对于有环路且资源类存在多个资源的情况
        1. 找一个请求资源可以满足的进程结点，删除结点并去掉分配边
        2. 重复上述步骤，如果可以删除所有进程则不存在死锁

    2. 利用矩阵表示资源分配图

      A向量：可用的资源数（一个元素一种资源）

      R矩阵：请求的资源数（一行对应进程、一列对应资源）

      C矩阵：占用的资源数（一行对应进程、一列对应资源）

      检测步骤：

      1. 寻找一个没有标记的进程$P_i$，对它而言R矩阵的第i行小于或等于A
      2. 如果找到了这样的一个进程，那么将C矩阵的第i行加到A中，标记该进程，转到第(i)步
      3. 如果没有这样的进程，则算法终止

      算法结束时，所有没有标记过的进程都是死锁进程。

    3. 利用抢占恢复：挂起一些进程并剥夺其资源；需要人工干预；可行性往往取决于资源特性和具体应用

    4. 利用回滚恢复：周期性设置检查点（备份进程的各种状态放入一个文件中）；一旦检测到死锁则回滚到前一个检查点，在该检查点之后所做的工作都丢失；如果该死锁仍存在则继续回滚

    5. 通过杀死进程恢复：

      进程的选择：

      1. 环中一个进程：如果仍然死锁则继续选择
      2. 环外一个进程：应当正好持有环中某些进程所需资源
      3. 尽量找不会出现副作用的进程/有些进程杀死后重新运行还会引起死锁

  3. 死锁避免

    每次资源分配的时候西安进程判断，如果分配后存在死锁风险则不分配，反之则分配。

    死锁风险的判断：

    1. 寻找一个没有标记的进程$P_i$，对于它而言R矩阵的第i行向量小于或等于A
    2. 如果找到了这样一个进程，那么将C矩阵的第i行向量加到A中，标记该进程并转到第(a)步
    3. 如果没有这样的进程则算法终止

    对于R矩阵：

    1. 死锁检测：申请的资源数量
    2. 死锁避免：预计所需最大资源数

    银行家算法：对每一个请求进行检查，监察如果满足这一请求是否会达到安全状态。若是，就满足该请求；若不是，则推迟对这一请求的满足。

  4. 死锁预防

    1. 破坏“互斥使用”条件

      资源转换技术：把独占资源变为共享资源

      虚拟化：SPOOLing技术的引入

      • 解决不允许任何进程直接占有打印机的问题
      • 设计一个守护进程/线程负责管理打印机，进程需要打印时，将请求发给守护进程，由守护进程完成打印任务

    2. 破坏“占有且等待条件

      实现方案1：要求每个进程在运行前必须一次性申请要求的所有资源，且仅当该进程所需资源均可满足时才给予一次性分配（==资源利用率低/饥饿现象==）

      实现方案2：当一个进程请求资源时，先释放当前占用的所有资源，然后再尝试一次获得所需的全部资源

    3. 破坏“不可抢占”条件

      实现方案：当一个进程申请的资源被其他进程占用时，可以通过操作系统抢占这一资源（两进程优先级不同）

      局限性：适用于状态易于保存和恢复的资源

    4. 破坏“循环等待”条件

      实现方案：把系统中所有资源编号，进程在申请资源时必须严格按照资源编号的递增次序进行，否则操作系统不予分配（很难确定现实中的编号次序）

2.4.5 其他死锁相关问题

  1. 两阶段加锁：

    1. 适用：数据库等
    2. 实现
      1. 第一阶段试图把所需资源都加上锁
      2. 第一阶段成功，才开始第二阶段——完成更新并释放
      3. 如果不成功，则释放第一阶段的所有资源并延期重新开始

  2. 通信死锁

    1. 死锁不一定必须与资源相关
    2. 在通信系统中
      • 进程A向进程B发送请求信息，然后阻塞直至B回复
      • 假设请求信息丢失，则A将阻塞以等待回复
      • 而B会阻塞等待一个向其发送命令的请求，因而发生死锁
    3. 通信死锁不能通过对资源排序避免
    4. 解决方案：超时重发机制（没有回复一段时间后重新发送）

  3. 活锁：任务或者执行者没有被阻塞，由于某些条件没有被满足导致一直重复尝试——失败——尝试——失败的过程


    活锁与死锁的区分：死锁是多个进程占用了资源导致的而活锁不是。

  4. 饥饿：由于调度策略等原因导致某些进程一直得不到运行而不是由于资源被其他进程占用。

为了解决内存超载的问题。

2.内存紧缩

交换在内存中产生了多个空闲区，容易造成浪费，因此需要定期进行内存紧缩。

把所有的进程尽可能向下移动，将这些小的空闲区块合并成一个大块。但是需要耗费大量的CPU时间。

3.进程创建时分配内存大小

  • 若进程创建时大小是固定的且不在改变，操作系统只需准确的按其需要的大小进行分配。

  • 如果进程的数据段可以增长则会出现问题，如果相邻空闲区则可以直接使用，若相邻另一个进程则需要把增长的进程转移到内存中一个足够大的区域，如果交换区也满，则这个进程只有一直挂起直到一些空间空闲。

4.碎片——不能利用的内存空间

  • 内碎片：已经分配给作业但不能被利用的内存空间

  • 外碎片：还没有分配给作业但是由于太小而无法分配给申请内存空间的新进程的存储块

3.2.3 空闲内存管理

1 固定分区分配

1.实现方式

  • 将内存空间划分为若干固定大小的分区

  • 每个分区装入一道程序

  • 分区大小可以不等，但必须事先确定，在运行时不能改变

  • 当有空闲分区时，从后备队列选择一个适当大小的作业装入运行

2.分区说明表：用以记录可用于分配的分区号、分区的大小、分区的起始地址及状态（通常按照分区大小顺序排序）。

3.优点：是一种最简单的存储分配方式。

4.缺点：利用率较低，会产生内碎片。

2 位图

1.实现方式

  • 内存被划分为若干的分配单元

  • 每个分配单元对应位图中的一位

4 基于链表的空闲内存适配算法

1 首次适配

算法：存储器沿着链表进行搜索，直到找到一个足够大的空闲区，除非空闲区大小和要分配的空间大小正好一样，否则将该空闲区分为两部分，一部分供进程使用，另一部分形成新的空闲区。

优点：速度很快，尽可能少地搜索链表结点。

缺点：形成大量外碎片。

2 下次适配

算法：每次找到合适的空闲区都记录当时的位置，以便在下次寻找时从上次结束的地方开始搜索。

优点：外碎片略少于首次适配。

缺点：性能略低于首次适配算法。

3 最佳适配

算法：搜索整个链表，找出能够容纳进程的最小的空闲区。

优点：找到最接近实际需要的空闲区。

缺点：比首次适配算法慢；可能比首次适配和下次适配算法浪费更多的内存，产生大量无用的小空闲区。

5 最差适配

算法：总是分配最大的可用空闲区，使新的空闲区比较大从而可以继续使用。

缺点：浪费内存多

注：如果进程和空闲区使用不同的链表，则可以按照大小对空闲区链表进行排序，提高适配速度。空闲区链表按大小排序时，首次适配算法和最佳适配算法一样。

6 快速适配

算法：为那些常用大小的空闲区维护单独的链表。

优点：寻找一个指定大小的空闲区十分快速。

缺点：在一个进程终止或者被换出时，寻找它的相邻快并查看是否可以合并的过程非常费时；如果不合并，内存很快分裂出大量的进程无法利用的小空闲区。

7 伙伴系统

主要思想：将内存按2的幂次进行划分，组成若干空闲链表，查找链表找到满足进程的最佳匹配块。

算法：

  1. 将可用空间看作一块$2^u$
  2. 假设进程申请s
  3. 若$2^{u-1} \lt s \le 2^u$，则分配；否则将该块划分为两个$2^{u-1}$重复上述过程。
8 slab机制

主要思想：针对一些经常分配并释放的对象（如进程描述符，大小比较小，并且如果直接采用伙伴系统容易产生内碎片并且处理速度慢）。

slab分配器基于对象管理，将相同类型的对象归为一类。每当要申请这样一个对象，slab分配器就从一个slab列表中分配一个这样大小的单元出去，当要释放时，将其重新保存在该列表中，而不是直接返回伙伴系统，避免这些内碎片。

3.3 虚拟内存

3.3.0 前导

1 背景

交换技术很难适用于程序大于内存的情况并且可能发生一个进程占用一整块内存的情况。

2 覆盖技术的实现

  • 把程序分割成许多片段，称为覆盖。
  • 程序开始执行时，将覆盖管理模块装入内存，该覆盖管理模块立即装入并运行覆盖0
  • 执行完成后，覆盖0通知管理模块装入覆盖1（如果有空间就占用覆盖0上方的位置；如果没有空间就占用覆盖0的位置）

3 存在问题

程序员必须自行分割程序，费时且容易出错。

3.3.1 分页

1 基本概念

0.由程序产生的地址称为虚拟地址，它们构成了一个虚拟地址空间。在没有虚拟内存的计算机上，系统直接将虚拟内存送到内存总线上，读写操作使用具有同样地址的物理内存字，而在使用虚拟内存的情况下，虚拟地址不是被直接送到内存总线上，而是被送到内存管理单元MMU，MMU把虚拟内存映射为物理内存地址。

1.用户进程地址空间被划分为大小相等的部分，称为页或页面，从0开始编号。

2.内存空间按照同样大小划分为大小相等的区域，称为页框，从0开始编号，也称为物理页面、页帧、内存块。

3.内存分配规则：以页为单位进行分配，并按进程需要的页数来分配；逻辑上相邻的页在物理上不一定相邻。

4.逻辑地址=页号+页内地址

5.划分是由系统自动完成、对用户透明的。

2 缺页

解决虚拟地址空间比内存地址空间大的问题：当程序访问了一个未被映射的页面，MMU（通过页表项中的在/不在位）注意到该页面没有被映射，于是CPU陷入到操作系统发生缺页中断/缺页错误。操作系统找到一个很少使用的页框且把它的内容写入磁盘（如果它不在磁盘上）。随后把需要访问的页面读到刚才回收的页框中，修改映射关系，然后重新启动引起陷阱的指令。

页号作为页表的索引以得出对应于该虚拟页面的页框号。如果在/不在位为0，则引起一次操作系统陷阱。如果为1，则在该页表中查到页框号复制到输出寄存器的高三位中，再加上虚拟地址中的低12位偏移量就构成了15位的物理地址。输出寄存器的内容随即被作为物理地址送到内存总线。

3.3.2 页表

1 基本概念

1.页表是一张表，负责实现虚拟地址到物理地址的映射，每个进程一个页表存放在内存。

2.虚拟地址=虚拟号+偏移量

3.虚拟号可以作为页表的索引，索引到具体的页表项。

2 页表项的结构

页表项：页表项的结构与机器密切相关，但不同机器的页表项存储的信息都大致相同。

  • 页框号：；页映射的目的。
  • 在/不在位：决定缺页中断
  • 保护位：决定读写
  • 修改位：记录当前页面是否被修改/决定换回硬盘是否需要被写会
  • 访问位：记录当前页面是否正在访问

注：(1)利用页表的查找和地址转换是硬件完成的。(2)页面在磁盘中的地址不在页表项中（因为缺页中断是软件完成的事情，而页表是硬件自动访问的）

3.3.3 加速分页过程

0 前导&最简单的实现

虚拟地址到物理地址的映射必须非常快；如果虚拟地址空间很大，页表也会很大。

实现方式1：使用一个快速页表寄存器阵列，把页表放入寄存器中。启动一个进程的时候，页表装入此寄存器阵列中。

  • 优点：页表查找速度最快。

  • 缺点：(1)代价高昂。(2)页表要一次性全部读入。

实现方式2：页表全部在内存中。

  • 优点：代价相对低一些。
  • 缺点：页表查找速度相对较慢。

1 转换检测缓冲区（相联存储器/快表）

1.目的：为了加速页表查找（相当于寄存器和内存之间的cache）

2.程序访问的局部性原理：大多数程序总是对少量的页面进行多次的访问；只有很少的页表项会被反复读取，而其他的页表项很少被访问。

3.设计思路：设置一个小型的硬件设备，将虚拟地址直接映射到物理地址而不必在访问页表。通常直接将其放在MMU中作为相联存储器，按内容并行查找，是cache的一种。

4.特征：与页表项中的记录一一对应；多了一个虚拟页号（在页表项中由于索引就作为了虚拟页号所以不是必须的）；如果不在TLB中则直接查找页表并替换TLB中的一行。

5.软件TLB管理

方式演变：

  • 过去的计算机中TLB和页表的管理完全由处理器硬件负责（对软件透明）
  • 现代的精简指令集处理器可能不再完全由硬件处理。由此减少处理器和指令执行的复杂度，减少MMU的实现代价；此外，用软件来判断哪些页表项放入TLB可能更加灵活和有效（增加TLB的命中率）

可能出现的问题：

  • 软失效：页表项不在TLB中，需要更新TLB，不需要产生磁盘I/O。需要10-20个机器指令并花费几纳秒完成操作。
  • 硬失效：页面本身不在内存中，此刻需要一次磁盘存取以装入该页面。需要几毫秒，处理时间是软失效的百万倍。

硬失效在实际情况中由三种可能性：

  • 所需的页面可能就在内存中，但却未记录在进程的页表中。
  • 需要从硬盘中重新调入页面，发生缺页错误。
  • 程序访问了一个非法地址，无需向TLB中新增映射。

3.3.4 针对大内存的页表

0 前导

1.页表需要连续存放，每个进程都有自己的页表。

2.例题：

  • 假设一个页表项8B，进程虚拟地址空间为4GB内存，页大小为4KB，一个进程页表需要多大空间？答案：4GB/4KB*8B
  • 64位计算机中，假设进程地址空间为$2^{48}$，一个页表项8B，页大小为4KB，一个进程页表需要多大空间？答案：$2^{48}$B/4KB*8B

1 多级页表

优点：

  • 不需要线性存储
  • 只需要部分页表存在内存（或TLB）即可
  • 增加级数可以很好地处理更大内存的情况

缺点：

  • 访问页表的次数增加

增加页表的级数本质上是允许页表非线性存放而不能减少页表项数。

2 倒排页表

思路：

  • 每个页表项对应一个页框
  • 之前是每个页表项对应一个页

优点：

  • 当虚拟地址空间比物理内存大得多的时候可以节省大量的空间

缺点：

  • 查表速度慢

实现方式：从物理地址空间出发，系统建立一张页表，页表项记录进程i的虚拟地址（虚页号）与页框号的映射关系，整个系统一张表。系统的每个地址有一个三元组<process-id,page-number,offset>

根据process-id和page-number从page table找到对应的页框号，加上offset到物理内存找查找。

3.4 页面置换算法

3.4.0 前导

最佳算法->先进先出->第二次机会->时钟->最近未使用->最近最少使用->最不经常使用->老化算法->工作集->工作集时钟

<设计思想，算法实现，算法应用>

颠簸：虚拟内存中，页面在内存与磁盘之间频繁调度，是的调度页面所需的时间比进程实际运行的时间还多，导致系统效率急剧下降，这种现象称为颠簸或抖动。

3.4.1 最佳页面置换算法

设计思想：置换以后不再需要或者最远的将来才会用到的页面。

优点：效果最好。

缺点：由于操作系统无法知道各个页面下一次将在什么时候被访问，所以无法实现。

应用：作为一种标准来衡量其他算法的性能。

3.4.2 先进先出置换算法

设计思想：由操作系统维护一个所有当前在内存中的页面的链表，最新进入的页面放在表尾，最早进入的页面放在表头。当发生缺页中断时，淘汰表头的页面并把新调入的页面加到表尾。

优点：实现简单。

缺点：几乎没有考虑页面的使用频率以及程序访问的局部性原理。

3.4.3 第二次机会算法

设计思想：页表项新增一个访问位R，每次MMU访问该页表项时R置为1。按照先进先出算法选择某一页面，检查其访问位R，如果为0则置换该页面；如果为1将R置为0并放到链表尾部（若修改后不放回队尾，则修改后在首部的页面很容易被再次找到被置换，不公平）。

优点：考虑了使用频率和局部性原理。

缺点：经常要在链表中移动页面，既降低了效率又不是很有必要。

3.4.4 时钟算法

算法：把所有页面都保存在一个类似钟面的环形链表中，当发生缺页中断时，算法首先检查表针指向的页面，如果R位是0就淘汰该页面并把新的页面插入这个位置，把表针前移一个位置，如果R位是1就清除R位并前移一个位置，重复上述过程直到找到一个R位为0的页面位置。

3.4.5 改进型时钟算法

规定：

  • 最近未访问 最近未修改 （0,0）

  • 最近未访问 最近被修改 （0,1）

  • 最近被访问 最近未修改 （1,0）

  • 最近被访问 最近被修改 （1,1）

算法：

  • 从指针的当前位置开始，扫描循环链表。在这次扫描过程中，对访问位和修改位不做修改。选择遇到的第一个（0,0）页面用于替换。
  • 若上一步没有找到，则重新扫描。寻找（0,1）页面用于替换，在找的过程中将R置为0。
  • 如果上一步仍没有找到，再次执行前两步。

优点：考虑了页面载入内存后是否被修改的问题，减少了I/O次数。

缺点：增加扫描次数。

3.4.6 最近最少使用算法

设计思想：选择在最近一段时间未使用过的一页并置换（向前看；而最佳置换算法是向后看）。

优点：充分考虑了程序访问的局部性。

缺点：费时；难以实现；需要硬件支持。

1 实现方式1

要求硬件有一个64位计数器C，在每条指令执行完后自动加1，每个页表项必须有一个足够容纳这个计数器值的域。在每次访问内存后，将当前的C值保存到被访问页面的页表项中。一旦发生缺页中断，操作系统就检查所有页表项中计数器的值，找到值最小的一个页面（最近最少使用的页面）。

2 实现方式2

设置页表表项的两位——访问位R和修改位M，初始值为0，访问一个页面置1，修改一个页面置1，定期让所有的页面R置0。

发生缺页中断时，操作系统检查R、M。随机从编号最小的非空类中选择一页计算。

规定：

  • 第1类 无访问 无修改 （0,0）
  • 第2类 无访问 有修改 （0,1）
  • 第3类 有访问 无修改 （1,0）
  • 第4类 有访问 有修改 （1,1）

3.4.7 最不常用算法

算法：算法将每个页面与一个软件计数器相关联，计数器的初始值为0.每次时钟中断时由操作系统扫描页表中所有的页面，将每个页面的R位（0或1）加到它的计数器上。这个计数器大体上跟踪了所有页面被访问的频繁程度。发生缺页中断时，置换计数器值最小的页面。在时钟中断时由中断处理程序实现，是一种软件实现方式，计数器不用放在页表中。

缺点：最近经常使用的优先权小于之前经常使用的，而之前经常使用的由于一直换不出去导致对应的计数器值一直增大。

3.4.8 老化算法

算法：在最不常用算法基础上进行修改。首先，在R位被加进之前先将计数器右移一位；其次，将R位计数器加到最左端的位而不是最右端的位。

优点：最接近最近最少适用算法。

3.4.9 工作集算法

设计思想：根据程序的局部性原理，一般情况下，进程在一段时间内总是集中访问一些页面，这些页面被称为活跃页面。如果分配给一个进程的物理页面太少使该进程所需的活跃页面不能全部装入内存则进程在运行过程中将频繁发生中断。如果能为进程提供与活跃页面数相等的物理页面数则可减少缺页中断次数。

工作集：一个进程当前正在使用的页框集合。

$w(t,\Delta)$:t时刻该进程在过去$\Delta$个虚拟时间单位中访问到的页面的集合，以进程运行时间为准，不考虑挂起时间。

基本思路：找出一个不在工作集中的页面并置换之。

实现：

  • 每个页表项中有一个字段记录该页面最后一次被访问的时间，设置一个时间值T，根据一个页面的访问时间是否落在当前时间-T之前或之中决定其在工作集之内或之外。
  • 定期置零所有R位
  • 除了R和M，还有每个页表项要对应一个上次使用时间的域（不用放在也表中）
  • 当缺页中断发生时，扫描整个页表。在扫描时，检查R位，如果为1就把当前时间写进页表项的上次使用时间；如果R是0则表示在当前时钟周期中，该页面还没有被访问过，不更改表项对应的上次使用时间。计算生存时间判断是否处于工作集中，置换发现的第一个不在工作集中的页，发现之后继续扫描修改其他页。

特殊情况：

  • 所有页面都在工作集中就淘汰R=0且生存时间最长的页面。
  • 在最坏情况下，在当前时钟滴答下，所有的页面R=1，此时随机选择一个页面淘汰。

缺点：当缺页中断发生后需要扫描整个页表。

3.4.10 工作集时钟算法

设计思路：

  • 当找到一个R=0并且生存时间大于阈值的页时
    • 如果此页面被修改过，就不能立即申请页框，并调用I/O，将此页写回。
    • 指针继续往前走，算法继续对下一个页面进行操作。
    • 找一个R=0、生存时间大于阈值、没有被修改的页。
  • 如果指针经过一圈返回起始点：
    • 如果至少调度了一次写操作，说明指针移动过程中最终会有某个写操作完成，下次再到这个页可以直接被置换。
    • 如果没有调度过写操作，说明所有的页面都在工作集中，随机置换一个干净的页面来使用。

类比优先级(0,0)，(0,1)，(1,0)，(1,1)。

4.3 硬盘

  1. 概述：文件存储介质；非易失性，长期保存数据；含磁盘、固态硬盘、Flash等。

  2. 特点：存储容量大、数据传输率高、存储数据可长期保存。

  3. 结构特性

    任何时刻只有一个磁头处于活动状态，输入输出==数据流以位串的形式==出现

    物理地址形式为==磁头号（盘面号）、磁道号（柱面号）、扇区号==

    扇区标题10B、数据512B、ECC纠错信息12-16B

  4. 访盘请求

    参数：读/写，磁盘地址（设备号、盘面号、柱面号、扇区号），源/目内存地址

    过程：

    1. 寻道：磁头移动到指定磁道
    2. 旋转延迟：等待指定扇区从磁头下旋转经过
    3. 数据传输：数据在磁盘与内存之间的实际传输

  5. 固态硬盘（Solid State Disk，SSD）：基于闪存的存储技术；利用高电压擦去电容；无磁盘、无机械驱动器

    1. 一个SSD中有多个NAND Flash，一个NAND Flash中有多个块，一个块中有多个页。擦除一次一块，因此顺序更新比随机更新快，写入一次一页。

    2. 优点：速度快（平均比磁盘快3-6倍），没有寻道时间和旋转延迟；具有一定的抗冲击性；静音省电；体积小

    3. 缺点：存在写次数的极限，有一定的寿命；需要先擦除再写，擦除延时较长1ms，随机更新的速度较慢

    4. SSD 的over-provisioning和garbage-collection

      SSD的实际存储空间比可写入的空间大。

4.4.1 文件系统的实现

4.4.1.1 文件系统的布局

  1. 文件系统存放在硬盘上。多数硬盘划分为一个或多个分区，每个分区有一个独立的文件系统。

    磁盘的0号扇区/块称为主引导记录（Master Boot Record，MBR），用来引导计算机，在MBR的结尾是分区表。

    文件系统包含的项目；引导块：引导装载在该分区的操作系统（即便没有此分区操作系统也存在）；超级块：文件系统的关键参数如魔数magic number、块的数量等；空闲块的信息；i节点；目录；文件

4.4.1.2 文件的实现

  1. 连续分配：把每个文件作为一连串数据块存储在磁盘上。

    优点：实现简单；操作性能较好

    缺点：随时间推移产生碎片；更适合光盘（只写一次，后续都只读）

  2. 链表分配：为每个文件构造磁盘块链表

    优点：充分利用每个磁盘块，不会因为磁盘碎片而浪费存储空间；顺序读文件方便

    缺点：随机访问慢

  3. ==采用内存中的表进行链表分配==

    在链表的基础上，取出每个磁盘块的指针字并把它们放在内存的一个表上（文件分配表 File Allocation Table，FAT）。

    优点：解决了链表的不足，随机访问快。

    缺点：必须把整个表放在内存中；FAT的管理方式不能较好地扩展并应用于大型磁盘中。

  4. i节点：给每个文件赋予一个称为i节点的数据结构，包括文件属性和文件的磁盘块地址

  5. 只有打开的文件的i节点才在内存中。

    最后一个磁盘地址指向一个包含额外磁盘块地址的块的地址，可以由两个或更多个包含磁盘地址的块。

4.4.1.4 目录的实现

  1. 目录系统的功能：把文件名映射成定位文件数据所需的信息。未见属性可以放在目录里或文件的i节点中。

  2. 文件名的存储：限制长度/固定结尾符/文件名放在堆中

    加速查找：存储文件名的散列值

4.4.1.5 共享文件

  1. 共享文件的方式：链接

  2. 硬链接：直接共享iNode即一个iNode对应多个文件名。

    LINUX命令：link/ln

    特点：

    1. 文件有相同的iNode以及data block
    2. 不能交叉文件系统进行硬链接的创建
    3. 只能对文件创建，不能对目录进行创建
    4. 删除一个硬链接文件并不影响其他有相同iNode号的文件

  3. 软链接/符号链接：文件数据块中存放的内容是另一文件的路径名的指向

    LINUX命令：ln -s

    WINDOWS：快捷方式

    特点：

    1. 软链接文件有自己的文件属性及权限等
    2. 可对不存在的文件或目录创建软链接
    3. 软链接可交叉文件系统
    4. 软链接可对文件或目录创建
    5. 创建软连接时，链接计数器i_nlink不会增加
    6. 删除软链接并不影响被指向的文件，但若被指向的原文件被删除则相关软链接被称为死链接（dangling link，若被指向路径文件被重新创建，死链接可被恢复为正常的软链接）

4.4.1.6 日志结构文件系统（Log-structured File System：LFS）

  1. 基于硬盘的特性

    顺序I/O慢，随机I/O慢：随机I/O比连续I/O性能差很多，如果只用顺序读写就可以大幅度提高I/O效率。写操作占据了全部操作的大部分且零碎。

  2. 设计思路

    不更新任何数据，只提供append操作；建立缓冲区将多次操作放进一个段里一次性写入，从而有效利用顺序I/O

  3. 连续写

    创建一个段，把所有要写入硬盘的操作写入此段中（每次写入的操作都要加入对应i节点的信息）

    LFS：i节点会分散在各个段中，段最终会整体写入硬盘，也就是i节点分散在整个硬盘。iNode map用来顺序存放每个iNode的最新位置，每次写入都更新这个表。

    其他文件系统：i节点在固定位置存放，方便系统访问

  4. 垃圾回收

    不覆盖写任何数据，只是进行append操作，系统会存在同一个数据的很多版本。

    需要定期删除旧版本，从而会产生碎片，因此定期把几个段中的有效信息提取出来组成新的完整的段。

  5. 错误恢复

    设置检查点，若发生错误就从最近的一个检查点恢复。

4.4.1.7 日志文件系统

  1. 提高文件文件系统在出错时候的鲁棒性。

  2. 举例，对于一个文件删除操作，包含

    1. 在目录中删除文件

    2. 释放i节点，把i节点放入空闲i节点池

    3. 将所有磁盘块归还空闲磁盘块池

    日志文件系统先写一个日志项，列出上述三个将要完成的操作。

    将日志项写入磁盘，按照日志执行相应的操作。

    日志中的操作可以反复执行——幂等的。

    原子事务：一组动作可以被界定在开始事务和结束事务操作之间，要么一次性做完，要么什么也不做。

4.4.2 文件系统的管理和优化

4.4.2.1 磁盘空间管理

  1. 分块存储：把文件分为若干块，分别存放在磁盘不同/不连续的位置上

    连续存储：存在一段空间不足而整个文件需要移动的问题，同内存连续分配类似，但磁盘中移动文件效率更低

  2. 关键问题：块大小、记录空闲块、磁盘配额

  3. 块大小——理想状态一块最多给一个文件使用

    1. 块过大：小的文件浪费了大量的磁盘空间

      块过小：大多数文件会跨越多个块，需要多次寻道和旋转延迟才能读出，降低性能，浪费时间。

    2. 考虑指标

      传输数据率：一段时间内，真正在传输的数据比率

      空间使用率：真正在使用的空间比率

    3. 历史1-4KB，目前块提升到64KB并接受浪费。

  4. 记录空闲块

    1. 采用磁盘块链表，链表的每个块中包含尽可能多的空闲磁盘块号。

      当空闲块大多连在一起时，链表方式的每一个表项可以包括第一块和连续空闲块数；但如果空闲块分散，上述方法就很浪费。

      对于1KB大小的块和32位的磁盘块号，空闲表中的每个块包含有255个空闲块的块号，需要有一个位置存放指向下一块的指针。

      ==4KB大小的块，32位的磁盘块号，空闲表中每个块包含有一个指向下一个空闲块的指针。1TB的磁盘需要多少的空间保存所有空闲块？==

    2. 位图

  5. 磁盘配额——系统分配给每个用户拥有文件和块的最大数量

    目的：多用户系统中防止单个用户占用太多空间，操作系统确保每个用户不超过分到的配额。

    实现方式：软限制；硬限制。

4.4.2.2 文件系统备份

  1. 数据恢复：从意外灾难中恢复；从错误操作中恢复

  2. 整体备份和局部备份

    1. 增量转储：只备份修改的数据；周期性急性全面转储，插入增量转储（根据上次全面转储后的修改做备份或根据上次增量转储后的修改做备份）

    2. 压缩：慎重选择，可能损坏难以恢复

    3. 快照：操作系统运行过程中实时监控文件系统的修改，监控的信息称为快照，然后定期按照快照来备份

    4. 备份应远离现场

  3. 转储备份

    1. 物理转储：从第0块开始，将全部的磁盘块按序输出到磁带，没有使用的块无需备份，记录坏块的表需要备份

      优点：简单快速

      缺点：不能根据特定的目录备份或者跳过某些目录；无法增量转储；无法根据文件的性质进行会转储

    2. 逻辑转储：从一个或几个指定的目录开始递归转储，往往考虑某个基准日期后修改的文件，同时也要转储修改过得文件路径上的所有文件夹（为了把转储的内容恢复到一个新的文件系统中；可以对单个文件做增量恢复）

      四阶段的算法：

      1. 标记所有修改过的文件和目录
      2. 去掉不包含修改文件和目录的标记
      3. 转储前面标记的目录并去掉标记
      4. 转储前面标记的文件

4.4.2.3 文件系统的一致性

  1. 操作系统中专门存在一个检测文件一致性的程序，系统启动时运行该程序。

  2. 一致性检查

    1. 块的一致性检查
    2. 目录的一致性检查

    二者集成到一起使用

  3. 块的一致性

    1. 构造两张==位表==，每一位对应一块，初始化为0
    2. 检查每个i节点的块，更新第一张表，出现一块则对应位+1
    3. 检查空闲链表或位图，更新第二个表
    4. 检查两个表是否对应

  4. 目录的一致性

    1. 每个文件设定一个使用计数
    2. 递归搜索每个目录，目录中每个文件对应的计数+1，硬链接也+1，软链接不需+1
    3. 随后看使用计数与每个文件的硬链接数目（iNode中）比较
    4. 不一致则修改iNode中的计数

  5. 除了块一致性和目录一致性外，还需要

    1. 检查目录中iNode的数目是否有意义，不可能大于整个系统允许iNode的数量
    2. 检查iNode中权限是否合理

4.4.2.4 文件系统的性能

  1. 前提：访问硬盘比访问内存以数量级的形式慢。

  2. 改善性能的方法

    1. 高速缓存：内存中开辟一个空间，专门缓存部分硬盘的空间

      1. 以块为单位缓存

      2. 先看访问的块在不在缓冲区中，在则直接访问，不在就从硬盘调入缓冲区，下次如果访问不需要操作硬盘

      3. 往往会预先读，局部性

      4. 硬盘访问不像内存访问十分频繁，可以用设备+磁盘地址做散列，没有MMU类似的模块。

      5. 可能用到页面置换算法，可以使用精确地LRU（为什么）

      6. 需要考虑文件系统的特点：将块分为iNode、间接块、目录、满数据块、部分数据块等；最近不需要的块放在LRU链表的前部（根据块的类型和之前使用的情况来决定最近是否需要）

      7. 有些块需要快速写回以==保障系统的一致性==，减少在系统崩溃后文件系统损坏的可能

        UNIX：提供写回的系统调用以及后台服务

        WINDOWS：只要写入高速缓存就把每个修改的块写入磁盘

      8. 内存映射文件：高速缓存和页式管理集成；按照块缓存->按照页缓存

    2. 块提前读：顺序提前多读一些块到缓冲区

    3. 减少磁盘运动：顺序访问的块放在一起

  3. 磁盘碎片整理

    碎片：磁盘使用较多后空间变得不连续的现象，产生碎片。

    当创建一个新文件时所使用的块会散布在整个磁盘上造成性能的降低。

    1. 磁盘的寻道时间
    2. 固态硬盘：对数据的修改效率低
    3. 同时也和具体的文件系统有关

    因此需要有个专门的磁盘碎片整理程序定期运行。

4.5.1 FAT

  1. FAT：MS-DOS、早期Windows使用的文件系统，目前仍被嵌入式系统广泛使用。

  2. 簇：数据存储单元。一个簇由一组连续的扇区组成，簇所包含的扇区个数必须是2的整数次幂。

  3. FAT16目录项

    FAT文件系统内每个文件和文件夹都被分配一个==32字节的目录项==。

    1. 属性域：只读的、存档的、隐藏的、系统文件位。用户不能对只读文件进行修改，避免了文件意外受损。存档位没有专门的功能，用作用户参照。隐藏位能够使一个文件在目录列表中不出现。系统文件也是隐藏文件并且不可以用del命令删除。
    2. 时间和日期各占两字节。存在2107年溢出的问题，可以推迟到2116年。
    3. FAT用32位存储文件大小，所以理论上文件能够达到4GB但实际小于4GB。

  4. 文件分配表

    目录表中有一项指向文件分配表，文件分配表表项有一项指向地址。

    文件分配表以链表方式实现，可以把文件分配表看成一个整型数组，每个整数代表磁盘分区的一个簇号状态（未使用、坏簇、系统保留、被文件占用、下一簇簇号、最后一簇）

    每个FAT表项的大小有12位、16位和32位三种情况。

5.1 引言

  1. 计算机组成：CPU、内存、外存、输入/输出

    2. I/O性能经常成为系统性能的瓶颈

  4. I/O管理的目标和任务

    1. 按照用户的请求，控制设备的各种操作，完成I/O设备与内存之间的数据交换，最终完成用户的I/O请求。
      1. 设备的分配和回收：记录设备的状态；根据用户的请求和设备的类型采用一定的分配算法，选择一条数据通路

      3. 设备中断管理：处理外部设备的中断
      4. 缓冲区管理：管理I/O缓冲区
    2. 建立方便、统一的独立于设备的接口
      1. 方便性：向用户提供使用外部设备的方便接口，使用户编程时不考虑设备的复杂物理特性
      2. 统一性：对不同的设备采取统一的操作方式，即在用户程序中使用的是逻辑设备（屏蔽硬件细节如设备的物理特性、错误处理、不同I/O过程的差异性）
    3. 充分利用各种技术（通道、中断、缓冲、异步I/O等）提高CPU与设备、设备与设备之间的并行工作能力，充分利用资源，提高资源利用率
      1. 并行性
      2. 均衡性（使设备充分忙碌）
    4. 保护：设备传送或管理的数据应该是安全的、不被破坏的、保密的

5.2 硬件

5.2.1 概述

  1. I/O设备一般由机械和电子两部分组成：

    1. 机械部分是设备本身（物理装置）
    2. 电子部分又称设备控制器（或适配器）
      1. 端口地址译码
      2. 按照主机与设备之间约定的格式和过程接收计算机发来的数据和控制信号或向主机发送数据和状态信号
      3. 将计算机的数字信号转换成机械部分能识别的模拟信号或者将模拟信号转换成数字信号
      4. 实现设备内部硬件缓冲、数据加工等提高性能或增强功能

    上层命令写入控制器的接口寄存器/接口缓冲区，以实现输入/输出并从接口寄存器读取状态信息或结果信息。

    当控制器接受一条命令后，可独立于CPU完成指定操作，CPU可以另外执行其他计算；命令完成时，控制器产生一个中断，CPU响应中断，进而控制转交操作系统；通过读控制寄存器中的信息获取操作结果和设备状态。

    控制器与设备之间的接口常常是一个低级接口。

  2. 控制器的任务

    把串行的位流转换为字节块并进行必要的错误修正。首先，控制器按位进行组装，然后存入控制器内部的缓冲区形成以字节为单位的块，在对块验证检查并证明无错误时，在将其复制到内存中。

  3. I/O端口地址：接口电路中每个寄存器具有的唯一的地址，是一个整数。所有的I/O端口地址形成I/O端口空间。

    I/O指令形式与I/O地址是相互关联的，主要存在两种形式：

    1. 内存映像编址（内存映像I/O模式）
    2. I/O独立编址（I/O专用指令）

  4. I/O独立编址

    1. 分配给系统中所有端口的地址空间完全独立，与内存地址空间无关
    2. 使用专门的I/O指令对端口进行操作
    3. 优点：外设不占用内存的地址空间；编程时易于区分是对内存操作还是I/O端口操作
    4. 缺点：I/O端口操作的指令类型少，操作不灵活

  5. 内存映射编址

    1. 分配给系统中所有端口的地址空间与内存的地址空间统一编址
    2. 把I/O端口看作一个存储单元，对I/O的读写操作等同于对内存的操作
    3. 优点：凡是可对内存操作的指令都可对I/O端口操作；不需要专门的I/O指令；I/O端口可占有较大的地址空间
    4. 缺点：占用内存空间，不容易区分是对内存操作还是对I/O端口操作

  6. I/O设备分类

    1. 块设备：硬盘、光盘等；可按块寻址，每块通常512B-65536B
    2. 字符设备：网络接口、鼠标、打印机、键盘灯；以字符为单位接收或发射一个字符流，不可寻址
    3. 特例：时钟、内存映射的显示器

5.2.2 磁盘

  1. 磁盘的低级格式化：在磁盘能够使用之前，每个盘片必须经受由软件完成的低级格式化，用于确定磁道、扇区等。

    格式化用到的技术：柱面斜进、低级格式化之后再进行分区等工作、交错

  2. 磁盘臂调度算法

    影响读写磁盘块时间的因素：

    1. 寻道时间：将磁盘臂移动到适当的柱面上所需的时间
    2. 旋转延迟：等待适当扇区旋转到磁头下所需的时间
    3. 实际数据传输时间

    当同时出现多个不同磁道磁盘访问请求时确定访问次序的方式：

    1. 先来先服务

      不好。

    2. 最近寻道优先

      不公平，如果磁盘负载很重，那么大部分时间磁盘臂将停留在磁盘的中部区域，而两端极端区域的请求将不得不等待。

    3. 电梯算法

      公平，对任意一组给定的请求，磁盘臂移动的总次数是固定的即柱面数的两倍。

  3. 错误处理

    原因：磁盘总会出现错误的扇区，制造时的瑕疵也不可避免，不能因为有一个错误的扇区就放弃整个磁盘。

    错误的两道防线：

    1. 第一道防线：ECC——磁盘控制器修复，错误太多则无法修复

    2. 第二道防线：再读一次

    坏扇区的处理方式：

    1. 在控制器中处理

      磁盘在出厂之前要进行测试并将一个坏扇区列表写在磁盘上，对于每一个坏扇区用一个备用扇区替换之。

    2. 在操作系统中处理

      首先获得坏扇区的列表（可以读出、也可以自行测试），重新建立映射

    机械臂不准的解决方式：重新校准，移动到边缘。

  4. RAID 磁盘阵列：把多个硬盘组合起来，成为一个或多个硬盘阵列组，逻辑上只有一个虚拟化的硬盘。

    目的：提升性能；数据冗余。

    1. RAID 0 带区集 ：将两个以上的磁盘并联起来，称为一个大容量的磁盘。在存放数据时，分段后分散存储在这些磁盘中，读写可以并行处理。

      ==在所有级别的RAID中，RAID 0的速度是最快的，但RAID 0既没有冗余功能，也不具备容错能力，如果一个物理磁盘损坏则所有的数据都丢失。==

    2. RAID 1：两组以上的N个磁盘相互作镜像，在一些多线程操作系统中读取速度很快，读取速度等于磁盘数量倍的速度，写入速度微小降低。

      ==只要有一个磁盘正常即可维持运作，但无论使用多少个磁盘做RAID 1仅算作一个磁盘的容量，是所有RAID中磁盘利用率最低的级别。==

    3. RAID 2：以汉明码的方式将数据进行编码后分割为独立的比特，并将数据分别写入硬盘中。因为在数据中加入错误修正吗ECC所以数据整体的容量会比原始数据大一些。至少需要三台磁盘驱动器才能运作。

    4. RAID 5：是一种存储性能、数据安全和存储成文兼顾的存储解决方案。把数据和相对应的奇偶校验信息存储到组成RAID 5的各个磁盘上（分别存储于不同的磁盘）。当RAID 5 的一个磁盘数据发生损坏后，可以利用剩下的数据和相应的奇偶校验信息去恢复被损坏的数据。

      ==保障程度要比RAID 1低而磁盘空间利用率比RAID 1高。数据读取速度和RAID 0近似，单独写入一块磁盘的速度略慢（因为多了奇偶校验信息故写入数据速度相对变慢）。==

5.2.3 时钟

  1. 定义：定时器，每隔一个周期产生一个中断——时钟中断、时钟滴答。

  2. 实现方式：基于交流电压；基于晶体振荡器（计数器+存储寄存器；可编程）

  3. 功能

    1. 维护日期时间：可以记录时钟滴答数或者直接记录秒。
    2. 防止进程超时运行：保证合理时间片。
    3. 记录CPU的使用情况：记录每个进程的运行时间。
    4. 处理用户进程提出的alarm系统调用：如网络的超时重传机制。
    5. 为系统本身的各个部分提供监视定时器。
    6. 完成概要剖析、监视和统计信息收集：让用户了解程序每一部分运行时间占比情况。

  4. 监视定时器：自动增长，系统每隔一段时间重置监视定时器为0，如果出现监视定时器溢出即系统无法重置监视计时器则认为出现死机。

    作用：检测死机。

  5. 软时钟：由于中断开销大，为了效率而取消中断，计数器自行增加或降低，由操作系统自行检查其数值。

5.2.4 网络、用户界面相关的硬件

  1. 网卡：网络接口适配器/网络适配器/局域网接收器，拥有MAC地址，位于OSI模型的第2层，数据量大。
  2. 键盘：每个按键对应一个码。
  3. 鼠标：发送的信息包括$\Delta x、\Delta y、$按键。
  4. 显示器：一般和显卡display card/adapter相连接。
    1. 将计算机系统所需要的显示信息进行转换驱动显示器，并向显示器提供逐行或隔行扫描信号。
    2. 早期显卡只起到信号转换的作用，目前显卡一般具备3D画面运算和图形加速功能，所以也叫图形/3D加速卡。
    3. 可能集成在CPU内使用CPU部分内容作为自己的缓冲区；也可能在CPU外部使用单独的内存。
    4. 并行性带来的显卡新用途：科学计算、深度学习、虚拟货币。

5.3 中断与DMA

  1. I/O最原始的方式——轮询

    CPU循环查询某个地址中的数据，该数据对应I/O的状态如传输完成等。

    常用于简单的嵌入式系统。

    适用范围：几乎所有的I/O操作。

  2. I/O最基本的方式——中断

    目的：减少CPU的干预，解放CPU

    范围：几乎所有支持中断的I/O操作

    注意：

    1. 中断嵌套有特定的层数限制。

    2. 由于流水线的存在，有些情况可以使用不精确中断，有些不可；可以对不同类型的中断分别设计成精确和不精确。

      精确中断的要求：

      • PC保存在一个已知的位置
      • PC所指向的指令之前的所有指令已经完全执行
      • PC所指向的指令之后的所有指令都没有执行
      • PC所指向的指令的执行状态是已知的

  3. I/O最高效的方式——DMA Direct Memory Access 直接存储器存取

    目的：进一步减少CPU的干预

    适用范围：硬盘、显卡、网络接口

    使用总线的方式：

    1. 周期窃取：一次传一个字节；CPU等待DMA；轻微延迟
    2. 突发模式：一次传一块；CPU等待DMA；延迟较长

    数据流动方式：

    1. 飞跃模式：设备-内存
    2. 其他：设备-DMA控制器-内存

    注意：DMA不一定更高效，目的是节省CPU时间，CPU中断传输地更快（但是CPU可能需要做其他更为重要的事情），对于大量数据才有效（I/O开始和结束仍需要CPU中断）

5.4 设备驱动程序

  1. 设备驱动程序一般由设备制造商编写并随设备一起交付，一个设备驱动程序通常处理一种类型的设备。

  2. 宏内核中驱动程序和操作系统的关系

    驱动程序与内核编译在一起（早期UNIX），加一个设备就要重新编译一次内核，不适于个人电脑。

    驱动程序单独编译，内核中有个专门管理驱动程序的程序，需要的时候内核动态装入，有些特定的驱动程序还是需要重新编译内核。

  3. 功能

    1. 接收来自设备无关软件所发出的I/O请求并执行
    2. 接收设备的请求（来自中断）并处理
    3. 对设备进行初始化
    4. 对电源需求和日志时间进行管理
    5. ==在运行过程中可能自身被再次调用，因此必须可重入==

  4. 驱动程序的典型结构

    1. 检查输入参数
    2. 把上层的I/O请求翻译为硬件可识别的格式，如将一个线性磁盘块号转换成磁头、磁道、扇区和柱面号等信息
    3. 检查设备当前是否正在使用，如果在使用则请求被排入队列以备稍后处理
    4. 驱动程序检查硬件状态，同时进行一些准备工作使设备就绪
    5. 发出正式的I/O命令如写数据、读数据、控制信息等。
      1. I/O命令执行很快，驱动程序继续运行
      2. I/O命令执行很慢，驱动程序阻塞，等待命令执行完毕后设备利用中断唤醒驱动程序
    6. 检查错误，返回上层

5.5 与设备无关的I/O软件

  1. 基本功能：执行对所有设备公共的I/O功能，并且向用户层提供一个统一的接口。

  2. 设备驱动程序的统一接口

    对于每一种设备类型，操作系统定义一组驱动程序必须支持的函数，驱动程序需要实现这些函数并告知操作系统函数的位置。

    与设备无关的软件负责把符号化的设备名映射到适当的驱动程序上，UNIX中，设备名唯一确定了一个特殊文件的iNode。

    ==在UNIX和Windows中，设备是出现在文件系统中的，因此系统管理员额可以为每个设备设置适当的访问权限。==

  3. 缓冲

  4. 错误报告

    错误发生时，操作系统必须尽最大努力处理；许多错误是设备特定的并且必须由适当的驱动程序来处理，但错误处理的框架是设备无关的。

    1. 编程错误：直接返回代码
    2. 实际错误：首先是设备驱动处理，然后操作系统处理，最后返回错误代码

  5. 分配与释放专用设备

    对于在任意时刻只能由一个进程使用的设备如打印机，应当先检查再使用。发现被占用的时候可以先阻塞，阻塞的进程被放入一个队列，等待设备可用时被唤醒。

  6. 与设备无关的块大小：设备无关的软件应隐藏不同设备处理的物理块的大小并向高层提供一个统一的块大小。

5.6 用户空间的I/O软件

  1. 通过系统调用。

  2. 假脱机系统

    为了缓和CPU的高速性和I/O设备低速性的矛盾；将低速I/O设备的数据传送到高速磁盘上或相反时。

    组成部分：

    1. 输入井和输出井

      在磁盘上开辟的两个大存储空间，用作缓冲的存储区域

    2. 输入缓冲区和输出缓冲区

      为了缓和CPU和磁盘之间速度不匹配的矛盾

    3. 输入进程和输出进程

      模拟脱机时I/O的外围控制机

    假脱机系统可以看成是对硬件或其他不可控制设备的虚拟化，应用程序或用户与输入输出进程交互而无需关注具体的硬件。

    暂存I/O任务，使用管理进程的方式统一管理I/O任务。

    [例]电子邮箱

    • 邮箱服务器-输入输出井
    • 收发邮件的计算机-I/O设备
    • 电子邮箱客户端-输入输出进程

Chapter 6 安全

6.1 系统本身的安全措施

  1. 概述

    1. 概念区分：data safety 数据防丢失 data security 数据防泄密
    2. 安全的三个部分：机密性；完整性；可用性

  2. 可信系统

    理论上可以建立一个安全的操作系统，但实际不如此因为操作系统过于复杂且安全意味着牺牲功能性。

    可信系统在形式上声明了安全要求并满足这些安全要求，核心是可信计算基Trust Computing Base TCB 即实施所有安全规则所必须的硬件和软件。

    访问监视器接受所有与安全有关的系统请求然后决定是否允许运行。

    可信计算基的使用更适于微内核。

  3. 保护机制

    域 Domain 是 对象、权限对的集合，每一对组合指定一个对象和一些可在其上运行的操作子集，一个域可对应多个用户，一个用户可对应多个域。

    访问控制列表 Access Control List ACL，一个文件对应一个ACL（矩阵过大、过于稀疏导致矩阵表示域不适用）。

    权能字用来表示保护矩阵。

  4. 认证

    安全的前提：操作系统确定当前使用该系统的用户的身份

    登录口令：口令加密存储，利用安全哈希；加盐即口令和一个公开的随机数一起加密存储；弱口令

    挑战-响应认证：计算机产生一个随机数发给用户，用户将这个随机数当成明文、口令当成密钥进行加密，密文发送给计算机。计算机同样将随机数当成明文、口令作为密钥加密，比较密文是否一致。更复杂的场景如用户也要认证计算机则可以使用密码协议。

    物理识别的认证：Token，侧信道攻击；生物识别，指纹、虹膜信息作为口令，信息长度长但不够随机因此需要使用随机数提取器压缩长度 extractor

6.2 不安全的因素

  1. 软件漏洞

    缓冲区溢出，原因是C语言不检查内存溢出。

  2. 内部攻击——软件编写人员引入的攻击

    逻辑炸弹：程序员在软件中插入恶意代码如删除所有数据，平时不运行而程序员可控制其运行用于报复和欺诈等。规避方式为源代码审查。

    后门

    欺骗登录：相同界面盗取口令。

  3. 恶意软件

    木马、病毒、蠕虫、间谍软件：破坏、获取信息、勒索、控制计算机（形成僵尸网络、虚拟货币挖矿）、继续扩散

    特点：隐匿性，与计算机自身程序共存（可修改系统文件、可驻留于内存、可隐藏在引导区、可编程软件也会隐藏病毒）；自身进化；难以有效删除

6.3 进一步防护

  1. 防火墙

  2. 杀毒软件：专用查找并删除恶意软件的病毒，通过匹配病毒特征来查找；与病毒共同发展（特征匹配随病毒发展而进步；一些病毒会专门设计防止被匹配；病毒也会删除或关闭杀毒软件；杀毒软件也会使用一些病毒用的技术防止自己被删除）

  3. 代码签名

  4. 入侵检测

    基于特定的入侵模式

    蜜罐：引诱攻击者攻击，用于分析攻击者的攻击方式进一步优化防护系统，使用前提是攻击蜜罐的动作可以被检测到。

  5. 封装代码

    沙盒：将软件运行于一个受限的系统环境中，控制程序可使用的资源如文件描述符、内存、磁盘空间等。可以直接把可疑软件运行在虚拟机上。

    囚禁：一种沙盒的经典实现，不直接编译成可执行文件而是编译成一个中间语言（由解释器执行中间语言，编译过程和结果与硬件无关，解释器是安装在具体硬件上的软件，不同硬件的解释器是不一样的，解释器保证安全），如果程序存在恶意代码解释器会发现并禁止运行。

  6. 总结

    很少有完美的操作系统，一般做到攻击者攻击代价足够大即可。攻击和防护是同时发展的。最好的策略就是多种技术一起使用实现全面防护。
