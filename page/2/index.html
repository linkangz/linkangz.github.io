[操作系统]文件管理

4.1.3 文件目录

  1. 文件控制块（FCB）

    FCB包含文件的基本信息（文件名、文件存放的物理地址、逻辑结构、物理结构）、存取控制信息（是否可读/写、禁止访问的用户名单）、使用信息（文件的建立时间、修改时间）；目录文件中的一条记录就是一个文件控制块FCB；FCB实现了文件名和文件之间的映射 。

  2. 操作：搜索；创建文件；删除文件；显示目录；修改目录

  3. 目录结构

    1. 单级目录结构：不允许重名

    2. 两级目录结构：主文件目录记录用户名及相应的用户文件目录的存放位置；用户文件目录由该用户的文件FCB组成

      允许不同用户文件重名、提供权限控制

    3. 多级目录结构/树形目录结构：不同目录下的文件可以重名；层次清晰、方便分类、提供有效的管理和保护；不便于实现文件共享

    4. 无环图目录结构：在树形目录结构的基础上增加了指向同一个结点的有向边从而使整个目录成为一个有向无环图；共享计数器，删除一个共享计数器减1，减为0时源文件彻底删除。

  4. 索引结点（iNode）

    目录包含文件名和指向索引结点的指针，目录项变小，占用磁盘块变小，访存次数变少。

4.1.4 文件的物理结构（文件的分配方式）

  1. 操作系统对磁盘块的管理：对非空闲磁盘块的管理；对空闲磁盘块的管理

  2. 分类

    1. 连续分配
    2. 链接分配：隐式链接、显式链接
    3. 索引分配

  3. 文件块、磁盘块

  4. 连续分配：要求每个文件在磁盘上占有一组连续的块

    用户给出要访问的逻辑块号，操作系统找到文件对应的目录项，物理块号=起始块号+逻辑块号。文件目录中存放的起始块号和长度。

    优点

    1. 连续分配支持顺序访问和直接访问。

    2. 读取某个磁盘块时需要移动磁头，访问两个磁盘块相距越远移动磁头所需时间就越长。因此，连续分配的文件在顺序读/写时速度最快。

    缺点：

    1. 连续分配的文件不方便扩展。

    2. 产生难以利用的外部碎片，存储空间利用率低。

  5. 链接分配：采取离散分配的方式，可以为文件分配离散的磁盘块，分为隐式链接和显式链接两种。

    隐式链接：

    1. 目录中记录了文件存放的起始块号和结束块号，除了文件的最后一个磁盘块之外，每个磁盘块都会保存指向下一个磁盘块的指针，指针对用户透明。

      用户给出要访问的逻辑块号i，操作系统根据文件名找到目录项，找到起始块号，将0号块读入内存，由此知道1号逻辑块存放的物理块号，于是读入1号逻辑块，再找到2号逻辑块的存放位置，以此类推。访问i号逻辑块需要i+1次磁盘I/O操作。

    2. 优点：便于文件拓展，不会有碎片问题 ，外存利用率高。

    3. 缺点：只支持顺序访问，不支持随机访问，查找效率低，指向下一个磁盘块的指针也要浪费少量存储空间。

    显式链接：

    1. 把用于链接文件各物理块的指针显式地存放在一张表中，即文件分配表（FAT， File Allocation Table）

    2. 用户给出要访问的逻辑块号，操作系统找到该文件对应的目录项FCB，从目录项中找到起始块号，若i>0，则查询内存中的文件分配表，往后找到i号逻辑块对应的物理块号。逻辑块号转换为物理块号的过程不需要读磁盘操作。

    3. 优点：方便文件拓展，不会有碎片问题，外存利用率高，支持随机访问。相比于隐式链接 ，地址转换不需要访问磁盘，因此文件的访问效率更高。

    4. 缺点：文件分配表需要占用一定的存储空间。

  6. 索引分配

    1. 允许文件离散地分配在各个磁盘块中，系统会为每个文件建立一张索引表，索引表中记录了文件的各个逻辑块对应的物理块。索引表存放的磁盘块称为索引块，文件数据存放的磁盘块称为数据块。

      用户给出要访问的逻辑块号i，操作系统找到该文件对应的目录项FCB。从目录项可知索引表存放位置，将索引表从外存读入内存，并查找索引表即可知i号逻辑块在外存中的存放位置。

    2. 优点：支持随机访问；文件拓展容易实现（给文件分配一个空闲块，并增加一个索引表项即可）

    3. 缺点：索引表需要占用一定的存储空间

    如果文件太大，表项太多。

    1. 链接方案：如果索引表过大，无法存储在一个索引块，那么可以将多个索引块链接起来存放。

    1. 多级索引：建立多层索引，使第一层索引块指向第二层索引块，还可根据文件大小的要求建立更多层级索引块。采用k层索引结构，且顶级索引表调入内存，则访问一个数据块只需要k+1次读磁盘操作。

    2. 混合索引：多种索引分配方式的结合。一个文件的顶级索引表中既包含直接地址索引（直接指向数据块），又包含一级间接索引（指向单层索引表）、还包含两级间接索引（两层索引表）。

4.1.5 文件存储空间管理

  1. 存储空间的划分：将物理磁盘划分为若干文件卷（逻辑卷、逻辑盘），每个文件卷分为目录区和文件区，目录区只要存放文件目录信息用于磁盘存储空间管理的信息，文件区用于存放文件数据。

  2. 存储空间的初始化：将各个文件卷划分为目录区、文件区。

  3. 空闲表法：适用于连续分配方式

    分配：首次适配、最佳适配、最差适配

    回收：①回收区的前后都没有相邻空闲区——空闲表新增表项目②回收区的前后都是空闲区——合并，表项减1③回收区前面是空闲区④回收区后面是空闲区

  4. 空闲链表法——空闲盘块、空闲盘区

    1. 空闲盘块链：操作系统保存着链头、链尾指针。

      分配方式：若某文件申请k个盘块，则从链头开始依次摘下k个盘块分配，并修改空闲链的头指针。

      回收方式：回收的盘块依次挂到链尾，并修改空闲链的链尾指针。

      适用于离散分配的物理结构，为文件分配多各磁盘块需要多次操作。

    2. 空闲盘区链：操作系统保存着链头、链尾指针。

      分配方式：若某文件申请k个盘块，则可以采用首次适配、最佳适配等算法，从链头开始检索，按照算法规则找到一个大小符合要求的空闲盘区，分配给文件，若没有合适的连续空袭那块也可以将不同盘区的盘块同时分配给一个文件，注意分配后可能要修改相应的指针、盘区大小等数据。

      回收方式：若回收区和某个空闲盘区相邻则需要将回收区合并到空闲盘区中，若回收区没有和任何空闲区相邻，将回收区作为一个单独的空闲盘区挂到链尾。

      离散分配、连续分配都适用，为一个文件分配多个磁盘块时效率高。

4.1.6 文件基本操作

  1. 创建文件

    Create()

    提供文件存放路径、文件名作为参数

    1. 在外存中找到文件所需的空间（根据空闲链表法、位示图、成组链接法等管理策略找到空闲空间）
    2. 根据文件存放路径的信息找到该目录对应的目录文件，在目录中创建该文件对应的目录项。目录项中包含文件名、文件在外存中存放位置等信息。

  2. 删除文件

    Delete()

    提供文件存放路径、文件名作为参数。

    1. 根据文件存放路径找到对应的目录文件，从目录文件找到文件名对应的目录项；

    2. 根据该目录项记录的文件在外存的存放位置、文件大小等信息，回收文件占用的磁盘块；

      （回收磁盘块时，根据空闲表法、空闲链表法、位图法等管理策略的不同作不同处理）；

    3. 从目录表中删除文件对应的目录项。

  3. 打开文件

    Open()

    提供文件存放路径、文件名、文件的操作类型作为参数。

    1. 根据文件存放路径在外村中找到对应的目录文件，从目录中找到文件名对应的目录项并检查该用户是否有指定的操作权限；
    2. 将目录项复制到内存中的打开文件表中，并将对应表目的编号返回给用户，之后用户使用打开文件表的编号来指明要操作的文件。

  4. 关闭文件

    Close()

    1. 将进程的打开文件表相应表项删除
    2. 回收分配给该文件的内存空间等资源
    3. 系统打开文件表的打开计数器count减1，若count=0，则删除对应表项。

  5. 读文件

    Read()

    提供打开文件表中的编号（指明读的文件）、读入数据大小（如1KB）、读入数据放在内存中的位置作为参数。

    操作系统在处理read系统调用时，会从读指针指向的外存中将用户制定大小的数据读入用户指定的内存区域中。

  6. 写文件

    Write()

    提供文件在打开文件表中的索引号（指明要写的文件）、写入数据量大小（如1KB）、写回外存的数据放在内存中的位置作为参数。

    操作系统在处理write系统调用时，会从用户hiding的内存区域中，将指定大小的数据写回写指针指向的外存。

4.1.6 文件共享

  1. 硬链接

    iNode中设置一个链接计数变量count，用于表示链接到本iNode上的用户目录项数。若某个用户决定删除该文件，则知识要把用户目录中与该文件对应的目录项删除，且iNode的count值减1。若count>0，说明还有别的用户在使用该文件，暂时不能把文件数据删除，否则会导致指针悬空；若count=0则由操作系统负责删除文件。

4.2.3 减少磁盘延迟时间的方法

  1. 交替编号

    1. 原因：磁头读入一个扇区数据后需要一小段时间处理，如果逻辑上相邻的扇区在物理上也相邻，则读入几个连续的逻辑扇区可能需要很长的延迟时间。
    2. 若采用交替编号策略，即让逻辑上相邻的扇区在物理上有一定的间隔，可以使连续的逻辑扇区所需要的延迟时间更小。

4.2.4 磁盘的管理

  1. 磁盘初始化

    1. 低级格式化：将磁盘的各个磁道划分为扇区。一个扇区通常可分为头、数据区域、尾三个部分。管理扇区所需要的各种数据结构一般存放在头、尾（如指针等，不占用数据区域）两个部分，包括扇区校验码（如奇偶校验、CRC循环冗余校验），校验码用于校验扇区中的数据是否发生错误；
    2. 分区：将磁盘分区，每个分区由若干柱面组成，分为C、D、E盘等；
    3. 逻辑格式化：创建文件系统。包括创建文件系统的根目录、初始化存储空间管理所用的数据结构（如位示图、空闲分区表）。

  2. 引导块

    计算机开机时需要进行一系列的初始化工作，通过初始化程序/自举程序完成。初始化程序通常放在ROM中，但由于ROM出厂时集成在主板上，无法更改，因此ROM只存放很小的自举装入程序，而完整的自举程序存放在磁盘的启动块/引导块/启动分区上，启动块位于磁盘的固定位置。

    开机时先运行自举装入程序，执行后找到引导块并将完整的自举程序读入内存，完成初始化。

    拥有启动分区的磁盘称为启动磁盘或系统磁盘。

  3. 坏块管理

    硬件故障，操作系统无法修复。

    对于简单的磁盘，可以在逻辑格式化时（建立文件系统时）对整个磁盘进行坏块检查，标明哪些扇区是坏扇区（如在FAT文件分配表上标明、坏块对操作系统不透明）。

    对于复杂的磁盘，磁盘控制器（磁盘设备内部的一个硬件部件）负责维护一个坏块链表。在磁盘出厂之前进行低级格式化时候就将坏块链表进行初始化。会保留一些备用扇区，用于替换坏块。此方案称为扇区备用，坏块对操作系统透明。

5.1.1 I/O设备的概念和分类

  1. I/O设备：可以将数据输入到计算机或者可以接收计算机输出设备的外部设备，属于计算机中的硬件部件。

    输入：鼠标、键盘；输出：显示器；既可输入又可输出：移动硬盘

  2. 分类

    1. 按使用特性分类
      1. 人机交互类外部设备：鼠标、键盘、打印机——用于人机交互；数据传输速度慢
      2. 存储设备：移动硬盘、光盘——用于数据存储；数据传输速度较快
      3. 网络通信设备：调制解调器——用于网络通信；介于二者之间
    2. 按传输速率分类
      1. 低速设备：鼠标键盘——传输速率为每秒几个到几百字节
      2. 中速设备：激光打印机——传输速率为每秒数千至上万个字节
      3. 高速设备：磁盘——传输速率为每秒数千字节到千兆字节
    3. 按信息交换的单位分类
      1. 块设备：磁盘——数据传输速率的基本单位是块；传输速率较高，可寻址（可随机读写任意一块）
      2. 字符设备：鼠标、键盘——数据传输的基本单位是字符；传输速率较慢，不可寻址，在输入/输出时常采用中断驱动方式

5.1.2 I/O控制器

  1. 存在意义：CPU无法直接控制I/O设备的机械部件，因此I/O设备还要有一个电子部件作为CPU和I/O设备机械部件之间的中介，用于实现CPU对设备的控制。

  2. 功能：

    1. 接收和识别CPU发出的命令：如CPU发来的write/read命令，I/O控制器中会有相应的控制寄存器来存放命令和参数；
    2. 向CPU报告设备的状态：I/O控制器中会有相应的状态寄存器用于记录I/O设备当前的状态如1表示空闲0表示忙碌；
    3. 数据交换：I/O设备中会设置相应的数据寄存器。输出时数据寄存器用于暂存CPU发来的数据，之后再由控制器传送设备。输入时，数据寄存器用于暂存设备发来的数据，之后CPU从数据寄存器中取走数据；
    4. 地址识别：类似于内存的地址，为了区分设备控制器中各个寄存器也需要给各个寄存器设置一个特定的地址。I/O控制器通过CPU提供的地址来判断CPU要读写的哪一个寄存器。

  3. I/O控制器的组成

    1. CPU与控制器的接口：用于完成CPU与控制器之间的通信。CPU通过控制线发送命令，通过地址线指明要操作的设备；通过数据线来取出（输入）数据和放入（输出）数据。
    2. I/O逻辑：负责接收和识别CPU的各种命令并负责向设备发出命令（设备可理解的命令）。
    3. 控制器与设备的接口：用于实现控制器与设备之间的通信。一个设备控制器可能控制多个I/O设备，因此设备编号/接口地址。

    注：

    1. 一个I/O控制器可能会对应多个设备；

    2. 数据寄存器、控制寄存器、状态寄存器可能有多个（每个寄存器对应一个具体的设备），且这些寄存器都有对应的地址才能方便CPU操作。

  4. 内存映像I/O与寄存器独立编址

    1. ==内存映像I/O：寄存器占内存地址的一部分，控制器中的寄存器与内存地址统一编址。==

      优点：简化了指令。可以采用对内存进行操作的指令来对控制器进行操作。

    2. ==寄存器独立编址：采用专用I/O地址，控制器中的寄存器使用单独的地址。==

      缺点：需要设置专门的指令来实现对控制器的操作，不仅要指明寄存器的地址，还要指明控制器的编号。

5.1.3 I/O控制方式

  1. 程序直接控制方式

    过程：

    1. CPU向I/O控制器发出读指令。于是设备启动，并且状态寄存器设为1（未就绪）
    2. 轮询检查控制器的状态（不断执行程序的循环，若状态位一直是1，说明设备还没准备好要输入的数据，于是CPU会不断地轮询）（I/O状态寄存器放入CPU寄存器）
    3. 输入设备准备好数据后将数据传给控制器并报告自身状态
    4. 控制器将输入的数据放到数据寄存器中并将状态修改为0（已就绪）
    5. CPU发现设备已就绪即可将数据寄存器中的内容读入CPU的寄存器中，再把CPU寄存器中的内容放到内存
    6. 若还要继续读入数据，则CPU继续发出读指令

    CPU干预的频率：很频繁，I/O操作开始之前、完成之后需要CPU介入，并且在等待I/O完成的过程中CPU需要不断地轮询检查

    数据传送的单位：每次读写一个字

    数据流向：输入：I/O到CPU到内存；输入：内存到CPU到I/O设备

    优点：实现简单。在读写指令之后加上实现循环检查的一系列指令即可。

    缺点：CPU和I/O设备只能串行工作，CPU需要一直轮询检查，长期处于忙等状态，CPU利用率低。

  2. 中断驱动方式

    原理：引入中断机制，由于I/O设备速度很慢，因此在CPU发出读写命令后可将等待I/O的进程阻塞，先切换到别的进程执行。当I/O完成后，控制器会向CPU发出一个中断信号，CPU检测到中断信号后会保存当前进程的运行环境信息，转去执行中断处理程序处理该中断。在处理中断的过程中，CPU从I/O控制器读一个字的数据传送到CPU寄存器再写入主存。接着，CPU恢复等待I/O的进程的运行环境然后继续执行。

    注意：

    1. CPU会在每个指令周期的末尾检查中断
    2. 中断处理过程中需要保存、恢复进程的运行环境。这个过程是需要一定时间开销的。如果中断发生的频率太高，也会降低系统性能。

    CPU干预频率：每次I/O操作开始之前、完成之后需要CPU的介入，等待I/O完成的过程中CPU可以切换到别的进程执行。

    数据传送的单位：每次读写一个字。

    数据流向：输入：I/O到CPU到内存；输入：内存到CPU到I/O设备。

    优点：与程序直接控制方式相比，中断驱动方式中，I/O控制器会通过中断信号主动报告I/O已完成，CPU不再需要不停地轮询。CPU和I/O设备可并行工作，CPU利用率得到明显提升。

    缺点：每个字在I/O设备和内存之间的传输都需要CPU，频繁的中断处理会消耗较多的CPU时间。

  3. DMA方式（直接存储器存取，主要用于块设备的I/O控制）

    DMA控制器

    数据传送的单位：每次读写一个块（每次读写的只能是连续的多个块且这些块读入内存后在内存中也必须是连续的）

    数据流向：I/O设备到内存，内存到I/O设备

    CPU干预频率：仅在传送一个或多个数据块的开始和结束时才需要CPU干预。

    CPU给I/O发送读/写一个块的指令（说明读入和写入多少数据、数据要存放在内存的什么位置、数据在外部设备上的地址），之后CPU做其他事情。DMA控制器根据命令参数完成一系列读写工作，读/写完毕后，由DMA控制器向CPU发送一个中断信号，CPU介入处理中断。

    优点：数据传输以块为单位，CPU介入频率进一步降低。数据的传输不再需要先经过CPU再写入内存，数据传输效率进一步增加。CPU和I/O设备的并行性得到提升。

    缺点：CPU每发出一条I/O指令，只能读写一个或多个连续的数据块。如果要读写多个离散的数据块，或者要将数据分别写到不同的内存区域时，CPU要分别发出多条I/O指令，进行多次中断处理才能完成。

  4. 通道控制方式* 通道 CPU 通道程序 任务清单

    CPU干预的频率：极低，通道会根据CPU的知识执行相应的通道程序，只有完成一组数据块的读写后才需要发出中断信号请求CPU的干预。

    数据传送的单位：每次读写一组数据块

    数据的流向：I/O到内存；内存到I/O

    优点：CPU、I/O、通道设备可并行工作，资源利用率很高。

    缺点：实现复杂，需要专门的通道硬件支持。

  5. 总结归纳

5.1.4 I/O软件层次结构

  1. 用户层软件：用户层软件实现了与用户交互的接口，用户可直接使用该层提供的、与I/O操作相关的库函数对设备进行操作。用户层软件将用户请求翻译成格式化的I/O请求，并通过系统调用请求操作系统内核的服务。

  2. 设备独立性软件：设备无关性软件，与设备的硬件特性无关的功能几乎都在这一层实现。

    1. 设备独立层软件向上层提供统一的调用接口（如read/write系统调用）；

    2. 设备的保护（控制访问权限）；

    3. 差错处理（对设备错误进行处理）；

    4. 设备的分配与回收；

    5. 数据缓冲区管理（屏蔽设备之间数据交换单位大小和传输速度的差异）

    6. 建立逻辑设备名（用户请求使用时的设备名，用户可见）到物理设备名的映射关系，根据设备类型选择调用相应的驱动程序；通过逻辑设备表来确定逻辑设备对应的物理设备并找到该设备对应的设备驱动程序。

  3. 设备驱动程序：主要负责对硬件设备的具体控制，将上层发出的一系列命令如read/write转化成特定设备可以理解的一系列操作。包括设置设备寄存器、检查设备状态等。不同的I/O设备有不同的硬件也行，具体细节只有设备的厂家了解，因此厂家需要根据设备的硬件特性设计并提供相应的驱动程序，驱动程序一般会以一个独立进程的方式存在。

  4. 中断处理程序：当I/O任务完成时，I/O控制器会发送一个中断信号，系统会根据中断信号类型找到相应的中断处理程序并执行。

    首先中断处理程序从I/O控制器读出设备状态判断I/O是否正常结束。

    若是，则中断处理程序从设备的数据寄存器中读入一个字的数据并经由CPU放到内存缓冲区中；若否，操作系统根据异常原因做出相应处理。

3.2 地址空间

3.2.2 交换技术

1.出现交换技术的原因

  • 单个进程所需内存越来越大。
  • 多进程下所需内存非常大。
  • 进程往往会动态增加。
  • 所有进程都放入内存中往往是不现实的，会出现内存超载。

为了解决内存超载的问题。

2.内存紧缩

交换在内存中产生了多个空闲区，容易造成浪费，因此需要定期进行内存紧缩。

把所有的进程尽可能向下移动，将这些小的空闲区块合并成一个大块。但是需要耗费大量的CPU时间。

3.进程创建时分配内存大小

  • 若进程创建时大小是固定的且不在改变，操作系统只需准确的按其需要的大小进行分配。

  • 如果进程的数据段可以增长则会出现问题，如果相邻空闲区则可以直接使用，若相邻另一个进程则需要把增长的进程转移到内存中一个足够大的区域，如果交换区也满，则这个进程只有一直挂起直到一些空间空闲。

4.碎片——不能利用的内存空间

  • 内碎片：已经分配给作业但不能被利用的内存空间

  • 外碎片：还没有分配给作业但是由于太小而无法分配给申请内存空间的新进程的存储块

3.2.3 空闲内存管理

1 固定分区分配

1.实现方式

  • 将内存空间划分为若干固定大小的分区

  • 每个分区装入一道程序

  • 分区大小可以不等，但必须事先确定，在运行时不能改变

  • 当有空闲分区时，从后备队列选择一个适当大小的作业装入运行

2.分区说明表：用以记录可用于分配的分区号、分区的大小、分区的起始地址及状态（通常按照分区大小顺序排序）。

3.优点：是一种最简单的存储分配方式。

4.缺点：利用率较低，会产生内碎片。

2 位图

1.实现方式

  • 内存被划分为若干的分配单元

  • 每个分配单元对应位图中的一位

对于上图，P表示已分配、H表示空闲，第一个表项中0表示起始地址、5表示长度/大小。

2.分配单元大小的影响

  • 分配单元越小，位图越大，分配粒度小，浪费少。
  • 分配单元越大，位图越小，分配粒度大，浪费大。

3.缺点

进程调入内存时，存储管理器必须搜索位图。

3 链表

1.实现方式：维护一个记录已分配内存段和空闲内存段的链表（使用双向链表效果更佳）。

2.优点：进程终止或者被换出时链表更新直接。

4 基于链表的空闲内存适配算法

1 首次适配

算法：存储器沿着链表进行搜索，直到找到一个足够大的空闲区，除非空闲区大小和要分配的空间大小正好一样，否则将该空闲区分为两部分，一部分供进程使用，另一部分形成新的空闲区。

优点：速度很快，尽可能少地搜索链表结点。

缺点：形成大量外碎片。

2 下次适配

算法：每次找到合适的空闲区都记录当时的位置，以便在下次寻找时从上次结束的地方开始搜索。

优点：外碎片略少于首次适配。

缺点：性能略低于首次适配算法。

3 最佳适配

算法：搜索整个链表，找出能够容纳进程的最小的空闲区。

优点：找到最接近实际需要的空闲区。

缺点：比首次适配算法慢；可能比首次适配和下次适配算法浪费更多的内存，产生大量无用的小空闲区。

5 最差适配

算法：总是分配最大的可用空闲区，使新的空闲区比较大从而可以继续使用。

缺点：浪费内存多

注：如果进程和空闲区使用不同的链表，则可以按照大小对空闲区链表进行排序，提高适配速度。空闲区链表按大小排序时，首次适配算法和最佳适配算法一样。

6 快速适配

算法：为那些常用大小的空闲区维护单独的链表。

优点：寻找一个指定大小的空闲区十分快速。

缺点：在一个进程终止或者被换出时，寻找它的相邻快并查看是否可以合并的过程非常费时；如果不合并，内存很快分裂出大量的进程无法利用的小空闲区。

7 伙伴系统

主要思想：将内存按2的幂次进行划分，组成若干空闲链表，查找链表找到满足进程的最佳匹配块。

算法：

  1. 将可用空间看作一块$2^u$
  2. 假设进程申请s
  3. 若$2^{u-1} \lt s \le 2^u$，则分配；否则将该块划分为两个$2^{u-1}$重复上述过程。
8 slab机制

主要思想：针对一些经常分配并释放的对象（如进程描述符，大小比较小，并且如果直接采用伙伴系统容易产生内碎片并且处理速度慢）。

slab分配器基于对象管理，将相同类型的对象归为一类。每当要申请这样一个对象，slab分配器就从一个slab列表中分配一个这样大小的单元出去，当要释放时，将其重新保存在该列表中，而不是直接返回伙伴系统，避免这些内碎片。

3.3 虚拟内存

3.3.0 前导

1 背景

交换技术很难适用于程序大于内存的情况并且可能发生一个进程占用一整块内存的情况。

2 覆盖技术的实现

  • 把程序分割成许多片段，称为覆盖。
  • 程序开始执行时，将覆盖管理模块装入内存，该覆盖管理模块立即装入并运行覆盖0
  • 执行完成后，覆盖0通知管理模块装入覆盖1（如果有空间就占用覆盖0上方的位置；如果没有空间就占用覆盖0的位置）

3 存在问题

程序员必须自行分割程序，费时且容易出错。

3.3.1 分页

1 基本概念

0.由程序产生的地址称为虚拟地址，它们构成了一个虚拟地址空间。在没有虚拟内存的计算机上，系统直接将虚拟内存送到内存总线上，读写操作使用具有同样地址的物理内存字，而在使用虚拟内存的情况下，虚拟地址不是被直接送到内存总线上，而是被送到内存管理单元MMU，MMU把虚拟内存映射为物理内存地址。

1.用户进程地址空间被划分为大小相等的部分，称为页或页面，从0开始编号。

2.内存空间按照同样大小划分为大小相等的区域，称为页框，从0开始编号，也称为物理页面、页帧、内存块。

3.内存分配规则：以页为单位进行分配，并按进程需要的页数来分配；逻辑上相邻的页在物理上不一定相邻。

4.逻辑地址=页号+页内地址

5.划分是由系统自动完成、对用户透明的。

2 缺页

解决虚拟地址空间比内存地址空间大的问题：当程序访问了一个未被映射的页面，MMU（通过页表项中的在/不在位）注意到该页面没有被映射，于是CPU陷入到操作系统发生缺页中断/缺页错误。操作系统找到一个很少使用的页框且把它的内容写入磁盘（如果它不在磁盘上）。随后把需要访问的页面读到刚才回收的页框中，修改映射关系，然后重新启动引起陷阱的指令。

页号作为页表的索引以得出对应于该虚拟页面的页框号。如果在/不在位为0，则引起一次操作系统陷阱。如果为1，则在该页表中查到页框号复制到输出寄存器的高三位中，再加上虚拟地址中的低12位偏移量就构成了15位的物理地址。输出寄存器的内容随即被作为物理地址送到内存总线。

3.3.2 页表

1 基本概念

1.页表是一张表，负责实现虚拟地址到物理地址的映射，每个进程一个页表存放在内存。

2.虚拟地址=虚拟号+偏移量

3.虚拟号可以作为页表的索引，索引到具体的页表项。

2 页表项的结构

页表项：页表项的结构与机器密切相关，但不同机器的页表项存储的信息都大致相同。

  • 页框号：；页映射的目的。
  • 在/不在位：决定缺页中断
  • 保护位：决定读写
  • 修改位：记录当前页面是否被修改/决定换回硬盘是否需要被写会
  • 访问位：记录当前页面是否正在访问

注：(1)利用页表的查找和地址转换是硬件完成的。(2)页面在磁盘中的地址不在页表项中（因为缺页中断是软件完成的事情，而页表是硬件自动访问的）

3.3.3 加速分页过程

0 前导&最简单的实现

虚拟地址到物理地址的映射必须非常快；如果虚拟地址空间很大，页表也会很大。

实现方式1：使用一个快速页表寄存器阵列，把页表放入寄存器中。启动一个进程的时候，页表装入此寄存器阵列中。

  • 优点：页表查找速度最快。

  • 缺点：(1)代价高昂。(2)页表要一次性全部读入。

实现方式2：页表全部在内存中。

  • 优点：代价相对低一些。
  • 缺点：页表查找速度相对较慢。

1 转换检测缓冲区（相联存储器/快表）

1.目的：为了加速页表查找（相当于寄存器和内存之间的cache）

2.程序访问的局部性原理：大多数程序总是对少量的页面进行多次的访问；只有很少的页表项会被反复读取，而其他的页表项很少被访问。

3.设计思路：设置一个小型的硬件设备，将虚拟地址直接映射到物理地址而不必在访问页表。通常直接将其放在MMU中作为相联存储器，按内容并行查找，是cache的一种。

4.特征：与页表项中的记录一一对应；多了一个虚拟页号（在页表项中由于索引就作为了虚拟页号所以不是必须的）；如果不在TLB中则直接查找页表并替换TLB中的一行。

5.软件TLB管理

方式演变：

  • 过去的计算机中TLB和页表的管理完全由处理器硬件负责（对软件透明）
  • 现代的精简指令集处理器可能不再完全由硬件处理。由此减少处理器和指令执行的复杂度，减少MMU的实现代价；此外，用软件来判断哪些页表项放入TLB可能更加灵活和有效（增加TLB的命中率）

可能出现的问题：

  • 软失效：页表项不在TLB中，需要更新TLB，不需要产生磁盘I/O。需要10-20个机器指令并花费几纳秒完成操作。
  • 硬失效：页面本身不在内存中，此刻需要一次磁盘存取以装入该页面。需要几毫秒，处理时间是软失效的百万倍。

硬失效在实际情况中由三种可能性：

  • 所需的页面可能就在内存中，但却未记录在进程的页表中。
  • 需要从硬盘中重新调入页面，发生缺页错误。
  • 程序访问了一个非法地址，无需向TLB中新增映射。

3.3.4 针对大内存的页表

0 前导

1.页表需要连续存放，每个进程都有自己的页表。

2.例题：

  • 假设一个页表项8B，进程虚拟地址空间为4GB内存，页大小为4KB，一个进程页表需要多大空间？答案：4GB/4KB*8B
  • 64位计算机中，假设进程地址空间为$2^{48}$，一个页表项8B，页大小为4KB，一个进程页表需要多大空间？答案：$2^{48}$B/4KB*8B

1 多级页表

优点：

  • 不需要线性存储
  • 只需要部分页表存在内存（或TLB）即可
  • 增加级数可以很好地处理更大内存的情况

缺点：

  • 访问页表的次数增加

增加页表的级数本质上是允许页表非线性存放而不能减少页表项数。

2 倒排页表

思路：

  • 每个页表项对应一个页框
  • 之前是每个页表项对应一个页

优点：

  • 当虚拟地址空间比物理内存大得多的时候可以节省大量的空间

缺点：

  • 查表速度慢

实现方式：从物理地址空间出发，系统建立一张页表，页表项记录进程i的虚拟地址（虚页号）与页框号的映射关系，整个系统一张表。系统的每个地址有一个三元组<process-id,page-number,offset>

根据process-id和page-number从page table找到对应的页框号，加上offset到物理内存找查找。

3.4 页面置换算法

3.4.0 前导

最佳算法->先进先出->第二次机会->时钟->最近未使用->最近最少使用->最不经常使用->老化算法->工作集->工作集时钟

<设计思想，算法实现，算法应用>

颠簸：虚拟内存中，页面在内存与磁盘之间频繁调度，是的调度页面所需的时间比进程实际运行的时间还多，导致系统效率急剧下降，这种现象称为颠簸或抖动。

3.4.1 最佳页面置换算法

设计思想：置换以后不再需要或者最远的将来才会用到的页面。

优点：效果最好。

缺点：由于操作系统无法知道各个页面下一次将在什么时候被访问，所以无法实现。

应用：作为一种标准来衡量其他算法的性能。

3.4.2 先进先出置换算法

设计思想：由操作系统维护一个所有当前在内存中的页面的链表，最新进入的页面放在表尾，最早进入的页面放在表头。当发生缺页中断时，淘汰表头的页面并把新调入的页面加到表尾。

优点：实现简单。

缺点：几乎没有考虑页面的使用频率以及程序访问的局部性原理。

3.4.3 第二次机会算法

设计思想：页表项新增一个访问位R，每次MMU访问该页表项时R置为1。按照先进先出算法选择某一页面，检查其访问位R，如果为0则置换该页面；如果为1将R置为0并放到链表尾部（若修改后不放回队尾，则修改后在首部的页面很容易被再次找到被置换，不公平）。

优点：考虑了使用频率和局部性原理。

缺点：经常要在链表中移动页面，既降低了效率又不是很有必要。

3.4.4 时钟算法

算法：把所有页面都保存在一个类似钟面的环形链表中，当发生缺页中断时，算法首先检查表针指向的页面，如果R位是0就淘汰该页面并把新的页面插入这个位置，把表针前移一个位置，如果R位是1就清除R位并前移一个位置，重复上述过程直到找到一个R位为0的页面位置。

3.4.5 改进型时钟算法

规定：

  • 最近未访问 最近未修改 （0,0）

  • 最近未访问 最近被修改 （0,1）

  • 最近被访问 最近未修改 （1,0）

  • 最近被访问 最近被修改 （1,1）

算法：

  • 从指针的当前位置开始，扫描循环链表。在这次扫描过程中，对访问位和修改位不做修改。选择遇到的第一个（0,0）页面用于替换。
  • 若上一步没有找到，则重新扫描。寻找（0,1）页面用于替换，在找的过程中将R置为0。
  • 如果上一步仍没有找到，再次执行前两步。

优点：考虑了页面载入内存后是否被修改的问题，减少了I/O次数。

缺点：增加扫描次数。

3.4.6 最近最少使用算法

设计思想：选择在最近一段时间未使用过的一页并置换（向前看；而最佳置换算法是向后看）。

优点：充分考虑了程序访问的局部性。

缺点：费时；难以实现；需要硬件支持。

1 实现方式1

要求硬件有一个64位计数器C，在每条指令执行完后自动加1，每个页表项必须有一个足够容纳这个计数器值的域。在每次访问内存后，将当前的C值保存到被访问页面的页表项中。一旦发生缺页中断，操作系统就检查所有页表项中计数器的值，找到值最小的一个页面（最近最少使用的页面）。

2 实现方式2

设置页表表项的两位——访问位R和修改位M，初始值为0，访问一个页面置1，修改一个页面置1，定期让所有的页面R置0。

发生缺页中断时，操作系统检查R、M。随机从编号最小的非空类中选择一页计算。

规定：

  • 第1类 无访问 无修改 （0,0）
  • 第2类 无访问 有修改 （0,1）
  • 第3类 有访问 无修改 （1,0）
  • 第4类 有访问 有修改 （1,1）

3.4.7 最不常用算法

算法：算法将每个页面与一个软件计数器相关联，计数器的初始值为0.每次时钟中断时由操作系统扫描页表中所有的页面，将每个页面的R位（0或1）加到它的计数器上。这个计数器大体上跟踪了所有页面被访问的频繁程度。发生缺页中断时，置换计数器值最小的页面。在时钟中断时由中断处理程序实现，是一种软件实现方式，计数器不用放在页表中。

缺点：最近经常使用的优先权小于之前经常使用的，而之前经常使用的由于一直换不出去导致对应的计数器值一直增大。

3.4.8 老化算法

算法：在最不常用算法基础上进行修改。首先，在R位被加进之前先将计数器右移一位；其次，将R位计数器加到最左端的位而不是最右端的位。

优点：最接近最近最少适用算法。

3.4.9 工作集算法

设计思想：根据程序的局部性原理，一般情况下，进程在一段时间内总是集中访问一些页面，这些页面被称为活跃页面。如果分配给一个进程的物理页面太少使该进程所需的活跃页面不能全部装入内存则进程在运行过程中将频繁发生中断。如果能为进程提供与活跃页面数相等的物理页面数则可减少缺页中断次数。

工作集：一个进程当前正在使用的页框集合。

$w(t,\Delta)$:t时刻该进程在过去$\Delta$个虚拟时间单位中访问到的页面的集合，以进程运行时间为准，不考虑挂起时间。

基本思路：找出一个不在工作集中的页面并置换之。

实现：

  • 每个页表项中有一个字段记录该页面最后一次被访问的时间，设置一个时间值T，根据一个页面的访问时间是否落在当前时间-T之前或之中决定其在工作集之内或之外。
  • 定期置零所有R位
  • 除了R和M，还有每个页表项要对应一个上次使用时间的域（不用放在也表中）
  • 当缺页中断发生时，扫描整个页表。在扫描时，检查R位，如果为1就把当前时间写进页表项的上次使用时间；如果R是0则表示在当前时钟周期中，该页面还没有被访问过，不更改表项对应的上次使用时间。计算生存时间判断是否处于工作集中，置换发现的第一个不在工作集中的页，发现之后继续扫描修改其他页。

特殊情况：

  • 所有页面都在工作集中就淘汰R=0且生存时间最长的页面。
  • 在最坏情况下，在当前时钟滴答下，所有的页面R=1，此时随机选择一个页面淘汰。

缺点：当缺页中断发生后需要扫描整个页表。

3.4.10 工作集时钟算法

设计思路：

  • 当找到一个R=0并且生存时间大于阈值的页时
    • 如果此页面被修改过，就不能立即申请页框，并调用I/O，将此页写回。
    • 指针继续往前走，算法继续对下一个页面进行操作。
    • 找一个R=0、生存时间大于阈值、没有被修改的页。
  • 如果指针经过一圈返回起始点：
    • 如果至少调度了一次写操作，说明指针移动过程中最终会有某个写操作完成，下次再到这个页可以直接被置换。
    • 如果没有调度过写操作，说明所有的页面都在工作集中，随机置换一个干净的页面来使用。

类比优先级(0,0)，(0,1)，(1,0)，(1,1)。

2.1 进程与线程

2.1.1 进程

2.1.1.1 进程模型

  1. 多道程序设计：允许多个程序同时进入内存并运行（目的是提高系统效率）。

  2. 并发环境：①一段时间间隔内由多个程序运行；②看上去同时；③次序不是事先确定的。

  3. 多道程序设计模型：利用多进程来提高CPU利用率

  4. CPU的利用率=$\frac{CPU运行时间}{总时间}$.

  5. CPU利用率评价公式：$\rho=\frac{等待I/0操作的时间}{停留在内存中的时间}$。当内存中同时有n个进程时，则所有n个进程都在等待I/0的概率为$\rho^n$，此时CPU利用率为$1-\rho^n$。

2.1.1.2 进程的状态

  1. 进程的状态：运行态、就绪态、阻塞态。

  2. （1：OS发现进程不能继续运行如等待I/O；2：操作系统调度；3：操作系统调度；4：当进程等待的一个事件发生时如一些输入到达）

2.1.1.3 进程控制块

  1. PCB（进程控制块/进程描述符/进程属性）：OS用于管理控制进程的数据结构，记录进程属性、描述动态变化过程；是系统感知进程存在的唯一标志（进程与PCB一一对应）。
  2. 进程表：所有进程PCB的集合；一般由链表实现（所有进程、就绪态进程…）
  3. PCB包含的信息：
    • 进程标识符
    • 进程名
    • 用户标识符
    • 进程组关系
    • CPU现场信息：寄存器值（通用寄存器、程序计数器、程序状态字、栈指针）、指向该进程页表的指针
    • 所拥有的资源和使用情况：虚拟地址空间的状况、打开文件列表
    • 进程控制信息：当前状态、优先级、代码执行入口地址、运行统计信息（执行时间、页面调度）、进程间同步与通信、进程的队列指针、进程的消息队列指针

2.1.1.4 进程的控制

进程的控制由特定功能的原语（完成某种特定功能的一段程序，具有不可分割性和不可中断性，源于的执行必须是连续的，在执行过程中不允许被中断）完成以防止出现不可预知的问题。

  1. 进程的创建（驱动：系统初始化、正在运行的程序执行了创建进程的系统调用、用户请求创建一个新进程、一个批处理作业的初始化）

    • 分配标识和PCB
    • 分配地址空间
    • 初始化PCB
    • 设置PCB默认值
    • 把新进程添加到就绪队列（PCB添加到就绪队列链表）

    [例]UNIX : fork()系统调用

    • 为子进程分配一个空闲的进程描述符
    • 分配给子进程唯一标识 P ID
    • 复制父进程地址空间
    • 从父进程继承共享资源如打开文件和当前工作目录
    • 将子进程设为就绪态插入到就绪队列
    • 对子进程返回标识符0
    • 向父进程返回子进程的P ID

    UNIX : execve()

    • 对当前进程进行替换，替换者为一个制定的程序（参数：文件名称、参数列表、环境变量）
    • 系统把代码段替换成新的程序的代码
    • 废弃原有的数据段和堆栈段并为新程序分配新的数据段和堆栈段
    • 仅保留原有进程号

    WINDOWS下进程创建：CreateProcess()

  2. 进程的层次结构

    • Linux：进程树、子进程继承父进程很多内容、父进程关闭导致子进程ppid的改变、最初进程为init
    • WINDOWS：更加扁平化、父进程关闭不影响子进程、句柄

2.1.2 线程

线程是一种轻量级的进程，一个进程拥有一个或多个线程，这些线程拥有共享的资源如地址空间和文件等，线程具有自己的堆栈和寄存器环境以保证独立运行的最小要求。

2.1.2.1 引入线程的原因

  1. 应用的需要：仅具备进程的情况下用户体验较差，卡顿。
  2. 开销的考虑：进程操作时间/空间开销过大，限制了并发度的提高；而创建/撤销一个新线程和切换线程花费时间少，同一进程内的线程之间相互通信无需调用内核。
  3. 性能的考量：线程提高了并发的粒度以适应同一进程内处理多个并行任务的需求，从进程并发到线程并发；适应多核CPU。

2.1.2.2 线程模型

线程是操作系统能够运行调度的最小单位，含于进程之中，是进程中的实际运作单位；而进程是系统分配资源的基本单位，是资源拥有者。

2.1.2.3 线程机制的基本实现

  1. 用户空间中实现线程

    1. 实现方式

      • 在用户空间建立线程库用来提供一组管理线程的过程

      • 在用户空间实现运行时系统用来完成线程的管理工作

      • 内核管理进程，线程对内核透明

      • 线程切换不需要内核态的特权

      • 一个内核PCB对应多个线程（1 : N）

    2. 优点

      • 线程切换快
      • 调度算法是应用程序特定的
      • 用户级线程可以运行在任何实现线程库的操作系统上

    3. 缺点

      • 大多数系统调用是阻塞的。因此，由于内核阻塞进程从而进程中所有线程也被阻塞

  2. 内核空间中实现线程

    1. 实现方式

      • 内核管理所有线程，并且向应用程序提供API接口
      • 内核维护进程和线程的上下文
      • 线程切换需要内核支持
      • 以线程为基本单位进程调度
      • 一个控制单位BLOCK对应一个线程（1 : 1）

  3. 混合实现（N : M）

  4. ==调度程序激活技术==

    • 目的：改进内核级线程的速度；避免在用户空间和内核空间之间不必要的转换
    • 需要一个运行时系统：内核空间可以看到这个运行时系统并在需要时控制之（上行调用），运行时系统的效率更高

  5. ==弹出式线程==

    • 一个线程阻塞等待消息->消息到来之后再创建线程

2.2 调度

2.2.1 调度简介

  1. 进程行为：计算密集型、I/O密集型。由于I/O密集型进程数量的增加导致研究方向从提高CPU利用率到提高I/O利用率。

  2. 调度时机

    1. 非抢占式：进程可以一直执行直到进程结束或者发生阻塞。
    2. 抢占式：进程可以被操作系统打断；每个进程不能一直占用太长时间CPU，如有高优先级进程就绪也可以抢占CPU。

  3. 分类

    1. 批处理
    2. 交互式
    3. 实时

  4. 进程切换：一个进程让出处理器并由另一个进程占用处理器的过程。进程切换的频率要恰当。

    • 切换全局页目录以加载一个新的地址空间
    • 切换内核栈和上下文
      • 保存进程A的上下文环境（程序计数器、程序状态字）
      • 用新状态和其他相关信息更新过程A的PCB
      • 把进程A移动到合适的队列
      • 把进程B的状态设置为运行态
      • 从进程B的PCB中恢复上下文环境

  5. 目标

    1. 所有系统
      • 公平——给每个相似进程公平的CPU份额
      • 策略强制执行——保证规定的策略被执行人
      • 平衡——保持系统的所有部分都忙碌
    2. 批处理系统
      • 吞吐量——每小时最大作业数
      • 周转时间——从提交到终止间的平均时间
    3. 交互式系统
      • 响应时间——快速响应请求
      • 均衡性——满足用户期望
    4. 实时系统
      • 满足截止时间——避免丢失数据、出现事故
      • 可预测性——在多媒体系统中避免品质降低

2.2.2 批处理系统中的调度——吞吐量、周转时间

2.2.2.1 先来先服务(FCFS First Come First Serve) 非抢占式

  1. 算法：按照进程创建的次序运行
  2. 优点：最简单
  3. 缺点：长进程后面的短进程需要等待很长时间，不利于用户体验

2.2.2.2 最短作业优先(SJF Shortest Job First) 非抢占式

  1. 算法：（假设可以精确预测每个作业运行时间、作业之间相互独立、同时运行）选择用时最短的作业优先执行
  2. 优点：平均等待时间、平均周转时间最短
  3. 缺点：对长作业不利，会导致饥饿；估计时间不一定真的做到最短作业优先

2.2.2.3 最短剩余时间优先(SRTN Shortest Remaining Time Next) 抢占式

  1. 算法：当一个新就绪的进程比当前运行进程具有更短的完成时间时，系统抢占该进程，选择新就绪的进程执行，是最短作业优先的抢占式版本
  2. 优点：在所有进程同时可运行时，最短的平均周转时间
  3. 缺点：不公平，可能使长的任务长时间得不到运行产生饥饿现象；无法精确直到一个作业的运行时间；完全未考虑作业的紧迫程度因而不能保证紧迫性作业被及时处理

2.2.2.4 最高响应比优先(HRRN Highest Response Ratio Next)

  1. 算法：调度时，首先计算每个进程的响应比R。此后总是选择R最高的进程执行。

    $响应比R=\frac{周转时间}{处理时间}={处理时间+等待时间}{处理时间}=1+\frac{等待时间}{处理时间}$

  2. 优点：克服饥饿，兼顾长作业。（等待时间相同，处理时间越短的优先处理；处理时间相同，等待时间越长的优先处理。）

2.2.3 交互式系统中的调度——响应时间、均衡性

2.2.3.1 轮转调度(RR Round Robin)

  1. 算法：将所有就绪进程按先来先服务的原则排成一个队列，每次调度时把CPU分配给队首进程并令其执行一个时间片，当这个时间片结束时强迫这个进程让出处理器并排列到就绪队列的尾部等候下一次调度。

    ==时间片应略大于一次典型的交互所需要的时间==

  2. 优点：公平；有利于交互式计算，响应时间快

  3. 缺点：由于进程切换，时间片轮转算法花费较高开销（假设时间片 10ms，如果进程切换花费 0.1ms，CPU开销约占 1%）

2.2.3.2 最高优先级调度(HDF Highest Priority First)

  1. 算法：在进程控制块PCB中加入优先级属性，选择优先级最高的进程投入运行，优先级可以是静态不变的也可以是动态调整的。
  2. 优点：实现简单
  3. 缺点：不公平

[例]动态优先级调度

  • 每个进程的优先级设置为$1/f$

    • $f=\frac{历史占用CPU时间}{历史I/O阻塞时间+历史占用CPU时间}$
    • 如果存在时间片，$f$为上一次该进程时间片中CPU运行时间
    • $I/O$型进程优先级应该比较高

  • 优先级调度+时间片轮转

    • 把一组进程按照优先级分为若干类
    • 处于同一优先级类内部的进程按照时间片轮转方式调度
    • 不同优先级类的进程按照最高优先级调度

  • 特点

    • 对于优先级队列i，只有当高优先级的队列为空时，该队列的进程才有可能运行
    • 若该队列存在多个进程，这些进程以时间片轮转的方式平等运行
    • 高优先级进程会抢占当前进程

2.2.3.3 多级反馈队列(Multiple Feedback Queue)

  1. 算法：
    1. 设置多个就绪队列，第一级队列优先级最高（允许进程在队列间移动）
    2. 给不同就绪队列中的进程分配不同长度的时间片，第一级队列时间片最小；随着队列优先级的降低时间片增大
    3. 当第一级队列为空时在第二级队列调度，以此类推
    4. 各级队列按照时间片轮转的方式进行调度
    5. 当一个新创建进程就绪后，进入第一级队列
    6. 进程用完时间片而放弃CPU进入下一级队列
    7. ==由于阻塞而放弃CPU的进程进入相应的等待队列，一旦等待的时间发生该进程就返回原来一级就绪队列==
  2. 优点：
    • 终端型用户：终端型用户提交作业通常较小，系统只要能使这些进程在第一队列所规定的时间片内完成便可使终端型用户满意
    • 短进程用户：周转时间依然较短
    • 长进程用户：用户不必担心其进程长期得不到处理
  3. 缺点：
    • 对于需要长时间运行并且持续交互的进程不友好（进程出现交互即放入最高优先级；间歇敲击键盘可使进程优先级永远处于最高态）

2.3.3.5 管程

  1. 定义：一个管程是一个由过程、变及数据结构等组成的一个集合，它们组成一个特殊的模块或软件包。

    注：可以理解为一个类；进程可在任何需要的时候调用管程中的过程，但不能在管程之外声明的过程中直接访问管程内的数据结构。

  2. 特性：==多个进程互斥访问一个管程==，是编程语言的组成部分。

    • 管程内部的共享变量
    • 互斥访问的过程
    • 管程内部的条件变量

  4. 解决的问题：

    1. 互斥
      • 管程中的过程是互斥进入的，保证管程中数据结构的完整性
      • 管程的互斥性是由编译器负责保证的
    2. 同步
      • 管程中设置条件变量以及等待/唤醒操作以解决同步问题
      • 可以让一个进程或线程在条件变量上等待（释放管程互斥权），也可以通过发送信号将等待在条件变量上的进程或线程唤醒

    对于条件变量可以执行wait和signal操作。

    1. wait(c)：被一个访问管程的进程或线程调用，调用后这个进程或者线程将被挂起，并等待某个事件（断言）被满足，此后方可恢复执行。进程挂在该条件变量上等待时，其他进程可进入管程。
    2. signal(c)：被一个进程调用，以指出事件（断言）现在为真，从而可以唤醒等待该事件被满足的进程继续执行。

  6. 注意：

    1. 当一个进入管程的进程执行wait操作时应当释放管程的互斥权，

    3. 处理策略：

      • P0等待P1执行——Hoare管程
      • P1等待P0继续执行——MESA管程
      • 规定唤醒操作为管程中最后一个可执行的操作——Hansen管程

2.3.3.6 信号量与管程辨析

  1. 信号量和管程都能解决编程中的同步互斥问题。二者本质上互通，可以用信号量实现管程、也可以用管程实现信号量。

  6. 信号量的V操作如果唤醒其他线程，当前线程和被唤醒线程并发执行；对于管程的signal操作，要么当前线程继续执行，要么被唤醒线程继续执行，二者不可并发

2.4 死锁

2.4.1 基本概念

  1. 定义：一组进程中，每个进程都无限等待被该组进程中另一个进程所占有的资源，因而永远无法得资源，这种现象称为进程死锁，这一组进程称为死锁进程。（参与死锁的所有进程都在等待资源）
  2. 产生死锁的==必要条件==：
    1. 互斥使用（资源独占）：一个资源每次只能给一个进程使用
    2. 占有且等待（请求和保持；部分分配）：进程在申请新的资源的同时保持对原有资源的占有
    3. 不可抢占（不可剥夺）：资源申请者不能强行从资源占有者手中夺取资源，资源只能由占有者自愿释放
    4. 循环等待：形成一个等待环路

2.4.2 死锁建模

二元组G=(V,E)

V：结点的集合，分为P（进程）、R（资源）两部分

E：有向边的集合($P_i,R_j$)、($R_i,P_j$)

分配（占有）边：资源->进程

申请（请求）边：进程->资源

2.4.3 死锁定理

如果资源分配图中没有环路，则系统中没有死锁；如果图中存在环路，则系统中可能存在死锁。

如果每个资源类中只包含一个资源，则环路是死锁存在的充分必要条件。

2.4.4 处理死锁的策略

  1. 忽略：假装不存在死锁/鸵鸟算法/系统因为其他原因崩溃的频率远远大于死锁

  2. 检测并恢复

    1. 利用资源分配图：

      1. 首先利用资源分配图来检测死锁（资源分配图中的环路是死锁的必要条件/如果每个资源类中只包含一个资源，则环路是死锁存在的充分必要条件）
      2. 对于有环路且资源类存在多个资源的情况
        1. 找一个请求资源可以满足的进程结点，删除结点并去掉分配边
        2. 重复上述步骤，如果可以删除所有进程则不存在死锁

    2. 利用矩阵表示资源分配图

      A向量：可用的资源数（一个元素一种资源）

      R矩阵：请求的资源数（一行对应进程、一列对应资源）

      C矩阵：占用的资源数（一行对应进程、一列对应资源）

      检测步骤：

      1. 寻找一个没有标记的进程$P_i$，对它而言R矩阵的第i行小于或等于A
      2. 如果找到了这样的一个进程，那么将C矩阵的第i行加到A中，标记该进程，转到第(i)步
      3. 如果没有这样的进程，则算法终止

      算法结束时，所有没有标记过的进程都是死锁进程。

    3. 利用抢占恢复：挂起一些进程并剥夺其资源；需要人工干预；可行性往往取决于资源特性和具体应用

    4. 利用回滚恢复：周期性设置检查点（备份进程的各种状态放入一个文件中）；一旦检测到死锁则回滚到前一个检查点，在该检查点之后所做的工作都丢失；如果该死锁仍存在则继续回滚

    5. 通过杀死进程恢复：

      进程的选择：

      1. 环中一个进程：如果仍然死锁则继续选择
      2. 环外一个进程：应当正好持有环中某些进程所需资源
      3. 尽量找不会出现副作用的进程/有些进程杀死后重新运行还会引起死锁

  3. 死锁避免

    每次资源分配的时候西安进程判断，如果分配后存在死锁风险则不分配，反之则分配。

    死锁风险的判断：

    1. 寻找一个没有标记的进程$P_i$，对于它而言R矩阵的第i行向量小于或等于A
    2. 如果找到了这样一个进程，那么将C矩阵的第i行向量加到A中，标记该进程并转到第(a)步
    3. 如果没有这样的进程则算法终止

    对于R矩阵：

    1. 死锁检测：申请的资源数量
    2. 死锁避免：预计所需最大资源数

    银行家算法：对每一个请求进行检查，监察如果满足这一请求是否会达到安全状态。若是，就满足该请求；若不是，则推迟对这一请求的满足。

  4. 死锁预防

    1. 破坏“互斥使用”条件

      资源转换技术：把独占资源变为共享资源

      虚拟化：SPOOLing技术的引入

      • 解决不允许任何进程直接占有打印机的问题
      • 设计一个守护进程/线程负责管理打印机，进程需要打印时，将请求发给守护进程，由守护进程完成打印任务

    2. 破坏“占有且等待条件

      实现方案1：要求每个进程在运行前必须一次性申请要求的所有资源，且仅当该进程所需资源均可满足时才给予一次性分配（==资源利用率低/饥饿现象==）

      实现方案2：当一个进程请求资源时，先释放当前占用的所有资源，然后再尝试一次获得所需的全部资源

    3. 破坏“不可抢占”条件

      实现方案：当一个进程申请的资源被其他进程占用时，可以通过操作系统抢占这一资源（两进程优先级不同）

      局限性：适用于状态易于保存和恢复的资源

    4. 破坏“循环等待”条件

      实现方案：把系统中所有资源编号，进程在申请资源时必须严格按照资源编号的递增次序进行，否则操作系统不予分配（很难确定现实中的编号次序）

2.4.5 其他死锁相关问题

  1. 两阶段加锁：

    1. 适用：数据库等
    2. 实现
      1. 第一阶段试图把所需资源都加上锁
      2. 第一阶段成功，才开始第二阶段——完成更新并释放
      3. 如果不成功，则释放第一阶段的所有资源并延期重新开始

  2. 通信死锁

    1. 死锁不一定必须与资源相关
    2. 在通信系统中
      • 进程A向进程B发送请求信息，然后阻塞直至B回复
      • 假设请求信息丢失，则A将阻塞以等待回复
      • 而B会阻塞等待一个向其发送命令的请求，因而发生死锁
    3. 通信死锁不能通过对资源排序避免
    4. 解决方案：超时重发机制（没有回复一段时间后重新发送）

  3. 活锁：任务或者执行者没有被阻塞，由于某些条件没有被满足导致一直重复尝试——失败——尝试——失败的过程


    活锁与死锁的区分：死锁是多个进程占用了资源导致的而活锁不是。

  4. 饥饿：由于调度策略等原因导致某些进程一直得不到运行而不是由于资源被其他进程占用。

3.2 地址空间

3.2.2 交换技术

1.出现交换技术的原因

  • 单个进程所需内存越来越大。
  • 多进程下所需内存非常大。
  • 进程往往会动态增加。
  • 所有进程都放入内存中往往是不现实的，会出现内存超载。

为了解决内存超载的问题。

2.内存紧缩

交换在内存中产生了多个空闲区，容易造成浪费，因此需要定期进行内存紧缩。

把所有的进程尽可能向下移动，将这些小的空闲区块合并成一个大块。但是需要耗费大量的CPU时间。

3.进程创建时分配内存大小

  • 若进程创建时大小是固定的且不在改变，操作系统只需准确的按其需要的大小进行分配。

  • 如果进程的数据段可以增长则会出现问题，如果相邻空闲区则可以直接使用，若相邻另一个进程则需要把增长的进程转移到内存中一个足够大的区域，如果交换区也满，则这个进程只有一直挂起直到一些空间空闲。

4.碎片——不能利用的内存空间

  • 内碎片：已经分配给作业但不能被利用的内存空间

  • 外碎片：还没有分配给作业但是由于太小而无法分配给申请内存空间的新进程的存储块

3.2.3 空闲内存管理

1 固定分区分配

1.实现方式

  • 将内存空间划分为若干固定大小的分区

  • 每个分区装入一道程序

  • 分区大小可以不等，但必须事先确定，在运行时不能改变

  • 当有空闲分区时，从后备队列选择一个适当大小的作业装入运行

2.分区说明表：用以记录可用于分配的分区号、分区的大小、分区的起始地址及状态（通常按照分区大小顺序排序）。

3.优点：是一种最简单的存储分配方式。

4.缺点：利用率较低，会产生内碎片。

2 位图

1.实现方式

  • 内存被划分为若干的分配单元

  • 每个分配单元对应位图中的一位

对于上图，P表示已分配、H表示空闲，第一个表项中0表示起始地址、5表示长度/大小。

2.分配单元大小的影响

  • 分配单元越小，位图越大，分配粒度小，浪费少。
  • 分配单元越大，位图越小，分配粒度大，浪费大。

3.缺点

进程调入内存时，存储管理器必须搜索位图。

3 链表

1.实现方式：维护一个记录已分配内存段和空闲内存段的链表（使用双向链表效果更佳）。

2.优点：进程终止或者被换出时链表更新直接。

4 基于链表的空闲内存适配算法

1 首次适配

算法：存储器沿着链表进行搜索，直到找到一个足够大的空闲区，除非空闲区大小和要分配的空间大小正好一样，否则将该空闲区分为两部分，一部分供进程使用，另一部分形成新的空闲区。

优点：速度很快，尽可能少地搜索链表结点。

缺点：形成大量外碎片。

2 下次适配

算法：每次找到合适的空闲区都记录当时的位置，以便在下次寻找时从上次结束的地方开始搜索。

优点：外碎片略少于首次适配。

缺点：性能略低于首次适配算法。

3 最佳适配

算法：搜索整个链表，找出能够容纳进程的最小的空闲区。

优点：找到最接近实际需要的空闲区。

缺点：比首次适配算法慢；可能比首次适配和下次适配算法浪费更多的内存，产生大量无用的小空闲区。

5 最差适配

算法：总是分配最大的可用空闲区，使新的空闲区比较大从而可以继续使用。

缺点：浪费内存多

注：如果进程和空闲区使用不同的链表，则可以按照大小对空闲区链表进行排序，提高适配速度。空闲区链表按大小排序时，首次适配算法和最佳适配算法一样。

6 快速适配

算法：为那些常用大小的空闲区维护单独的链表。

优点：寻找一个指定大小的空闲区十分快速。

缺点：在一个进程终止或者被换出时，寻找它的相邻快并查看是否可以合并的过程非常费时；如果不合并，内存很快分裂出大量的进程无法利用的小空闲区。

7 伙伴系统

主要思想：将内存按2的幂次进行划分，组成若干空闲链表，查找链表找到满足进程的最佳匹配块。

算法：

  1. 将可用空间看作一块$2^u$
  2. 假设进程申请s
  3. 若$2^{u-1} \lt s \le 2^u$，则分配；否则将该块划分为两个$2^{u-1}$重复上述过程。
8 slab机制

主要思想：针对一些经常分配并释放的对象（如进程描述符，大小比较小，并且如果直接采用伙伴系统容易产生内碎片并且处理速度慢）。

slab分配器基于对象管理，将相同类型的对象归为一类。每当要申请这样一个对象，slab分配器就从一个slab列表中分配一个这样大小的单元出去，当要释放时，将其重新保存在该列表中，而不是直接返回伙伴系统，避免这些内碎片。

3.3 虚拟内存

3.3.0 前导

1 背景

交换技术很难适用于程序大于内存的情况并且可能发生一个进程占用一整块内存的情况。

2 覆盖技术的实现

  • 把程序分割成许多片段，称为覆盖。
  • 程序开始执行时，将覆盖管理模块装入内存，该覆盖管理模块立即装入并运行覆盖0
  • 执行完成后，覆盖0通知管理模块装入覆盖1（如果有空间就占用覆盖0上方的位置；如果没有空间就占用覆盖0的位置）

3 存在问题

程序员必须自行分割程序，费时且容易出错。

3.3.1 分页

1 基本概念

0.由程序产生的地址称为虚拟地址，它们构成了一个虚拟地址空间。在没有虚拟内存的计算机上，系统直接将虚拟内存送到内存总线上，读写操作使用具有同样地址的物理内存字，而在使用虚拟内存的情况下，虚拟地址不是被直接送到内存总线上，而是被送到内存管理单元MMU，MMU把虚拟内存映射为物理内存地址。

1.用户进程地址空间被划分为大小相等的部分，称为页或页面，从0开始编号。

2.内存空间按照同样大小划分为大小相等的区域，称为页框，从0开始编号，也称为物理页面、页帧、内存块。

3.内存分配规则：以页为单位进行分配，并按进程需要的页数来分配；逻辑上相邻的页在物理上不一定相邻。

4.逻辑地址=页号+页内地址

5.划分是由系统自动完成、对用户透明的。

2 缺页

解决虚拟地址空间比内存地址空间大的问题：当程序访问了一个未被映射的页面，MMU（通过页表项中的在/不在位）注意到该页面没有被映射，于是CPU陷入到操作系统发生缺页中断/缺页错误。操作系统找到一个很少使用的页框且把它的内容写入磁盘（如果它不在磁盘上）。随后把需要访问的页面读到刚才回收的页框中，修改映射关系，然后重新启动引起陷阱的指令。

页号作为页表的索引以得出对应于该虚拟页面的页框号。如果在/不在位为0，则引起一次操作系统陷阱。如果为1，则在该页表中查到页框号复制到输出寄存器的高三位中，再加上虚拟地址中的低12位偏移量就构成了15位的物理地址。输出寄存器的内容随即被作为物理地址送到内存总线。

3.3.2 页表

1 基本概念

1.页表是一张表，负责实现虚拟地址到物理地址的映射，每个进程一个页表存放在内存。

2.虚拟地址=虚拟号+偏移量

3.虚拟号可以作为页表的索引，索引到具体的页表项。

2 页表项的结构

页表项：页表项的结构与机器密切相关，但不同机器的页表项存储的信息都大致相同。

  • 页框号：；页映射的目的。
  • 在/不在位：决定缺页中断
  • 保护位：决定读写
  • 修改位：记录当前页面是否被修改/决定换回硬盘是否需要被写会
  • 访问位：记录当前页面是否正在访问

注：(1)利用页表的查找和地址转换是硬件完成的。(2)页面在磁盘中的地址不在页表项中（因为缺页中断是软件完成的事情，而页表是硬件自动访问的）

3.3.3 加速分页过程

0 前导&最简单的实现

虚拟地址到物理地址的映射必须非常快；如果虚拟地址空间很大，页表也会很大。

实现方式1：使用一个快速页表寄存器阵列，把页表放入寄存器中。启动一个进程的时候，页表装入此寄存器阵列中。

  • 优点：页表查找速度最快。

  • 缺点：(1)代价高昂。(2)页表要一次性全部读入。

实现方式2：页表全部在内存中。

  • 优点：代价相对低一些。
  • 缺点：页表查找速度相对较慢。

1 转换检测缓冲区（相联存储器/快表）

1.目的：为了加速页表查找（相当于寄存器和内存之间的cache）

2.程序访问的局部性原理：大多数程序总是对少量的页面进行多次的访问；只有很少的页表项会被反复读取，而其他的页表项很少被访问。

3.设计思路：设置一个小型的硬件设备，将虚拟地址直接映射到物理地址而不必在访问页表。通常直接将其放在MMU中作为相联存储器，按内容并行查找，是cache的一种。

4.特征：与页表项中的记录一一对应；多了一个虚拟页号（在页表项中由于索引就作为了虚拟页号所以不是必须的）；如果不在TLB中则直接查找页表并替换TLB中的一行。

5.软件TLB管理

方式演变：

  • 过去的计算机中TLB和页表的管理完全由处理器硬件负责（对软件透明）
  • 现代的精简指令集处理器可能不再完全由硬件处理。由此减少处理器和指令执行的复杂度，减少MMU的实现代价；此外，用软件来判断哪些页表项放入TLB可能更加灵活和有效（增加TLB的命中率）

可能出现的问题：

  • 软失效：页表项不在TLB中，需要更新TLB，不需要产生磁盘I/O。需要10-20个机器指令并花费几纳秒完成操作。
  • 硬失效：页面本身不在内存中，此刻需要一次磁盘存取以装入该页面。需要几毫秒，处理时间是软失效的百万倍。

硬失效在实际情况中由三种可能性：

  • 所需的页面可能就在内存中，但却未记录在进程的页表中。
  • 需要从硬盘中重新调入页面，发生缺页错误。
  • 程序访问了一个非法地址，无需向TLB中新增映射。

3.3.4 针对大内存的页表

0 前导

1.页表需要连续存放，每个进程都有自己的页表。

2.例题：

  • 假设一个页表项8B，进程虚拟地址空间为4GB内存，页大小为4KB，一个进程页表需要多大空间？答案：4GB/4KB*8B
  • 64位计算机中，假设进程地址空间为$2^{48}$，一个页表项8B，页大小为4KB，一个进程页表需要多大空间？答案：$2^{48}$B/4KB*8B

1 多级页表

优点：

  • 不需要线性存储
  • 只需要部分页表存在内存（或TLB）即可
  • 增加级数可以很好地处理更大内存的情况

缺点：

  • 访问页表的次数增加

增加页表的级数本质上是允许页表非线性存放而不能减少页表项数。

2 倒排页表

思路：

  • 每个页表项对应一个页框
  • 之前是每个页表项对应一个页

优点：

  • 当虚拟地址空间比物理内存大得多的时候可以节省大量的空间

缺点：

  • 查表速度慢

实现方式：从物理地址空间出发，系统建立一张页表，页表项记录进程i的虚拟地址（虚页号）与页框号的映射关系，整个系统一张表。系统的每个地址有一个三元组<process-id,page-number,offset>

根据process-id和page-number从page table找到对应的页框号，加上offset到物理内存找查找。

3.4 页面置换算法

3.4.0 前导

最佳算法->先进先出->第二次机会->时钟->最近未使用->最近最少使用->最不经常使用->老化算法->工作集->工作集时钟

<设计思想，算法实现，算法应用>

颠簸：虚拟内存中，页面在内存与磁盘之间频繁调度，是的调度页面所需的时间比进程实际运行的时间还多，导致系统效率急剧下降，这种现象称为颠簸或抖动。

3.4.1 最佳页面置换算法

设计思想：置换以后不再需要或者最远的将来才会用到的页面。

优点：效果最好。

缺点：由于操作系统无法知道各个页面下一次将在什么时候被访问，所以无法实现。

应用：作为一种标准来衡量其他算法的性能。

3.4.2 先进先出置换算法

设计思想：由操作系统维护一个所有当前在内存中的页面的链表，最新进入的页面放在表尾，最早进入的页面放在表头。当发生缺页中断时，淘汰表头的页面并把新调入的页面加到表尾。

优点：实现简单。

缺点：几乎没有考虑页面的使用频率以及程序访问的局部性原理。

3.4.3 第二次机会算法

设计思想：页表项新增一个访问位R，每次MMU访问该页表项时R置为1。按照先进先出算法选择某一页面，检查其访问位R，如果为0则置换该页面；如果为1将R置为0并放到链表尾部（若修改后不放回队尾，则修改后在首部的页面很容易被再次找到被置换，不公平）。

优点：考虑了使用频率和局部性原理。

缺点：经常要在链表中移动页面，既降低了效率又不是很有必要。

3.4.4 时钟算法

算法：把所有页面都保存在一个类似钟面的环形链表中，当发生缺页中断时，算法首先检查表针指向的页面，如果R位是0就淘汰该页面并把新的页面插入这个位置，把表针前移一个位置，如果R位是1就清除R位并前移一个位置，重复上述过程直到找到一个R位为0的页面位置。

3.4.5 改进型时钟算法

规定：

  • 最近未访问 最近未修改 （0,0）

  • 最近未访问 最近被修改 （0,1）

  • 最近被访问 最近未修改 （1,0）

  • 最近被访问 最近被修改 （1,1）

算法：

  • 从指针的当前位置开始，扫描循环链表。在这次扫描过程中，对访问位和修改位不做修改。选择遇到的第一个（0,0）页面用于替换。
  • 若上一步没有找到，则重新扫描。寻找（0,1）页面用于替换，在找的过程中将R置为0。
  • 如果上一步仍没有找到，再次执行前两步。

优点：考虑了页面载入内存后是否被修改的问题，减少了I/O次数。

缺点：增加扫描次数。

3.4.6 最近最少使用算法

设计思想：选择在最近一段时间未使用过的一页并置换（向前看；而最佳置换算法是向后看）。

优点：充分考虑了程序访问的局部性。

缺点：费时；难以实现；需要硬件支持。

1 实现方式1

要求硬件有一个64位计数器C，在每条指令执行完后自动加1，每个页表项必须有一个足够容纳这个计数器值的域。在每次访问内存后，将当前的C值保存到被访问页面的页表项中。一旦发生缺页中断，操作系统就检查所有页表项中计数器的值，找到值最小的一个页面（最近最少使用的页面）。

2 实现方式2

设置页表表项的两位——访问位R和修改位M，初始值为0，访问一个页面置1，修改一个页面置1，定期让所有的页面R置0。

发生缺页中断时，操作系统检查R、M。随机从编号最小的非空类中选择一页计算。

规定：

  • 第1类 无访问 无修改 （0,0）
  • 第2类 无访问 有修改 （0,1）
  • 第3类 有访问 无修改 （1,0）
  • 第4类 有访问 有修改 （1,1）

3.4.7 最不常用算法

算法：算法将每个页面与一个软件计数器相关联，计数器的初始值为0.每次时钟中断时由操作系统扫描页表中所有的页面，将每个页面的R位（0或1）加到它的计数器上。这个计数器大体上跟踪了所有页面被访问的频繁程度。发生缺页中断时，置换计数器值最小的页面。在时钟中断时由中断处理程序实现，是一种软件实现方式，计数器不用放在页表中。

缺点：最近经常使用的优先权小于之前经常使用的，而之前经常使用的由于一直换不出去导致对应的计数器值一直增大。

3.4.8 老化算法

算法：在最不常用算法基础上进行修改。首先，在R位被加进之前先将计数器右移一位；其次，将R位计数器加到最左端的位而不是最右端的位。

优点：最接近最近最少适用算法。

3.4.9 工作集算法

设计思想：根据程序的局部性原理，一般情况下，进程在一段时间内总是集中访问一些页面，这些页面被称为活跃页面。如果分配给一个进程的物理页面太少使该进程所需的活跃页面不能全部装入内存则进程在运行过程中将频繁发生中断。如果能为进程提供与活跃页面数相等的物理页面数则可减少缺页中断次数。

工作集：一个进程当前正在使用的页框集合。

$w(t,\Delta)$:t时刻该进程在过去$\Delta$个虚拟时间单位中访问到的页面的集合，以进程运行时间为准，不考虑挂起时间。

基本思路：找出一个不在工作集中的页面并置换之。

实现：

  • 每个页表项中有一个字段记录该页面最后一次被访问的时间，设置一个时间值T，根据一个页面的访问时间是否落在当前时间-T之前或之中决定其在工作集之内或之外。
  • 定期置零所有R位
  • 除了R和M，还有每个页表项要对应一个上次使用时间的域（不用放在也表中）
  • 当缺页中断发生时，扫描整个页表。在扫描时，检查R位，如果为1就把当前时间写进页表项的上次使用时间；如果R是0则表示在当前时钟周期中，该页面还没有被访问过，不更改表项对应的上次使用时间。计算生存时间判断是否处于工作集中，置换发现的第一个不在工作集中的页，发现之后继续扫描修改其他页。

特殊情况：

  • 所有页面都在工作集中就淘汰R=0且生存时间最长的页面。
  • 在最坏情况下，在当前时钟滴答下，所有的页面R=1，此时随机选择一个页面淘汰。

缺点：当缺页中断发生后需要扫描整个页表。

3.4.10 工作集时钟算法

设计思路：

  • 当找到一个R=0并且生存时间大于阈值的页时
    • 如果此页面被修改过，就不能立即申请页框，并调用I/O，将此页写回。
    • 指针继续往前走，算法继续对下一个页面进行操作。
    • 找一个R=0、生存时间大于阈值、没有被修改的页。
  • 如果指针经过一圈返回起始点：
    • 如果至少调度了一次写操作，说明指针移动过程中最终会有某个写操作完成，下次再到这个页可以直接被置换。
    • 如果没有调度过写操作，说明所有的页面都在工作集中，随机置换一个干净的页面来使用。

类比优先级(0,0)，(0,1)，(1,0)，(1,1)。

    3. 数据传输：数据在磁盘与内存之间的实际传输

4.4 文件系统——操作系统实现视角

4.4.1 文件系统的实现

4.4.1.1 文件系统的布局

  1. 文件系统存放在硬盘上。多数硬盘划分为一个或多个分区，每个分区有一个独立的文件系统。

    磁盘的0号扇区/块称为主引导记录（Master Boot Record，MBR），用来引导计算机，在MBR的结尾是分区表。

    文件系统包含的项目；引导块：引导装载在该分区的操作系统（即便没有此分区操作系统也存在）；超级块：文件系统的关键参数如魔数magic number、块的数量等；空闲块的信息；i节点；目录；文件

4.4.1.2 文件的实现

  1. 连续分配：把每个文件作为一连串数据块存储在磁盘上。

    优点：实现简单；操作性能较好

    缺点：随时间推移产生碎片；更适合光盘（只写一次，后续都只读）

  2. 链表分配：为每个文件构造磁盘块链表

    优点：充分利用每个磁盘块，不会因为磁盘碎片而浪费存储空间；顺序读文件方便

    缺点：随机访问慢

  3. ==采用内存中的表进行链表分配==

    在链表的基础上，取出每个磁盘块的指针字并把它们放在内存的一个表上（文件分配表 File Allocation Table，FAT）。

    优点：解决了链表的不足，随机访问快。

    缺点：必须把整个表放在内存中；FAT的管理方式不能较好地扩展并应用于大型磁盘中。

  4. i节点：给每个文件赋予一个称为i节点的数据结构，包括文件属性和文件的磁盘块地址

  5. 只有打开的文件的i节点才在内存中。

    最后一个磁盘地址指向一个包含额外磁盘块地址的块的地址，可以由两个或更多个包含磁盘地址的块。

4.4.1.4 目录的实现

  1. 目录系统的功能：把文件名映射成定位文件数据所需的信息。未见属性可以放在目录里或文件的i节点中。

  2. 文件名的存储：限制长度/固定结尾符/文件名放在堆中

    加速查找：存储文件名的散列值

4.4.1.5 共享文件

  1. 共享文件的方式：链接

  2. 硬链接：直接共享iNode即一个iNode对应多个文件名。

    LINUX命令：link/ln

    特点：

    1. 文件有相同的iNode以及data block
    2. 不能交叉文件系统进行硬链接的创建
    3. 只能对文件创建，不能对目录进行创建
    4. 删除一个硬链接文件并不影响其他有相同iNode号的文件

  3. 软链接/符号链接：文件数据块中存放的内容是另一文件的路径名的指向

    LINUX命令：ln -s

    WINDOWS：快捷方式

    特点：

    1. 软链接文件有自己的文件属性及权限等
    2. 可对不存在的文件或目录创建软链接
    3. 软链接可交叉文件系统
    4. 软链接可对文件或目录创建
    5. 创建软连接时，链接计数器i_nlink不会增加
    6. 删除软链接并不影响被指向的文件，但若被指向的原文件被删除则相关软链接被称为死链接（dangling link，若被指向路径文件被重新创建，死链接可被恢复为正常的软链接）

4.4.1.6 日志结构文件系统（Log-structured File System：LFS）

  1. 基于硬盘的特性

    顺序I/O慢，随机I/O慢：随机I/O比连续I/O性能差很多，如果只用顺序读写就可以大幅度提高I/O效率。写操作占据了全部操作的大部分且零碎。

  2. 设计思路

    不更新任何数据，只提供append操作；建立缓冲区将多次操作放进一个段里一次性写入，从而有效利用顺序I/O

  3. 连续写

    创建一个段，把所有要写入硬盘的操作写入此段中（每次写入的操作都要加入对应i节点的信息）

    LFS：i节点会分散在各个段中，段最终会整体写入硬盘，也就是i节点分散在整个硬盘。iNode map用来顺序存放每个iNode的最新位置，每次写入都更新这个表。

    其他文件系统：i节点在固定位置存放，方便系统访问

  4. 垃圾回收

    不覆盖写任何数据，只是进行append操作，系统会存在同一个数据的很多版本。

    需要定期删除旧版本，从而会产生碎片，因此定期把几个段中的有效信息提取出来组成新的完整的段。

  5. 错误恢复

    设置检查点，若发生错误就从最近的一个检查点恢复。

4.4.1.7 日志文件系统

  1. 提高文件文件系统在出错时候的鲁棒性。

  2. 举例，对于一个文件删除操作，包含

    1. 在目录中删除文件

    2. 释放i节点，把i节点放入空闲i节点池

    3. 将所有磁盘块归还空闲磁盘块池

    日志文件系统先写一个日志项，列出上述三个将要完成的操作。

    将日志项写入磁盘，按照日志执行相应的操作。

    日志中的操作可以反复执行——幂等的。

    原子事务：一组动作可以被界定在开始事务和结束事务操作之间，要么一次性做完，要么什么也不做。

4.4.2 文件系统的管理和优化

4.4.2.1 磁盘空间管理

  1. 分块存储：把文件分为若干块，分别存放在磁盘不同/不连续的位置上

    连续存储：存在一段空间不足而整个文件需要移动的问题，同内存连续分配类似，但磁盘中移动文件效率更低

  2. 关键问题：块大小、记录空闲块、磁盘配额

  3. 块大小——理想状态一块最多给一个文件使用

    1. 块过大：小的文件浪费了大量的磁盘空间

      块过小：大多数文件会跨越多个块，需要多次寻道和旋转延迟才能读出，降低性能，浪费时间。

    2. 考虑指标

      传输数据率：一段时间内，真正在传输的数据比率

      空间使用率：真正在使用的空间比率

    3. 历史1-4KB，目前块提升到64KB并接受浪费。

  4. 记录空闲块

    1. 采用磁盘块链表，链表的每个块中包含尽可能多的空闲磁盘块号。

      当空闲块大多连在一起时，链表方式的每一个表项可以包括第一块和连续空闲块数；但如果空闲块分散，上述方法就很浪费。

4.4.2.2 文件系统备份

  1. 数据恢复：从意外灾难中恢复；从错误操作中恢复

  2. 整体备份和局部备份

    1. 增量转储：只备份修改的数据；周期性急性全面转储，插入增量转储（根据上次全面转储后的修改做备份或根据上次增量转储后的修改做备份）

    2. 压缩：慎重选择，可能损坏难以恢复

    3. 快照：操作系统运行过程中实时监控文件系统的修改，监控的信息称为快照，然后定期按照快照来备份

    4. 备份应远离现场

  3. 转储备份

    1. 物理转储：从第0块开始，将全部的磁盘块按序输出到磁带，没有使用的块无需备份，记录坏块的表需要备份

      优点：简单快速

      缺点：不能根据特定的目录备份或者跳过某些目录；无法增量转储；无法根据文件的性质进行会转储

    2. 逻辑转储：从一个或几个指定的目录开始递归转储，往往考虑某个基准日期后修改的文件，同时也要转储修改过得文件路径上的所有文件夹（为了把转储的内容恢复到一个新的文件系统中；可以对单个文件做增量恢复）

      四阶段的算法：

      1. 标记所有修改过的文件和目录
      2. 去掉不包含修改文件和目录的标记
      3. 转储前面标记的目录并去掉标记
      4. 转储前面标记的文件

4.4.2.3 文件系统的一致性

  1. 操作系统中专门存在一个检测文件一致性的程序，系统启动时运行该程序。

  2. 一致性检查

    1. 块的一致性检查
    2. 目录的一致性检查

    二者集成到一起使用

  3. 块的一致性

    1. 构造两张==位表==，每一位对应一块，初始化为0
    2. 检查每个i节点的块，更新第一张表，出现一块则对应位+1
    3. 检查空闲链表或位图，更新第二个表
    4. 检查两个表是否对应

  4. 目录的一致性

    1. 每个文件设定一个使用计数
    2. 递归搜索每个目录，目录中每个文件对应的计数+1，硬链接也+1，软链接不需+1
    3. 随后看使用计数与每个文件的硬链接数目（iNode中）比较
    4. 不一致则修改iNode中的计数

  5. 除了块一致性和目录一致性外，还需要

    1. 检查目录中iNode的数目是否有意义，不可能大于整个系统允许iNode的数量
    2. 检查iNode中权限是否合理

4.4.2.4 文件系统的性能

  1. 前提：访问硬盘比访问内存以数量级的形式慢。

  2. 改善性能的方法

    1. 高速缓存：内存中开辟一个空间，专门缓存部分硬盘的空间

      1. 以块为单位缓存

      2. 先看访问的块在不在缓冲区中，在则直接访问，不在就从硬盘调入缓冲区，下次如果访问不需要操作硬盘

      3. 往往会预先读，局部性

      4. 硬盘访问不像内存访问十分频繁，可以用设备+磁盘地址做散列，没有MMU类似的模块。

      5. 可能用到页面置换算法，可以使用精确地LRU（为什么）

      6. 需要考虑文件系统的特点：将块分为iNode、间接块、目录、满数据块、部分数据块等；最近不需要的块放在LRU链表的前部（根据块的类型和之前使用的情况来决定最近是否需要）

      7. 有些块需要快速写回以==保障系统的一致性==，减少在系统崩溃后文件系统损坏的可能

        UNIX：提供写回的系统调用以及后台服务

        WINDOWS：只要写入高速缓存就把每个修改的块写入磁盘

      8. 内存映射文件：高速缓存和页式管理集成；按照块缓存->按照页缓存

    2. 块提前读：顺序提前多读一些块到缓冲区

    3. 减少磁盘运动：顺序访问的块放在一起

  3. 磁盘碎片整理

    碎片：磁盘使用较多后空间变得不连续的现象，产生碎片。

    当创建一个新文件时所使用的块会散布在整个磁盘上造成性能的降低。

    1. 磁盘的寻道时间
    2. 固态硬盘：对数据的修改效率低
    3. 同时也和具体的文件系统有关

    因此需要有个专门的磁盘碎片整理程序定期运行。

      1. 并行性
      2. 均衡性（使设备充分忙碌）

5.2 硬件

5.2.1 概述

  1. I/O设备一般由机械和电子两部分组成：

    1. 机械部分是设备本身（物理装置）
    2. 电子部分又称设备控制器（或适配器）
      1. 端口地址译码
      2. 按照主机与设备之间约定的格式和过程接收计算机发来的数据和控制信号或向主机发送数据和状态信号
      3. 将计算机的数字信号转换成机械部分能识别的模拟信号或者将模拟信号转换成数字信号
      4. 实现设备内部硬件缓冲、数据加工等提高性能或增强功能

    上层命令写入控制器的接口寄存器/接口缓冲区，以实现输入/输出并从接口寄存器读取状态信息或结果信息。

    当控制器接受一条命令后，可独立于CPU完成指定操作，CPU可以另外执行其他计算；命令完成时，控制器产生一个中断，CPU响应中断，进而控制转交操作系统；通过读控制寄存器中的信息获取操作结果和设备状态。

    控制器与设备之间的接口常常是一个低级接口。

  2. 控制器的任务

    把串行的位流转换为字节块并进行必要的错误修正。首先，控制器按位进行组装，然后存入控制器内部的缓冲区形成以字节为单位的块，在对块验证检查并证明无错误时，在将其复制到内存中。

  3. I/O端口地址：接口电路中每个寄存器具有的唯一的地址，是一个整数。所有的I/O端口地址形成I/O端口空间。

    I/O指令形式与I/O地址是相互关联的，主要存在两种形式：

    1. 内存映像编址（内存映像I/O模式）
    2. I/O独立编址（I/O专用指令）

  4. I/O独立编址

    1. 分配给系统中所有端口的地址空间完全独立，与内存地址空间无关
    2. 使用专门的I/O指令对端口进行操作
    3. 优点：外设不占用内存的地址空间；编程时易于区分是对内存操作还是I/O端口操作
    4. 缺点：I/O端口操作的指令类型少，操作不灵活

  5. 内存映射编址

    1. 分配给系统中所有端口的地址空间与内存的地址空间统一编址
    2. 把I/O端口看作一个存储单元，对I/O的读写操作等同于对内存的操作
    3. 优点：凡是可对内存操作的指令都可对I/O端口操作；不需要专门的I/O指令；I/O端口可占有较大的地址空间
    4. 缺点：占用内存空间，不容易区分是对内存操作还是对I/O端口操作

  6. I/O设备分类

    1. 块设备：硬盘、光盘等；可按块寻址，每块通常512B-65536B
    2. 字符设备：网络接口、鼠标、打印机、键盘灯；以字符为单位接收或发射一个字符流，不可寻址
    3. 特例：时钟、内存映射的显示器

5.2.2 磁盘

  1. 磁盘的低级格式化：在磁盘能够使用之前，每个盘片必须经受由软件完成的低级格式化，用于确定磁道、扇区等。

    格式化用到的技术：柱面斜进、低级格式化之后再进行分区等工作、交错

  2. 磁盘臂调度算法

    影响读写磁盘块时间的因素：

    1. 寻道时间：将磁盘臂移动到适当的柱面上所需的时间
    2. 旋转延迟：等待适当扇区旋转到磁头下所需的时间
    3. 实际数据传输时间

    当同时出现多个不同磁道磁盘访问请求时确定访问次序的方式：

    1. 先来先服务

      不好。

    2. 最近寻道优先

      不公平，如果磁盘负载很重，那么大部分时间磁盘臂将停留在磁盘的中部区域，而两端极端区域的请求将不得不等待。

    3. 电梯算法

      公平，对任意一组给定的请求，磁盘臂移动的总次数是固定的即柱面数的两倍。

  3. 错误处理

    原因：磁盘总会出现错误的扇区，制造时的瑕疵也不可避免，不能因为有一个错误的扇区就放弃整个磁盘。

    错误的两道防线：

    1. 第一道防线：ECC——磁盘控制器修复，错误太多则无法修复

    2. 第二道防线：再读一次

    坏扇区的处理方式：

    1. 在控制器中处理

      磁盘在出厂之前要进行测试并将一个坏扇区列表写在磁盘上，对于每一个坏扇区用一个备用扇区替换之。

    2. 在操作系统中处理

      首先获得坏扇区的列表（可以读出、也可以自行测试），重新建立映射

    机械臂不准的解决方式：重新校准，移动到边缘。

  4. RAID 磁盘阵列：把多个硬盘组合起来，成为一个或多个硬盘阵列组，逻辑上只有一个虚拟化的硬盘。

    目的：提升性能；数据冗余。

    1. RAID 0 带区集 ：将两个以上的磁盘并联起来，称为一个大容量的磁盘。在存放数据时，分段后分散存储在这些磁盘中，读写可以并行处理。

      ==在所有级别的RAID中，RAID 0的速度是最快的，但RAID 0既没有冗余功能，也不具备容错能力，如果一个物理磁盘损坏则所有的数据都丢失。==

    2. RAID 1：两组以上的N个磁盘相互作镜像，在一些多线程操作系统中读取速度很快，读取速度等于磁盘数量倍的速度，写入速度微小降低。

      ==只要有一个磁盘正常即可维持运作，但无论使用多少个磁盘做RAID 1仅算作一个磁盘的容量，是所有RAID中磁盘利用率最低的级别。==

    3. RAID 2：以汉明码的方式将数据进行编码后分割为独立的比特，并将数据分别写入硬盘中。因为在数据中加入错误修正吗ECC所以数据整体的容量会比原始数据大一些。至少需要三台磁盘驱动器才能运作。

    4. RAID 5：是一种存储性能、数据安全和存储成文兼顾的存储解决方案。把数据和相对应的奇偶校验信息存储到组成RAID 5的各个磁盘上（分别存储于不同的磁盘）。当RAID 5 的一个磁盘数据发生损坏后，可以利用剩下的数据和相应的奇偶校验信息去恢复被损坏的数据。

      ==保障程度要比RAID 1低而磁盘空间利用率比RAID 1高。数据读取速度和RAID 0近似，单独写入一块磁盘的速度略慢（因为多了奇偶校验信息故写入数据速度相对变慢）。==

5.2.3 时钟

  1. 定义：定时器，每隔一个周期产生一个中断——时钟中断、时钟滴答。

  2. 实现方式：基于交流电压；基于晶体振荡器（计数器+存储寄存器；可编程）

  3. 功能

    1. 维护日期时间：可以记录时钟滴答数或者直接记录秒。
    2. 防止进程超时运行：保证合理时间片。
    3. 记录CPU的使用情况：记录每个进程的运行时间。
    4. 处理用户进程提出的alarm系统调用：如网络的超时重传机制。
    5. 为系统本身的各个部分提供监视定时器。
    6. 完成概要剖析、监视和统计信息收集：让用户了解程序每一部分运行时间占比情况。

  4. 监视定时器：自动增长，系统每隔一段时间重置监视定时器为0，如果出现监视定时器溢出即系统无法重置监视计时器则认为出现死机。

    作用：检测死机。

  5. 软时钟：由于中断开销大，为了效率而取消中断，计数器自行增加或降低，由操作系统自行检查其数值。

5.3 中断与DMA

  1. I/O最原始的方式——轮询

    CPU循环查询某个地址中的数据，该数据对应I/O的状态如传输完成等。

    常用于简单的嵌入式系统。

    适用范围：几乎所有的I/O操作。

  2. I/O最基本的方式——中断

    目的：减少CPU的干预，解放CPU

    范围：几乎所有支持中断的I/O操作

    注意：

    1. 中断嵌套有特定的层数限制。

    2. 由于流水线的存在，有些情况可以使用不精确中断，有些不可；可以对不同类型的中断分别设计成精确和不精确。

      精确中断的要求：

      • PC保存在一个已知的位置
      • PC所指向的指令之前的所有指令已经完全执行
      • PC所指向的指令之后的所有指令都没有执行
      • PC所指向的指令的执行状态是已知的

  3. I/O最高效的方式——DMA Direct Memory Access 直接存储器存取

    目的：进一步减少CPU的干预

    适用范围：硬盘、显卡、网络接口

    使用总线的方式：

    1. 周期窃取：一次传一个字节；CPU等待DMA；轻微延迟
    2. 突发模式：一次传一块；CPU等待DMA；延迟较长

    数据流动方式：

    1. 飞跃模式：设备-内存
    2. 其他：设备-DMA控制器-内存

    注意：DMA不一定更高效，目的是节省CPU时间，CPU中断传输地更快（但是CPU可能需要做其他更为重要的事情），对于大量数据才有效（I/O开始和结束仍需要CPU中断）

5.4 设备驱动程序

  3. 功能

    1. 接收来自设备无关软件所发出的I/O请求并执行
    2. 接收设备的请求（来自中断）并处理
    3. 对设备进行初始化
    4. 对电源需求和日志时间进行管理
    5. ==在运行过程中可能自身被再次调用，因此必须可重入==

  4. 驱动程序的典型结构

    2. 把上层的I/O请求翻译为硬件可识别的格式，如将一个线性磁盘块号转换成磁头、磁道、扇区和柱面号等信息
    3. 检查设备当前是否正在使用，如果在使用则请求被排入队列以备稍后处理
    4. 驱动程序检查硬件状态，同时进行一些准备工作使设备就绪
    5. 发出正式的I/O命令如写数据、读数据、控制信息等。
      1. I/O命令执行很快，驱动程序继续运行
      2. I/O命令执行很慢，驱动程序阻塞，等待命令执行完毕后设备利用中断唤醒驱动程序

Chapter 6 安全

6.1 系统本身的安全措施

  1. 概述

    1. 概念区分：data safety 数据防丢失 data security 数据防泄密
    2. 安全的三个部分：机密性；完整性；可用性

  2. 可信系统

    理论上可以建立一个安全的操作系统，但实际不如此因为操作系统过于复杂且安全意味着牺牲功能性。

    可信系统在形式上声明了安全要求并满足这些安全要求，核心是可信计算基Trust Computing Base TCB 即实施所有安全规则所必须的硬件和软件。

    访问监视器接受所有与安全有关的系统请求然后决定是否允许运行。

    可信计算基的使用更适于微内核。

  3. 保护机制

    域 Domain 是 对象、权限对的集合，每一对组合指定一个对象和一些可在其上运行的操作子集，一个域可对应多个用户，一个用户可对应多个域。

    访问控制列表 Access Control List ACL，一个文件对应一个ACL（矩阵过大、过于稀疏导致矩阵表示域不适用）。

    权能字用来表示保护矩阵。

  4. 认证

    安全的前提：操作系统确定当前使用该系统的用户的身份

    登录口令：口令加密存储，利用安全哈希；加盐即口令和一个公开的随机数一起加密存储；弱口令

    挑战-响应认证：计算机产生一个随机数发给用户，用户将这个随机数当成明文、口令当成密钥进行加密，密文发送给计算机。计算机同样将随机数当成明文、口令作为密钥加密，比较密文是否一致。更复杂的场景如用户也要认证计算机则可以使用密码协议。

    物理识别的认证：Token，侧信道攻击；生物识别，指纹、虹膜信息作为口令，信息长度长但不够随机因此需要使用随机数提取器压缩长度 extractor

6.2 不安全的因素

  1. 软件漏洞

    缓冲区溢出，原因是C语言不检查内存溢出。

  2. 内部攻击——软件编写人员引入的攻击

    逻辑炸弹：程序员在软件中插入恶意代码如删除所有数据，平时不运行而程序员可控制其运行用于报复和欺诈等。规避方式为源代码审查。

    后门

    欺骗登录：相同界面盗取口令。

  3. 恶意软件

    木马、病毒、蠕虫、间谍软件：破坏、获取信息、勒索、控制计算机（形成僵尸网络、虚拟货币挖矿）、继续扩散

    特点：隐匿性，与计算机自身程序共存（可修改系统文件、可驻留于内存、可隐藏在引导区、可编程软件也会隐藏病毒）；自身进化；难以有效删除

6.3 进一步防护

  1. 防火墙

  2. 杀毒软件：专用查找并删除恶意软件的病毒，通过匹配病毒特征来查找；与病毒共同发展（特征匹配随病毒发展而进步；一些病毒会专门设计防止被匹配；病毒也会删除或关闭杀毒软件；杀毒软件也会使用一些病毒用的技术防止自己被删除）

  3. 代码签名

  4. 入侵检测

    基于特定的入侵模式

    蜜罐：引诱攻击者攻击，用于分析攻击者的攻击方式进一步优化防护系统，使用前提是攻击蜜罐的动作可以被检测到。

  5. 封装代码

    沙盒：将软件运行于一个受限的系统环境中，控制程序可使用的资源如文件描述符、内存、磁盘空间等。可以直接把可疑软件运行在虚拟机上。

    囚禁：一种沙盒的经典实现，不直接编译成可执行文件而是编译成一个中间语言（由解释器执行中间语言，编译过程和结果与硬件无关，解释器是安装在具体硬件上的软件，不同硬件的解释器是不一样的，解释器保证安全），如果程序存在恶意代码解释器会发现并禁止运行。

  6. 总结

    很少有完美的操作系统，一般做到攻击者攻击代价足够大即可。攻击和防护是同时发展的。最好的策略就是多种技术一起使用实现全面防护。

9 数字签名

  1. 数字签名用以保障消息的认证性

    消息的完整性：消息在传输过程中未被修改

    消息来源的真实性：消息由所声明的用户签署

  2. 数字签名与手工签名的区别

    手工签名：模拟的，因人而异，与真实签名比较进行验证（用收到的签名与之前签过的签名字迹进行比较），容易伪造

    数字签名：数字的，因消息而异（同一人不同消息签名一般不同），可公开验证，容易复制，是一种公钥密码，任何人获得公钥都可以输入消息运行数字签名算法验证签名真伪

  3. 数字签名的组成

    1. 密钥生成算法$KeyGen:(sk,vk)\gets KeyGen(1^{\lambda})$， $sk$签名密钥——私钥；$vk$验证密钥——公钥。

      1. 签名算法$Sig:s\gets Sig_{sk}(m)$。
      2. 验证算法：$Vrfy:Vrfy_{vk}(s,m)=1\ or\ 0$

    表示： $Sig_{sk}(m)\ Sig(sk,m)$

9.1 RSA数字签名

  1. 算法描述

    1. $KeyGen:选取大素数p和q，令n=pq，\phi(n)=(p-1)(q-1),选择e\in{1,…,\phi(n)}\满足gcd(e,\phi(n))=1，计算d，满足ed=1(mod \phi(n)),vk=(n,e)是验证密钥(公钥)\,sk=(n,d)是签名密钥(私钥)$

    2. $Sig$：签名者使用私钥对信息$m$进行签名

      $s=Sig_{sk}=m^dmodn$

    3. $Vrfy:$使用签名者公钥验证签名真伪

      $m’=s^emodn，如果m=m’则签名为真输出1，反之输出0。$,

  2. 安全性

    1. 攻击目的 依次降低
      1. 安全破解
      2. 选择性伪造：对除了敌手之外其他人选择的消息进行伪造
      3. 存在性伪造：伪造一个消息和签名对使之通过验证，对消息本身无要求
    2. 攻击手段 依次增强
      1. 唯密钥攻击：敌手只有签名者公钥
      2. 已知消息攻击：敌手可以获得某些消息和签名对
      3. 选择消息攻击：敌手可以选择消息并获得签名

    如果在选择消息攻击下不能达成存在性伪造则说明数字签名算法是安全的。

  3. 攻击类型

    1. 唯密钥攻击下的存在性伪造

      在$Z_n$中随机选择一个数$s$作为签名，再计算$s^emodn$作为消息$m$。

    2. 已知消息攻击下的存在性伪造

      利用乘法同态特性：

      已知$s_1=Sig_{sk}(m_1)={m_1}^d，s_2=Sig_{sk}(m_2)={m_2}^d$，则$s_1s_2={m_1}^d{m_2}^d=(m_1m_2)^d$是对消息$m_1m_2$的伪造签名且可以通过验证$Vrfy(m_1m_2,s_1s_2)=1$。

      避免利用乘法同态特性的攻击：对消息的hash值进行签名，先hash后签名。

      $s=Sig_{sk}(H(m))=H(m)^dmodn$

      1. 提高签名效率，对任意长度的消息通过hash压缩到$Z_n$，从而实现对任意长度消息的签名

      2. 破坏数学结构，抵抗相关攻击。通过添加hash函数，抵抗唯密钥攻击和已知消息攻击下的存在性伪造。

        唯密钥攻击下，$S\in Z_n，S^e=m，这里的m是H(m’)，求解m’$是困难的，基于hash函数的单向性

        已知消息攻击下的存在性伪造：

        $H(m_1)H(m_2)\neq H(m_1m_2)$

      Hash函数需要满足抗碰撞特性，以防止不同消息对应相同的签名。

9.2 基于离散对数的数字签名

9.2.1 ElGamal数字签名

  1. ElGamal签名的特点

    1. 非确定性算法：对于相同的消息每次生成的签名一般是不同的

    2. 基于求解离散对数的困难性

  2. 算法描述

    1. $KeyGen$：设素数$p$，满足$Z_p^*$上离散对数问题是困难的，$p-1$阶元素$g$(本原元)和随机数$x\lt p(x\in[1,p-1])$，计算$y\equiv g^xmodp$，公钥为$y,g,p$，私钥是$x$

    2. $Sign$：选择一个随机数$k\in[1,p-1]$，$k与p-1$互素，计算$r=g^kmodp$与$s=(H(m)-xr)k^{-1}mod(p-1)(位于指数位置)$

      输出签名：$(r,s)$

    3. $Vrfy$：计算$y^rr^s=g^{H(m)}modp$

  3. 安全性分析

    在没有私钥$x$的条件下伪造签名$(r,s)$

    1. 给定消息m，首先选择r再求解s；首先选择s再求解r；同时求解r和s，困难。

    2. 首先选择r和s再求解m，离散对数问题+单向函数求原象，困难。

    3. 同时求解r、s、m，若没有hash函数则可产生存在性伪造。

      1. 令$r=g^iy^jmodp,gcd(j,p-1)=1$

        则验证等式为$y^r(g^iy^j)^s=g^mmodp$

        $y^{r+js}=g^{m-is}modp$

        只需令$r+js=0modp-1 \and m-js=0modp-1$

      2. 由已知签名(r,s,m)伪造新的签名

        令$r’=r^hg^iy^jmodp$

        代入验证等式$y^{r’}{r’}^{s’}=y^{r’}(r^hg^iy^j)^{s’}=y^{r’+js’}r^{hs’}g^{is’}$

        又$y^rr^s=g^m$

        $y=g^{mr^{-1}}r^{-sr^{-1}}$

        代入上式，令$r$的指数为0即可。

      引入Hash函数即可抵抗上述两种攻击。

  4. 随机数$k$不能泄露，否则可以求出密钥。

    $y^rr^s=g^{H(m)}\ r=g^k$

    $s=(H(m)-xr)k^{-1}mod(p-1)$

    公钥$y,g,p$，$r,s$已知，若已知$k$则可求其逆，从而可求私钥$x$。

    每次签名选择不同的随机数$k$否则可能求出密钥。

    $y^rr^{s_1}=g^{m_1}\y^rr^{s_2}=g^{m_2}\g^{k(s_1-s_2)=g^{m_1-m_2}}modp\k(s_1-s_2)=m_1-m_2mod(p-1))$

    互素直接可求逆解方程；不互素除以最大公因子可求多个验证。

9.2.2 Schnorr数字签名

  1. Schnorr签名

    密钥生成：设素数$p$满足${Z_p}^*$上离散对数问题是困难的，素数$q|p-1$，$g$是${Z_p}^中的q阶元素$，$x\gets {Z_q}^，y=g^xmodp$

    公钥$p,q,g,y$;私钥$x$

    $g^q=1modp$

    签名：任选随机数$k\in[1,q-1]$，计算$c=H(m||g^k)$与$s=k+xc\ modq$，输出消息$m$的签名$(c,s)$

    验证：$H(m||g^sy^{-c})=c$

    ==$q$阶子群上的运算==

  2. 参数长度

    一般$p$长度为1024-bit，$q$为160-bit，签名$(c,s)$长度为320-bit

    $q$阶元素的生成方式：$g_0$是$Z_p$的本原元，计算$g={g_0}^{\frac{p-1}{q}}$即为$q$阶元素。

    Hash函数的作用:

    既在输入又在输出，不易伪造。

9.2.3 DSA数字签名

DSA是ElGamal签名的变形并吸收了Schnorr方案的思想，采用了SHA-1。

  1. 算法描述

    1. 密钥生成：设素数$p$满足${Z_p}^*$上离散对数问题是困难的，素数$q|p-1$，$g$是${Z_p}^*$中的$q$阶元素，$x\gets {Z_q}*$(均匀随机选择)，$y=g^xmodp$

      公钥：$p,q,g,y$ 私钥：$x$

    2. 签名：任选随机数$k\in[1,q-1]$，==计算$(r=g^kmodp)modq$==与$s=(H(m)+xr)k^{-1}modq$

      输出消息$m$的签名为$(r,s)$

    3. 验证算法：$r=(g^{H(m)s^{-1}modq}y^{rs^{-1}modq}modp)modq$

      (先两边变成s次方消s)

  2. 参数选取

    p推荐1024-bit q为160-bit Hash函数SHA-1 签名长度320-bit

    s可能不存在逆，此时需要重新选择k

    ==预计算r，不依赖于m==，预先选好k计算r，签名时候可以直接用。

10.1.1 基于对称密钥的密钥分发

  1. 安全性要求

    1. 协议结束后，合法参与者拥有一个相同的密钥K，且任何其他人无法获知K的信息（可信第三方除外）

  2. 攻击手段

    1. 被动；监听信道信息

    2. 主动：修改信道信息、重放信息、冒充合法用户

  3. 密钥分发的可能方式

    1. A选择一个密钥并以物理的方式亲自交给B

    2. 第三方C选择密钥后以物理的方式亲自交给A和B

      效率低，人工交付对于大范围的分布式系统并不实用

    3. 如果A和B以前或最近使用过一个密钥，则其中一方可以用它加密一个新密钥后再发送给另一方

      如果敌手成功获得其中一个密钥，则随后的密钥都会泄露

    4. A和B与第三方C均有加密连接，则C可以利用加密连接将密钥分别发送给A和B

      1. 依赖于密钥分发中心KDC

      2. 两类密钥

        1. 主密钥：每个用户与KDC共享唯一一个主密钥 ，通过物理方式或公钥传递。
        2. 会话密钥：定期变化的临时密钥，用于用户间的短期会话。由KDC生成，利用主密钥加密后传递给用户。

      3. 优势

        限制了攻击者得到的用同一密钥加密的密文数量，降低破解风险。

        降低密钥（临时会话密钥——频繁更换）泄露导致的损失。

  4. 大型网络的KDC层级体系

    1. 做法

      本地域内各个实体的密钥分发由本地KDC负责

      不同域的实体间的密钥分发由两个本地KDC通过全局KDC协商

    2. 优势

      使主密钥分发的开销最小化，降低单个KDC的工作量

      将被破坏KDC的危害限制在本地域，其他域不受影响

10.2 公钥密码密钥分发

  1. 公钥密码学的最大挑战：保障公钥的真实性。

  2. 由于不需要保密，因此关键在于证明公钥真实性。

  3. 方法

    1. 用户公开发布

      描述：任意通信方可将公钥发送或广播给其他通信各方

      缺点：任何人都可以伪造公钥并公开发布

    2. 公钥授权方维护公开目录

      描述：可信实体或组织负责维护一个动态可访问的公钥目录。

      1. 每一通信方通过目录管理方以安全的方式注册公钥，
      2. 目录管理方为每一个通信方建立目录项{name,public-key}，
      3. 每一通信方能够可靠地知道目录管理方的公钥（每一个通信方可以验证目录管理方提供公钥的真实性）。

      $T_1$：发送时的时间。

      第5步之后已经获取公钥

      第6、7步确认对方已经拿到了自己的公钥。

      缺点：目录管理方成为协议运行的瓶颈，大量用户造成巨大压力。

    3. 利用公钥证书

      描述：通信双方使用证书交换公钥。

      1. 用户A向证书颁发机构（Certificate Authority ，CA）申请证书（一般需要用户现场注册申请，提供身份识别信息$ID_A$，公钥$pk_A$等信息）
      2. CA为用户A提供证书$C_A=(T||ID_A||pk_A,Sig_{sk_{C_A}}(T||ID_A||pk_A))$，本质上是一个签名，将用户身份信息和公钥进行绑定
      3. 用户A可将证书$C_A$发送给任何参与者，后者可读取并验证证书合法性（如何验证证书合法性？）

  4. X509证书（IPsec、TLS、S/MIME）

    有可信证书颁发机构创建，被CA或用户放入目录服务器中。

  5. 公钥基础设施PKI

    描述：用来管理和控制证书的安全系统

    组成：

    1. 证书颁发

      CA向用户颁发新证书

      用户的身份和凭证通过==非密码手段==验证，亲自

    2. 证书撤销

      在证书有效期内撤销证书（密钥丢失、被盗、用户不再信任CA等情况下）

      证书本身无法标识自身是否已经被撤销，需要辅助机制表示被撤销的证书。

      证书撤销列表（CRL）：所有被撤销的但没有过期的证书序列号列表；由CA保留并签发，并周期性更新——用户收到证书时，除了验证有效性之外还需检查是否在CRL中

    3. 密钥备份/恢复/更新

      密钥备份：用户私钥由PKI管理员安全存储以防丢失、忘记

      密钥恢复：恢复或激活私钥

      密钥更新：处于安全考虑定期更新密钥；证书过期也要更新

    4. 时间戳：密钥在颁发、撤销、更新等时候都需明确时间或时间段。

  6. 信任模型

    用户对CA的信任程度决定了用户对其发布证书的信任程度

    通常情况下，证书Cert并不全是由同一个CA直接签发，从被信任的CA到给定的证书有一个证书路径。

    1. 路径上每个证书都由前一个证书签名
    2. 通过验证路径上的所有证书验证证书是否可信

  3. 组成

1.1.3 性能指标

  1. 速率（数据率、数据传输率、比特率）：连接在计算机网络上的主机在数字信道上传送数据位数的速率。

    速率单位以$10^3$为单位；存储单位以$2^{10}$为单位。

  2. 带宽：某个信号具有的频带宽度——最高频率与最低频率之差（以赫兹为单位）；网络的通信线路传送数据的能力，单位时间从网络中某一点到另一点所能通过的最高数据率；链路入口的注入/传送速率；理想状态能够达到的最高速率，实际比带宽小。

  3. 吞吐量：单位时间通过某个网络/信道/端口的数据量；受网络的带宽或额定速率的限制；所有链路速率加和。

  4. 时延：数据/报文/分组/比特流从网络/链路的一端传送到另一端所需的时间。

    1. ==$发送时延/传输时延=\frac{数据长度}{信道带宽/发送速率}$==：从发送分组的第一个比特算起，到该分组的最后一个比特发送完毕所需时间；机器内数据放到信道上。
    2. ==$传播时延=\frac{信道长度}{电磁波在信道上的传播速率}$==：取决于电磁波传播速度和链路长度；比特流经过调制由电磁波作为载体，速率用电磁波传输速率；机器外。
    3. 排队时延：等待输入/输出链路可用的时间；路由器。
    4. 处理时延：检错、找出口；路由器。

  5. ==$时延带宽积={传播时延} * {带宽}$==：某段链路目前有多少比特。

  6. ==$往返时延(RTT)=往返传播时延(=传播时延*2)+末端处理时间$==：从发送方发送数据开始到发送方收到接收方的确认总共经历的时延；不包括传输时延。

  7. 利用率

    1. $信道利用率=\frac{有数据通过时间}{(有+无)数据通过时间}$
    2. $网络利用率=信道利用率的加权平均值$

1.2.1 分层结构、协议、接口、服务

  1. 分层的原因——==解决计算机网络复杂的大问题；拆解==

    发送文件前要完成的工作：通路激活、告知网络如何识别目的主机、查明目的主机状况（开机与否、网络连接正常与否）、查明对方中相关管理程序是否准备好、差错和意外的解决…——十分繁琐，因此考虑分层。

  2. 概念

    1. 实体：第n层中的活动元素成为n层实体；同一层的实体叫对等实体。

    2. 协议：为进行网络中的对等实体数据交换而建立的规则、标准或约定；水平向。

      语法——规定数据传输的格式、语义——规定所要完成的功能、同步——规定各种操作的顺序

    3. 接口（访问服务点SAP）：上层使用下层服务的入口。

    4. 服务：下层为相邻上层提供的功能调用；垂直向。

    5. 服务数据单元SDU：为完成用户所要求的功能而应传送的数据

    6. 协议控制信息PCI：控制协议操作的信息。

    7. 协议数据单元PDU：对等层次之间传送的数据单位。

  3. 分层的基本原则

    1. 各层之间相互独立，每层只实现一种相对独立的功能。
    2. 每层之间界面自然清晰，易于理解，相互交流尽可能少。
    3. 结构上可分割开，每层采用最合适的技术来实现。
    4. 保持下层对上层的独立性，上层单向使用下层提供服务。
    5. 整个分层结构促进标准化工作。

1.2.2 OSI（开放系统互联）参考模型——7层——法定标准

  1. 目的：支持异构网络的互联互通。

  2. 中间设备最多到网络层；传输层及以上——端到端；低三层——点到点。

  3. 应用层：所有能和用户交互产生网络流量的程序。

    文件传输FTP、电子邮件SMTP 、万维网HTTP

  4. 表示层：用于处理在两个通信系统中交换信息的表示方式（语法和语义）

    功能：数据格式变换、数据加密和解密、数据压缩及恢复

  5. 会话层：向表示层实体/用户进程提供建立连接并在连接上有序地传输数据，建立同步（SYN），会话彼此独立。

    功能：建立、管理、终止会话；使用校验点可使会话在通信失效时从检验点/同步点继续恢复通信，实现数据同步（适用传输大文件）

2.1.4 奈氏准则和香农定理

  1. 失真：信号与初始状态相比发生了扭曲和变化

    影响因素：码元传输速率；信号传输距离；噪声干扰；传输媒体质量

    ==失真的一种现象：码间串扰==

    码元传输速率过快——接收端收到的信号波形失去了码元之间清晰界限的现象；高频信号难以区分。

  2. 奈氏准则/奈奎斯特定理

    内容：在理想（信号在传输过程中不会受到外界的影响）低通（带宽受限，==低于最高频率的可以通过==）条件下，为了避免码间串扰，极限码元传输速率为2 W Baud，W为信道带宽，单位Hz。

    $理想低通信道下的极限数据传输率=2Wlog_2V(b/s)$

    1. 在任何信道中，码元传输速率是有上限的；超过上限就会出现严重的码间串扰

    2. 信道的频带越宽即能通过的信号高频分量越多，就可以用更高的S速率进行码元的有效传输。

    3. 奈氏准则给出了码元传输速率的限制，但没有对信息传输速率给出限制

    4. 由于码元传输速率受奈氏准则的制约，所以要提高数据的传输速率，就必须设法使每个码元携带更多个比特的信息量，需要多元制的调制方法

  3. 香农定理

    信噪比：信号的平均功率/噪声的平均功率

    $信噪比(dB)=10log_{10}(S/N)$

    在==带宽受限且有噪声==的信道中，为了不产生误差，信息的数据传输速率有上限值。

    $信道的极限传输速率=Wlog_2(1+S/N)$ b/s

    带宽或者信道中的信噪比越大，信息的极限传输速率越高

    对一定的传输带宽，信息传输速率的上限确定

    只要信息的传输速率低于信道的极限传输速率就一定能找到某种方法实现==无差错的传输==

2.1.5 编码与调制

3.2 封装成帧与透明传输

  1. 在一段数据的前后部分添加首部和尾部构成一个帧，接收端在收到物理层上交的比特流后就能根据首部和尾部标记从收到的比特流中识别帧的开始和结束。

    首部和尾部包含许多控制信息，用于==帧定界==。

  2. 透明传输：不管所传数据是什么样的比特组合，都应当能够在连路上传送。==因此，链路层就看不见有什么妨碍数据传输的东西。==

    1. 字符计数法：真受不使用一个计数字段 8位 表明帧内字符数 一个错全错

    2. 字符填充法：SOH帧首部的第一个字节 EOF帧尾部的最后一个字节

      当传送的帧是由文本文件组成时 文本文件的字符都是从键盘输入 ASCII码 不管输入什么字符都可以放在帧中传过去 透明传输

      当发送的帧由非ASCII码的文本文件组成时如二进制代码的程序或图像 就采用字符填充方法 实现透明传输

      发送端添加 接收端删除

      复杂。

    3. 零比特填充法：帧首部和帧尾部均为01111110

      在发送端，扫描整个信息字段，只要出现连续5个1就立即填入1个0。

      在接收端，收到一个帧时，先找到标志字段确定边界，再用硬件对比特流进行扫描，发现连续5个1时就把后面的0删除。

      保证了透明传输，在传送比特流中可以传送任意比特组合，而不会引起对帧边界的判断错误。

    4. 违规编码法

      用编码中不会用到的电平作为标识。

3.3 差错控制

差错源于噪声。

差错分类

  1. 位错：0/1变换
  2. 帧错：丢失；重复；失序

3.3.1 检错编码

  1. 检错编码：奇偶校验码、循环冗余码

  2. 数据链路层编码和物理层数据编码与调制不同。好组合针对==单个比特==，解决传输过程中单个比特的同步问题；而后者针对==一组比特==，通过冗余码的技术实现一组二进制比特串在传输过程中是否出现了差错。

  3. 奇偶校验码：校验码+信息 中 1的个数为奇数/偶数

    11100101奇校验码

    01100101偶校验码

    加上一个比特 1的个数为奇数 奇校验

    加上一个比特 1的个数为偶数 偶校验

    只能检查出奇数个比特错误，检错能力50%。偶数个变化抵消

  4. CRC循环冗余码 FCS生成以及接收端CRC检验都由硬件实现 迅速 不会延误数据传输

    待传数据 生成多项式 冗余码

    发送数据=待传数据+冗余码

    待传数据1101011011 生成多项式10011 最终发送数据为

    $1101011011\div 10011=$

    计算冗余码：

    1. 加0 假设生成多项式$G(x)$的阶为$r$，则加$r$个0.

      $10011=x^4+x+1$，阶为4

      1101011011 0000

    2. 模2除法：异或：同0异1

      $1101011011 0000\div 10011=x…1110$

    接收端：把收到的每一个帧除以同样的除数，然后检查得到的余数r。若余数为0，判断未出错，接收；余数不为0，有差错，丢弃。

  5. ==可靠传输是什么？数据链路层发送什么，接收端就收到什么。==

    CRC检验只能实现无比特差错的传输，因为传输过程中有丢弃。

3.4 流量控制

3.4.1 流量控制（控制发送速率，使接收方有足够的缓冲空间来接收每一个帧）与可靠传输（发送端发送数据原样发送到接收端）机制

  1. 流量控制：较高的发送速度和较低的接收能力不匹配会造成传输出错。

  2. 流量控制方法

    1. 停止-等待协议：每发送完一个帧就停止发送，等待对方确认，收到确认后发送下一个帧。 窗口大小为1的滑动窗口

    2. 滑动窗口协议：收到一次确认 发送窗口向前移动一格

      滑动窗口解决了流量控制（收不下就固定不动）可靠传输（发送方自动重传，发送后一段时间没收到确认就再次发送）

      1. 后退N帧协议 发送窗口大小$\gt$1 接收窗口=1
      2. 选择重传协议 发送窗口$\gt1$，接收窗口$\gt$1

  2. 应用情况

    1. 无差错情况

    2. 有差错情况：超时计时器：每发送一个帧就启动一个计时器，设置的重传时间应当比帧传输的平均RTT更长一些。

    信道吞吐率=信道利用率*发送方的发送速率

3.5.3 CSMA协议

载波监听多路访问协议

CS载波侦听监听 每一个站在发送数据之前要检测一下总线上是否有其他计算机在发送数据。

当几个站同时在总线上发送数据时，总线上的信号电压摆动值会增大（互相叠加）。当一个站检测到的信号电压摆动值超过一定门限值时，就认为总线上至少有两个站同时在发送数据，表明产生了碰撞即发生冲突。

MA 多点接入 表示许多计算机以多点接入的方式连接在一根总线上

4.1 网络层提供的两种服务

虚电路是逻辑上的连接，分组沿着这条逻辑连接按照存储转发方式发送；电路交换的电话通信建立了真正的连接。

网络提供数据报服务(TCP/IP)：网络层向上只提供简单灵活的无连接的尽最大努力交付的数据报服务。网络在发送分组时不需要先建立连接。每一个分组（IP数据报）独立发送，与其前后的分组无关（不编号），网络层不提供服务质量的承诺（所发送的分组可能出错、丢失、重复、失序、超时）。

尽最大努力交付：传输网络不提供端到端的可靠传输服务，路由器可以比较简单、价格低廉。如果主机中的进程之间通信需要是可靠的那么就由网络中的主机中的运输层负责可靠交付。优点是网络造假降低、运行方式灵活、适应多种应用。

4.3.0 前导

  • 解决了链路层广播域太大带来的问题。

网络层-划分子网

  • 把大的广播域划分为较小的广播域；
  • 提高IP地址空间的利用率；保证网络的安全性；
  • 提高网络的灵活性。

数据链路层广播与IP广播的区别：

  • 数据链路层广播是第二层的广播，在一个以太网上实现对该局域网上所有主机的MAC帧进行广播；
  • IP广播是第三层的广播，在二层交换机上实现。VLAN可以理解为一台物理交换机，在逻辑上分割成若干台虚拟交换机。跨越VLAN间的通信，必须通过子网划分这种三层技术。

    4.3.1 划分子网

1 从两级IP地址到三级IP地址

早期IP地址：

  • IP地址空间利用率低；
  • 给每一个物理网络分配一个网络号会使路由表太大从而性能变差；
  • 不够灵活。

在IP地址中增加子网号字段就从两级IP变成三级IP。划分子网属于单位内部的事情，对外部网络透明，对外表现为一个没有划分子网的网络。

基本思路：从主机号借用若干个位作为子网号，同时主机号减少若干个位。

  • 从其他网络发送给本单位某主机的数据报仍根据IP数据报的目的网络号找到连接在本网络上的路由器；
  • 此路由器收到IP数据报后按目的网络号和子网号找到目的子网；
  • 将IP数据报交付目的主机。

优点：

  • 减少了IP地址的浪费；
  • 使网络的组织更加灵活；
  • 便于维护和管理。

主机号全0表示网络地址，主机号全1表示广播地址。

2.子网掩码

从IP数据报的首部无法判断源目主机所连接的网络是否进行了子网划分，使用子网掩码可以找出IP地址中的子网部分。长度32位，左1串对应网络号和子网号，右0串对应主机号。IP地址与子网掩码逐位相与即可获得网络地址。

路由器在和相邻路由器交换路由信息时必须把自己的子网掩码传送给乡音路由器。

路由表项包含目的网络地址及其子网掩码。若一个路由器连接在两个子网则具备两个网络地址和两个子网掩码。

子网划分方法：固定长度子网和变长子网。

全0、全1子网在现代网络中可以正常使用。

不同的子网掩码对于同一IP地址可以得出相同的网络地址但其效果不同。

4.3.2 使用子网时分组的转发

由于从IP地址不能唯一得出网络地址而数据报首部并没有提供子网掩码信息所以分组转发算法也要做出改动。

划分子网情况下路由器转发分组的算法：(直接相连->特定主机路由->表中下一跳->默认路由->报错)

  1. 从收到的分组的首部提取目的IP地址D；
  2. 先判断是否是直接交付。对路由器直接相连的网络逐个检查，用各网络的子网掩码与D相与判断是否匹配，若匹配直接交付否则间接交付执行3；
  3. 若路由表中有目的地址为D的特定主机路由，则将分组传送给指明的下一跳路由器，否则执行4；
  4. 对路由表的每一行，将子网掩码与D相与。若结果与该行的目的网络地址匹配，则将分组传送给该行指明的下一跳路由器，否则执行5；
  5. 若路由表有一个默认路由，则将愤怒传送给路由表中指明的默认路由器；否则执行6；
  6. 报告转发分组出错。

4.3.2 无分类编址

1.网络前缀

CIDR使用各种长度网络前缀来代替分类地址中的网络号和子网号，变回两级编址。采用斜线记法，在IP地址后加斜线表示网络前缀所占位数。

CIDR把网络前缀相同的连续IP地址组成CIDR地址块，全0和全1的主机号地址一般不使用。

路由聚合：一个CIDR地址块可以表示很多地址，这种地址的聚合通常称为路由聚合，使得路由表中的一个项目可以表示多个原来传统分类地址的路由。

路由聚合减少路由器之间的路由选择信息的交换，提高互联网性能，也称为构成超网。

前缀长度不超过23位的CIDR地址块都包含了多个C类地址，这些C类地址合起来构成了超网。CIDR地址块中的地址数一定是2的整数次幂。网络前缀越短，地址块包含的地址数越多。

优点：可以根据需求分配适当的CIDR地址块，更加有效地分配了IPv4地址空间。

2.最长前缀匹配

使用CIDR时，路由表项由网络前缀和下一跳地址组成。在查找路由表时可能达到多个匹配结果，此时选择具有最长网络前缀的路由（网络前缀越长，地址块越小，路由越具体）。

3.在划分子网情况下路由器转发分组的算法

根据最长前缀匹配规则，对于特定主机路由，特定主机被视为一个掩码为255.255.255.255的特殊网络，此后匹配了前32位，优先使用。对于默认路由，这个路由在所有表项都不匹配的时候有最短的前缀匹配，匹配0位。

6.5 RIP协议

  1. RIP是一种分布式的基于距离向量的路由选择协议，是因特网的协议标准，最大优点是简单

  2. RIP协议要求网络中每一个路由器都维护从它自己到其他每一个目的网络的唯一最佳距离记录。

  1. 距离：通常为跳数，即从源端口到目的端口所经过的路由器个数，经过一个路由器跳数+1。从一个路由器到直接相连的网络距离为1。==RIP允许一条路由最多只能包含15个路由器，因此距离为16表示网络不可达==。适用于较小的互联网络。

  2. 特性：

    1. 仅和相邻路由器交换信息
    2. 路由器交换的信息是自己的路由表
    3. 每30s交换一次路由信息，然后路由器根据新信息更新路由表。若超过180s没收到邻居路由器的通告，则判定邻居无，并更新自己的路由表

    路由器刚开始工作时，只知道直接相连的网络的距离为1，接着每一个路由器也只和数目非常有限的相邻路由器交换并更新路由信息。

    经过若干次更新后，所有路由器最终都会知道到达本自治系统的任何一个网络的最短距离和下一跳路由器的地址——收敛

  3. ==距离向量算法==

    1. 修改相邻路由器发来的RIP报文中的所有表项

      对地址为x的相邻路由器发来的RIP报文，修改此报文中的所有条目，把下一跳字段中的地址改为x，并把原有的距离字段加1

    2. 对修改后的RIP报文中的每一个项目，进行如下步骤：

      1. 路由表中没有该项目，直接写入
      2. 路由表中已经有该项目，如果下一跳仍为x则用收到的项目替换原有项目；若下一跳不是x，原来距离比从x走的距离远则更新，否则不做处理

    3. 若180s还没收到相邻路由器x的更新路由表，则把x记为不可达的路由器（即把距离设置为16）

    4. 返回

      RIP是应用层的信息，一个RIP报文最多包括25个路由。

      当网络出现故障时，要经过比较长的时间才能将此信息传送到所有的路由器。更新为16,16还没传递，其他的不改动，循环往复直到所有距离都变成16

      6.6 OSPF协议 开放最短路径优先协议

  4. OSPF最主要的特征：使用分布式的链路状态协议。

  5. 特点：

    1. 使用洪泛法向AS内所有路由器发送信息，即路由器通过输出端口向所有相邻的路由器发送信息，而每一个相邻路由器又再次将此信息发往其所有的相邻路由器
    2. 发送的信息就是与本路由器相邻的所有路由器的链路状态（本路由器与哪些路由器相邻以及该链路的度量/代价——费用、距离、时延、带宽）（==不一定是跳数==）
    3. 只有当链路状态发生变化时，路由器才向所有路由器洪泛发送此信息

    最后所有路由器都能建立一个链路状态数据库即全网拓扑图。

  6. ==链路状态路由算法==

    1. 每个路由器向其邻居节点发送==hello问候分组==并了解邻居节点的网络地址（任两个 每隔10s）
    2. 设置到它每隔邻居的成本度量metric
    3. 构造==DD数据库描述分组==，向邻站给出自己的链路状态数据库中所有链路状态项目的摘要信息
    4. 如果DD分组中的摘要自己都有，则邻站不做处理，如果有没有的或是更新的，则发送==LSR链路状态请求分组==，请求自己没有的和更新的信息
    5. 收到邻站的LSR后，发送==LSU链路状态更新分组==进行更新/只要有一个路由器的链路状态发生变化，就洪泛发送LSU链路状态更新分组进行更新。
    6. 更新完毕后，邻站返回一个==LSAck链路状态确认分组==进行确认。/更新完毕后，其他站返回一个LSAck链路确认分组进行确认
    7. 使用Dijkstra根据自己的链路状态数据库构造到其他结点的最短路径

  7. OSPF的区域

    OSPF是一个网络层协议。

5.1 运输层协议概述

5.1.1 进程之间的通信

  1. 运输层的存在原因

    网络层已解决：分组从一台主机经过网络到达另一台主机，即主机到主机间的通信。

    网络层未解决：主机中的数据发送方和接收方、接收进程处理无需到达目的主机的IP分组。

  2. 运输层的作用

    从IP层来说，通信的两端是两台主机；从运输层来说，通信的真正断点并不是主机而是主机中的进程，端到端的通信是应用进程之间的通信。

    运行在主机A上的某个程序和运行在主机B上的另一个程序进行通信，运输层为相互通信的应用进程提供了逻辑通信。

    网络层是为主机之间提供逻辑通信；运输层为应用进程之间提供端到端的逻辑通信。

  3. 运输层的复用和分用

    在一台主机中经常有多个进程同时与另一台主机中的多个进程通信，而运输层会根据应用程序不同需求使用不同的协议即面向连接的TCP和无连接的UDP，因此运输层需要存在复用和分用功能。

  4. 屏蔽作用（异构计算机进程间的通信）

    运输层向高层用户屏蔽了下面网络核心的细节，使应用程序认为两个运输层实体之间有一条端到端的逻辑通信信道。

  5. 可靠信道与不可靠信道

    当运输层采用面向连接的TCP协议时，尽管下面的网络是不可靠的（尽最大努力服务） ，但这种逻辑通信信道就相当于一条全双工的可靠信道。

    当运输层采用无连接的UDP协议时，这种逻辑通信信道是一条不可靠信道。

5.1.2 运输层的两种协议

  1. 用户数据报协议UDP（User Datagram Protocol），传送数据单位为UDP报文或用户数据报。不需要先建立连接，提供无连接服务。收到UDP报文后，无需任何确认。UDP不提供可靠交付。简单、支持单播、多播、广播。适用于多媒体应用如域名解析服务、动态主机配置、路由选择等。

    IP数据报要经过互联网中许多路由器的存储转发，UDP用户数据报是在运输层的端到端抽象的逻辑信道中传送的。

  2. 传输控制协议TCP（Tranformation Control Protocol），传送数据单位是TCP报文段。不提供广播和多播服务（因为广播和面向连接是矛盾的）。收到TCP报文段后，需要确认。协议复杂、开销大、占用较多处理机资源。适用于万维网、电子邮件、文件传送等。

    TCP报文段是在运输层抽象的端到端逻辑信道中传送，这种信道是可靠的全双工信道。但这种信道不知道经过的哪些路由器，这些路由器也根本不知道运输层是否建立了TCP连接。

5.1.3 运输层的端口

  1. 端口的存在原因

    运行在计算机中的进程是用进程标识符来标志的，但运行在应用层的各种应用进程却不应当让计算机操作系统指派它的进程标识符（不同操作系统是用不同格式的进程标识符），为了使不同操作系统下的应用进程互相通信就必须用统一的方法对TCP/IP体系中的应用进程进行标志。

    通信的终点是应用进程，但可将端口想象为通信的终点，只把要传送的报文交到目的主机的某一个合适的目的端口，最后交付目的进程由TCP来完成。

  2. TCP/IP运输层端口

    端口用一个16位端口号标志，范围0-65535。

    端口号只具有本地意义，只是为了标志计算机应用层中的各进程，不同计算机的相同端口号无联系。

    两个计算机进程通信需要知道对方的IP地址（找到对方计算机），也需要知道对方的端口号（找到对方计算机中的应用进程）

  3. 两类端口

    1. 服务器端使用的端口号

      1. 熟知端口：0-1023

      2. 登记端口号：1024-49151，为没有熟知端口号的应用程序使用。使用此范围的端口号需要在IANA登记，以防止重复（未登记也可使用）。

    2. 客户端使用的端口号/短暂端口号：49152-65535，留给客户进程选择暂时使用。当服务器进程收到客户进程的报文时，就知道了客户进程所使用的动态端口号。通信结束后，此端口号可供其他客户进程以后使用。

5.2 用户数据报协议UDP

5.2.1 UDP概述

  1. 特点

    1. 无连接。发送数据之前不需要建立连接，因此减少了开销和发送数据之前的时延。

    2. 使用尽最大努力交付。不保证可靠交付，因此主机不需要维持复杂的连接状态表。

    3. 面向报文。UDP对应用层交下来的报文既不合并也不拆分，而是保留报文的边界。UDP一次交付一个完整的报文。发送方UDP对应用程序交下来的报文，添加首部后就向下交付IP层。接收方去除UDP报文首部，原封不动交付上层应用进程。

      注：若报文太长，UDP把它交给IP层后，IP层在传送时可能要进行分片，这会降低IP层的效率；若报文太短，UDP把它交给IP层后，会使IP数据报的首部的相对长度太大，也降低了IP层的效率。

    4. 无拥塞控制。网络中出现的拥塞不会使源主机的发送速率降低，适合实时应用、多媒体通信的要求。

    5. 支持一对一、一对多、多对一和多对多的交互通信。

    6. 首部开销小。只有8个字节，而TCP的首部20个字节。

    7. 在IP数据报的服务之上增加了复用和分用、差错检测功能。

5.2.2 UDP的首部格式

  1. 首部组成

    源端口2B、目的端口2B、长度2B、检验和2B

  2. 基于端口的分用

    当运输层从IP层收到UDP数据报时，根据首部中的目的端口把UDP数据报通过相应的端口，上交给最后的终点应用进程。

  3. 检验和

    在计算检验和时，临时把12字节的伪首部和UDP用户数据报连接在一起。伪首部仅用于计算检验和。

  4. 伪首部
    伪首部不是UDP真实首部，可以理解为UDP的两次检查：一次是对IP地址进行检验，确认该IP分组是发送给本机的；一次是对端口号和数据进行检验，确认交给哪个进程并且数据是无误的。

    发送方：

    1. 增加伪首部，UDP首部检验和填充为0
    2. 数据部分填充0至4字节整数倍
    3. 对伪首部+首部+数据计算检验和
    4. 首部填上检验和
    5. 删除伪首部，发送UDP数据报

    接收方：

    1. 增加伪首部
    2. 对伪首部+首部+数据计算检验和
    3. 检验和全1无差错，否则丢弃或上交应用进程并附上错误警告

5.3 传输控制协议TCP概述

5.3.1 TCP最主要的特点

  1. 面向连接的服务/通信模式回顾

    1. 面向连接的电路交换（物理层保证可靠）

      通信双方之间必须有一条物理连接的通路（直接相连），且被通信双方独享，数据按序发送并按序接收。

    2. 面向连接的虚电路（网络层保证可靠）

      通信双方采用复用技术，逐段占用物理通路，每段物理通路可悲多对通信使用，分组按序发送并按序接收。

    3. 面向连接的TCP（运输层协议保证可靠）

      采用协议的方法（确认、序号、重传），确保通信双方有一条全双工的、可靠的逻辑信道，字节按序发送并按序接收。（提供服务的IP数据报是可靠的且不一定按序到达）

  2. 特点

    1. 面向连接：只能有两个端点（点对点连接）；提供可靠交付服务；提供全双工通信。
    2. 面向字节流：虽然应用程序和TCP的交互是一次一个数据块，但TCP把应用程序交下来的数据看成仅仅是一串无结构的字节流（流入或流出进程的字节序列）。

  3. 面向流的概念

    TCP不保证接收方应用程序所收到的数据块和发送方应用程序所发出的数据块具有对应大小的关系，但接收方应用程序收到的字节流必须和发送方应用程序发出的字节流完全一样。

    TCP不关心应用进程一次把多长的报文发送到TCP缓存；TCP对连续的字节流进行分段，形成TCP报文段；TCP根据对方给出的窗口值和网络拥塞程度决定报文段应包含多少个字节；TCP可把长的数据块划分短一些再传送，TCP也可等待积累足够多的字节后再构成报文段发送出去。

5.3.2 TCP连接

  1. 套接字/插口：由端口号拼接到IP地址构成。TCP把连接作为最基本的抽象，每一条TCP连接有两个端点，TCP连接的端点即为套接字/插口。

    socket=(192.168.1.1:8080)

    TCP连接={socket1,socket2}

  2. TCP连接、IP地址与套接字

    TCP连接是由协议软件提供的一种抽象。

    同一个IP地址可以有多个不同的TCP连接。

    同一个端口号可以出现在多个不同的TCP连接中。

  3. UDP

    UDP没有使用套接字因为**UDP不是面向连接的。**

5.4 可靠传输的工作原理

5.4.0 前导

  1. IP网络提供的是不可靠的传输。
  2. 理想的传输条件的特点：传输信道不产生差错；不管发送方以多块的速度发送数据，接收方总是来得及处理收到的数据。

5.4.1 停止等待（每发送完一个分组就停止发送，等待对方确认，收到确认后再发送下一个分组）协议

  1. 无差错情况

  2. 出现差错

    1. 接收端B可能出现的情况

      1. B接收M1后检测到了差错就丢弃M1，无其他操作
      2. M1在传输过程中丢失，B不知情故无操作

    2. 保证B收到M1的方法——超时重传

      A为每个已发送的分组设置超时计时器，A在超时计时器到期之前收到相应的确认则撤销该超时计时器，继续发送下一个分组M2。

  3. ACK丢失和ACK迟到

    1. 确认丢失
      1. 若B发送的对M1的确认丢失，那么A在设定的超时重传时间内不能收到确认，但A无法明确是发出的分组出错或丢失还是B发送的确认直接丢失。因此A在超时计时器到期后就要重传M1。
      2. 假定B又收到重传的分组M1，则B首先丢弃这个重复的分组M1，不向上层交付，然后向A发送确认（A之所以重传M1就表示A没有收到B对M1的确认）
    2. 确认迟到
      1. 传输过程中没有出现差错，但B对分组M1的确认迟到
      2. A会收到重复的确认，对于重复确认收下后就丢弃
      3. B收到重复的M1并且同样丢弃重复的M1并重传确认分组

    在发送完一个分组后，必须暂时保留已发送的分组的副本以备重发。分组和确认分组都必须进行编号。超时计时器的重传时间应当比数据在分组传输的平均往返时间更长一些。

  4. 自动重传请求ARQ

    通常A最终总是可以收到对所有发出的分组的确认。如果A不断重传分组但总是收不到确认则说明通信线路太差不能进行通信。

    使用确认和重传机制就可以在不可靠的传输网络上实现可靠的通信。

    自动重传请求ARQ中重传的请求是自动进行的，接收方不需要请求发送方重传某个出错的分组。（如何实现自动重传？）

  5. $信道利用率=\frac{T_D}{T_D+RTT+T_A}$

    优点：简单；缺点：信道利用率太低

  6. 停止等待协议

    1. 停止等待。发送方每次只发送一个分组，在收到确认后发送下一个分组。
    2. 编号。对发送的每个分组和确认都进行编号。
    3. 自动重传请求。发送方为每个发送的分组设置一个超时计时器，若超时则发送方自动重传分组。

5.4.2 连续ARQ协议

  1. 流水线传输：发送方可连续发送多个分组，不必每发完一个分组就停顿下来等待对方确认，可以使信道上一直有数据不间断地传送，从而信道利用率较高。

  2. 基本思想

    1. 发送方一次可以发出多个分组
    2. 使用滑动窗口协议控制发送方和接收方所能发送和接收欧的分组的数量和编号
    3. 每收到一个确认，发送方就把发送窗口向前滑动
    4. 接收方一般采用累积确认的方式
    5. 采用回退N方法进行重传

  3. 发送方发送窗口：发送窗口内的分组都可连续发送出去，无需等待对方的确认。即发送方一次可发送多个分组，滑动窗口协议控制收发双方所能发送和接收的分组数据量及分组编号。发送方每收到一个确认就把发送窗口向前滑动一个分组的位置。

    接收方采用累积确认，对按序到达的最后一个分组发送确认。

    优点：容易实现，即使确认丢失也不必重传。

    缺点：不能向发送方反映出接收方已经正确收到的所有分组的信息。

  4. 回退N：表示需要再退回来重传已经发送过的N个分组。当通信线路质量不好时，连续ARQ协议会带来负面的影响。

  5. TCP可靠通信的具体实现

    TCP连接的每一端都有两个窗口：一个发送窗口和一个接收窗口。

    TCP的可靠传输机制用字节的序号进行控制，TCP所有的确认都是基于序号而不是基于报文段。

    TCP两端的四个窗口经常处于动态变化之中。TCP连接的往返时间RTT不是固定不变的，需要使用特定的算法估算较为合理的重传时间。

  6. 连续ARQ与停止等待协议总结

    对比内容 连续ARQ协议 停止等待协议
    发送的分组数量 一次发送多个分组 一次发送一个分组
    传输控制 滑动窗口协议 停止、等待
    确认 单独确认、累计确认 单独确认
    重传计时器 每个发送的分组 每个发送的分组
    序号 每个发送的分组 每个发送的分组
    重传 回退N，多个分组 一个分组

5.5 TCP报文段的首部格式

  1. TCP虽然是面向字节流（虽然应用程序和TCP的交互是一次一个数据块，但TCP把应用程序交下来的数据看成仅仅是一串无结构的字节流（流入或流出进程的字节序列））的但TCP传送的数据单元却是报文段。

    TCP报文段首部的前20个字节是固定的，后面还有4n字节是根据需要而增加的选项，因此首部最小长度是20字节。

  2. 首部格式

    1. 源端口和目的端口：各占2字节，是运输层和应用层的服务接口

    2. 序号：4字节，本报文段所发送的数据的第一个字节的序号

    3. 确认号：4字节，期望收到对方的下一个报文段中数据的第一个字节的序号

    4. 数据偏移（首部长度）：4位，TCP报文段的数据起始处距离TCP报文段的起始处的距离。**以32位字（4字节）为单位计算，最大60字节（15位——1111）。**

    6. 窗口：2字节，用来让对方设置发送窗口的依据，以字节为单位。明确指出现在允许对方发送的数据量，窗口值动态变化。

    7. 检验和：2字节，检验和字段检验的范围包括首部和数据两部分。**计算检验和时，要在TCP报文段前面加上12字节的伪首部。**

    8. 选项：长度可变。

      1. MSS：TCP最初只有一种选项即最大报文段长度MSS。

        MSS用于告知对方TCP本地TCP缓存所能接收报文段的==数据字段==最大长度是MSS个字节。

        规定MSS的原因：

        MSS太小，效率太低；MSS太大，IP层需要分片，只要有一片出错，TCP都需要重传。MSS应尽可能大些以便在IP层不分片。

      2. ==窗口扩大选项：3字节，其中有一个字节表示移位值S。新的窗口值等于TCP首部中的窗口位数增大到16+S，相当于把窗口值向左移动S位后获得的实际的窗口大小。==

      3. 时间戳选项：10字节，其中最主要的字段时间戳值字段4字节和时间戳回送回答字段4字节。

        ==用途：用于计算往返时间RTT；防止序号绕回。==

      4. 选择确认选项：告诉对方收到的连续的字节块。

    9. 填充字段：使整个首部长度是4字节的整数倍。

  3. UDP首部格式

    源端口2B、目的端口2B、长度2B、检验和2B。

  4. 为什么TCP首部不包含长度字段而UDP的首部包含长度字段？

    TCP data length=IP length-IP header length-TCP header length

    UDP data length=IP length-IP header length-UDP header length

    历史遗留问题，都不需要，UDP的首部长度字段是冗余的。

5.6 TCP可靠传输的实现

5.6.1 以字节为单位的滑动窗口

  1. TCP使用流水线传输和滑动窗口协议实现高效可靠的传输，滑动窗口以字节为单位、按字节编号。

    可靠传输的三个要素：序号、确认、重传。

    TCP连接的两端各有一个发送窗口和接收窗口，四个窗口动态变化。

    发送窗口：在没有收到确认的情况下，可以连续把窗口内的数据全部发送出去，准备发送的数据和已发送但未收到确认的数据。

    接收窗口：只允许接收落入窗口内的数据，按序到达但未被应用程序接受的数据、不按序到达的数据。

  2. 工作方式

    发送方可以将落入发送窗口的序号字节一次连续性全部发送出去并且边发送边接收确认。接收方的确认号为x，则构造新的发送窗口[x,x+win]。

    当发送窗口内的序号都已发送，若未收到确认则停止发送。

    未按序到达的数据应临时存放在接收窗口中，接收方必须有累积确认的功能从而减少传输开销。

  3. 接收方发送确认的时机

    接收方可以在合适的时候发送确认，也可以在自己有数据要发送时把确认信息捎带发送。

    接收方不应过分推迟发送确认，否则会导致发送方不必要的重传；捎带确认并不经常发生，因为大多数应用程序很少同时在两个方向上发送数据。

  4. 发送缓存

    发送方的应用进程把字节流写入TCP的发送缓存，发送窗口通常知识发送缓存的一部分。

    发送缓存用来暂时存放发送应用程序准备发送的数据和TCP已发送但未收到确认的数据。

  5. 接收缓存

    接收缓存用来暂时存放按序到达但尚未被接受应用程序读取的数据和不按序到达的数据。

5.6.2 超时重传时间的选择

  1. 机制

    TCP每发送一个报文段，就对这个报文段设置一次计时器。只要计时器设置的重传时间已到但尚未收到确认就要重传这一报文段。

  2. 考虑因素

    超时重传时间设置太短，会引起过多不必要的重传导致网络负荷增大。

    超时重传时间设置太长，网络的空闲时间增大，网络传输效率降低。

  3. 加权平均往返时间

    记录一个报文段发出的时间以及收到相应的确认的时间，时间差即为报文段的往返时间RTT。

    TCP使用RTT的一个加权平均往返时间$RTT_S$。

    $新的RTT_S=(1-\alpha)旧的RTT_S+\alpha新的RTT_S样本$，首次$RTT_S$为样本值，$\alpha$的推荐值为$\frac{1}{8}$

    $\alpha \to 0 :更新较慢\\alpha \to 1:更新较快$

  4. 超时重传时间RTO

    RTO应略大于加权平均往返时间$RTT_S$

    $RTO=RTT_S+4*RTT_D$

    $RTT_D$是$RTT$的偏差的加权平均值。首次测量$RTT_D$为$RTT$样本值的一半，此后

    $新的RTT_D=(1-\beta)旧的RTT_D+\beta|RTT_S-新的RTT样本|$，$\beta$的推荐值为$\frac{1}{4}$

  5. Karn算法

    只要报文发生重传就不采用其往返时间样本。

    当时延突然增大较多，一段时间内，TCP可能重传很多报文段。根据Karn算法，不考虑重传的报文段的往返时间样本的话，超时重传时间就无法更新。

    $新的RTO=\gamma*旧的RTO$，$\gamma$的典型值为2。当不再发生报文段的重传时，才根据报文段的往返时延更新平均往返时延$RTT$和超时重传时间$RTO$的数值。

5.6.3 选择确认SACK

5.8 TCP的拥塞控制

5.8.1 拥塞控制的一般原理

  1. 拥塞：在某段时间内，若对网络中某资源的需求超过了该资源所能提供的可用部分（链路容量不足、资源分配不均衡、路由器缓存空间、流量分布不均衡、处理机速度过慢），网络的性能就要变坏的现象。拥塞的最坏结果是系统崩溃。

    随着负载增大，延迟逐渐增大，吞吐量逐渐增大达到峰值，此后分组丢失增多，进而系统崩溃，系统崩溃后，延迟急剧增大。

  2. 增加资源不能解决拥塞

    增加资源在许多情况下不仅不能解决拥塞问题而且还可能使网络的性能更坏。

    增大缓存但未提高输出链路的容量和处理机的速度，排队等待的时间会大大增加，引起大量超时重传，解决不了网络拥塞；提高处理机处理的速率也会将瓶颈转移到其他地方。

  3. 拥塞常常趋于恶化

    原因：路由器没有足够的缓存空间则会丢弃一些新到的分组。分组被丢弃后，发送方会重传（可能重传多次），进而引起更多的分组流入网络和被网络中的路由器丢弃，从而加剧网络拥塞。

  4. 拥塞控制：防止过多的数据注入网络中，使网络中的路由器或链路不致过载。涉及所有主机、路由器以及与降低网络传输性能有关的所有因素。

    前提：网络能够承受现有网络负荷。

  5. 拥塞控制与流量控制

    流量控制：端到端（接收端控制发送端），点对点通信量的控制。抑制发送端发送数据的速率，以便使接收端来得及接收。

    拥塞控制：是一个全局性过程，涉及到与降低网络传输性能有关的所有因素。防止过多数据注入到网络，使网络中的路由器或链路不致过载。

    存在拥塞算法与流量控制类似：通过向发送端发送控制报文告知对方网络出现问题须放慢发送速率。

  6. 分组丢失：发生拥塞的征兆

    数据链路层：帧出错被丢弃

    网络层：出错IP分组被丢弃

  7. 开环控制方法：在设计网络时考虑发生拥塞的因素，力求网络不产生拥塞。

  8. 闭环控制方法：

    1. 监测网络系统以便检测到拥塞在何时、何处发生
    2. 将拥塞发生的信息传送到可采取行动的地方
    3. 调整网络系统的运行以解决出现的问题

  9. 监测网络拥塞的指标

    1. 由于缺少缓存空间而被丢弃的分组的百分数
    2. 平均队列长度
    3. 超时重传的分组数
    4. 平均分组时延
    5. 分组时延的标准差

  10. 拥塞通知的传递与时机

    发送通知拥塞发生的分组，在分组中保留保留表示拥塞状态的字段，周期性发出探测分组。

    时机选择：过于频繁会使系统产生不稳定的振荡；过于迟缓采取行动则不具有任何价值。

5.8.2 TCP的拥塞控制方法

  1. TCP采用基于窗口的方法进行拥塞控制（闭环控制方法），发送方维持一个拥塞窗口cwnd（congestion window），发送端利用拥塞窗口根据网络的拥塞情况调整发送的数据量。发送窗口不仅取决于接收方窗口还取决于网络的拥塞状况。

    真正的发送窗口值=Min(接收方窗口值，拥塞窗口值)

  2. 控制拥塞窗口的原则

    1. 只要网络没有出现拥塞，拥塞窗口即可增大以便发送更多分组来提高网络利用率。
    2. 只要网络出现拥塞或有可能出现拥塞就必须把拥塞窗口减小一些以减少注入网络中的分组数以便环节网络出现的拥塞。

  3. 拥塞的判断依据

    1. 重传定时器超时：因传输出差错而丢弃分组的概率极低因此只要出现超时可以猜想网络出现拥塞。
    2. 收到三个相同/重复的ACK：个别报文段在网络中丢失预示可能出现拥塞（实际尚未拥塞），尽快采取措施避免拥塞。

  4. TCP拥塞控制算法 AIMD算法

    讨论前提：单向传输数据，另一方向发送确认；接收方接收缓存无限大；发送方根据网络拥塞情况估算窗口值。

    1. 慢开始

      目的：用来确定网络的负载能力或拥塞程度。

      思路：由小到大逐渐增大拥塞窗口数值。

      两个变量：

      1. 拥塞窗口

        初始拥塞窗口值：2-4个报文段。

        窗口值逐渐增大。

      2. 慢开始门限：防止拥塞窗口增长过大引起网络阻塞。

      控制方法：在每收到一个对新的报文段的确认后，可以把拥塞窗口增加最多一个SMSS（最大报文段）的数值。

    2. 拥塞避免

      思路：让拥塞窗口cwnd缓慢增大，避免出现拥塞，每经过一个传输轮次拥塞窗口+1。线性增长、加法增大。

      无论慢开始阶段还是在拥塞避免阶段，只要发送方判断网络出现拥塞，则慢开始门限=max(cwnd/2,2)，cwnd=1，执行慢开始算法。

      目的：迅速减少主机发送到网络中的分组数，使得发生拥塞的路由器有足够时间把队列中积压的分组处理完毕。

    3. 快重传

      发送方只要一连收到三个重复确认就知道接收方未收到报文段，因此立即进行重传从而不会出现超时，发送方也不会误认为出现了网络拥塞。

      快重传并非取消重传计时器，而是在某些情况下可以更早地重传丢失的报文段。

      快重传算法首先要求接收方不要等待自己发送数据时才进行捎带确认，而是要立即发送确认，即使收到了失序的报文段也要立即发出对已收到的报文段的重复确认。

    4. 快恢复

      当发送端收到连续三个重复的确认时，由于发送方现在认为网络很可能没有发生拥塞，因此现在不执行慢开始算法，而是执行快恢复算法：

      1. 慢开始门限ssthresh=当前拥塞窗口cwnd/2
      2. 新拥塞窗口cwnd=慢开始门限ssthresh
      3. 开始执行拥塞避免算法，使拥塞窗口线性增大

  5. 总结

    1. 慢开始：每个传输轮次之后，拥塞窗口值乘2
    2. 拥塞避免：每收到一个确认，拥塞窗口加1
    3. 快重传：收到三个连续重复确认，立即重传丢失的报文段
    4. 快恢复：收到三个连续重复确认，门限值乘0.5，执行拥塞避免算法

5.8.3 主动队列管理AQM

  1. FIFO处理规则

    1. 路由器的队列通常是按照FIFO的规则处理到来的分组。
    2. 当队列已满时，以后再到达的所有分组都将被丢弃——尾部丢弃策略，如果能够继续排队则排在队列尾部。
    3. 路由器的尾部丢弃往往会导致一连串分组的丢失，使发送方出现超时重传，TCP进入拥塞控制的慢开始状态，结果使TCP连接的 发送方突然把数据的发送速率降低到很小的数值。

  2. 全局同步

    路由器同一方向来源的TCP连接中的分组被丢弃，这一方向上所有TCP连接同一时间进入慢开始状态后，通信量突然下降，正常后，通信量突然增大。（若路由器进行尾部丢弃，则无论属于哪个TCP连接，所有到达的分组都被丢弃）

  3. 主动队列管理：在队列长度达到某个值得警惕的数值时（网络已有拥塞征兆时）就主动丢弃到达的分组。

  4. 随机早期检测 RED
    使路由器的队列维持两个参数：队列长度最小门限$TH_{min}$和最大门限$Th_{max}$

    RED对每个到达的分组计算平均队列长度$L_{AV}$

    1. 若平均队列长度小于最小门限$TH_{min}$，则将新到达的分组放入队列进行排队；
    2. 若平均队列长度超过最大门限$Th_{max}$，则将新到达的分组丢弃；
    3. 瑞平均队列长度在最小门限$TH_{min}$和最大门限$Th_{max}$之间，则按照某一概率p将新到达的分组丢弃。

5.9 TCP的运输连接管理

TCP是面向连接的协议，其连接有三个阶段：连接建立、数据传送、连接释放，TCP连接管理的目的是使TCP连接的建立和释放都能正常地运行。

需要解决的问题：要使每一方能够确知对方的存在；要允许双方协商一些参数（如最大窗口值、是否使用窗口扩大选项和时间戳选项以及服务质量等）；能够对运输实体资源（如缓存大小、连接表中的项目等）进行分配。

TCP连接的建立采用客户——服务器方式，主动发起连接建立的应用进程为客户，被动等待连接建立的应用进程为服务器。

    5. B的TCP收到主机A的确认后，也通知其上层应用进程TCP连接已经建立

5.9.2 TCP的连接释放

  1. 过程

    1. 数据传输结束后，双方都可释放连接。A的应用进程先向其TCP发出连接释放报文段，并停止再发送数据，主动关闭TCP连接。A把连接释放报文段的首部FIN=1，序号seq=u发送到B，等待B的确认。
    2. B收到后发回确认，确认号ack=u+1，seq=v，TCP服务器进程通知高层应用进程。自此从A到B方向的连接被释放，TCP连接处于半关闭状态。B若发送数据，A仍要接收。
    3. 若B没有要向A发送的数据，其应用进程就通知TCP释放连接。FIN=1，ACK=1，seq=w，ack=u+1。
    4. A收到连接释放报文段后发回确认。确认号ack=w+1，序号seq=u+1。

  2. A为什么必须等到2MSL（最长报文段寿命）的时间后进入CLOSED？

    1. ==保证TCP协议的全双工连接能够可靠关闭。==

      如果客户端直接进入CLOSED状态，服务器有可能没有收到客户端最后恢复的ACK，服务器超时后继续发送FIN。但客户端已经进入CLOSED状态，服务器重发的FIN找不到对应的连接，服务器就会收到RST而不是ACK。

      这种情况不会造成数据丢失，但导致TCP协议不符合可靠连接的请求。客户端不是直接进入CLOSED状态而是要保持TIME_WAIT，当再次收到FIN时能够保证对方收到ACK最后正确地关闭连接。

    2. 保证此次连接的重复数据从网络中消失

      如果客户端直接进入CLOSED状态后又向服务器发起一个新连接，有可能新连接和老连接的端口号相同。

      如果前一次连接的某些数据仍然滞留在网络中，这些数据在建立新连接之后才到达服务器，TCP协议就认为这些数据是属于新连接的，从而和真正的新连接的数据包发生混淆。

      客户端在TIME_WAIT状态等待2MSL可以保证本次连接的所有数据都从网络中消失（客户端和服务器的连接已经释放，此期间客户端收到的TCP报文段直接丢弃）

  3. 保活计时器：用来防止在TCP连接中出现长时期的空闲。

    保活计时器通常设置为2小时。若服务器过了2小时还没有收到客户的信息则发送探测报文段。若发送了10个探测报文段（每个相隔75s）还没有相应就假定客户端出了故障，进而终止连接。

    顶级域名：国家顶级域名；通用顶级域名（com net org edu gov mil int）；基础结构域名arpa（用于反向域名解析，反向域名）

  5. 域名服务器

    区：一个服务器所负责管辖/有权限的范围叫做区。

    各单位根据具体情况划分自己管辖范围的区，但在一个区中的所有结点必须是能够连通的。

    每个区设置相应的权限域名服务器，用来保存该区中所有主机的域名到IP地址的映射。

    DNS服务器的管辖范围是以区为单位而不是域。

    分类：

    1. 根域名服务器：最高层次；最重要；所有的根域名服务器都知道所有的顶级域名服务器的域名和IP地址；==本地域名服务器若要对互联网上一个域名进行解析且自己无法解析时首先求助于根域名服务器==；13个。[a-m].rootserver.net

      DNS根域名服务器并非仅有13台而是对应13个IP，对应a-m13个编号，借由任播技术，编号相同的根服务器使用同一个IP。

      根域名服务器并不直接把域名转换为IP地址，在使用迭代查询时，根域名服务器把下一步应当找的顶级域名服务器的IP地址告诉本地域名服务器。

    2. 顶级域名服务器：TLD服务器负责管理在该顶级域名服务器注册的所有二级域名；在收到DNS查询请求时给出相应回答（最后结果/下一步应当查找的域名服务器的IP地址）

    3. 权限域名服务器：负责一个区的域名服务器；当一个权限域名服务器还不能给出最后的查询回答时就会告诉发出查询请求的DNS客户下一步应当找哪个权限域名服务器。

    4. 本地域名服务器：当一个主机发出DNS查询请求时，这个查询请求报文就发送给本地域名服务器；默认域名服务器。

    提高域名服务器的可靠性：

    1. DNS域名服务器把数据复制到几个域名服务器保存，其中一个是主域名服务器，其他是辅助域名服务器；
    2. 当主域名服务器出现故障时，辅助域名服务器用来保证DNS查询工作不中断；
    3. 主域名服务器定期把数据复制到辅助域名服务器中，而更改数据只能在主域名服务器中进行，保证了数据一致性。

    域名解析过程：（递归查询：请别人帮忙完成查询；迭代查询：自己主动完成查询）

    1. 主机向本地域名服务器的查询采用递归查询。如果本地域名服务器不能解析域名的IP地址，本地域名服务器就以DNS客户的身份向根域名服务器发出查询请求报文。
    2. 本地域名服务器向根域名服务器的查询采用迭代查询。当根域名服务器收到本地域名服务器的迭代查询请求报文时，要么给出所要查询的IP地址，要么告诉本地域名服务器下一步要查询的域名服务器，由本地域名服务器进行后续的查询。

    DNS服务器的高速缓存：每个域名服务器维护一个高速缓存用来存放最近用过的名字以及从何处获得名字映射信息的记录；大大减轻根域名服务器的负荷，使互联网上的DNS请求和响应报文大量减少；为保证缓存内容正确，域名服务器应为每项内容设置计时器并处理超过合理时间的项（超出规定存放期限）

    DNS客户名字高速缓存：当权限域名服务器回答一个查询请求时，在响应中指明绑定有效存在的时间值，增加此时间可减少网络开销，减少此时间可提高域名转换的准确性；==解决了客户频繁向服务器解析不存在的域名问题。==

    DNS查询过程：

    1. 本地查询：DNS缓存中查询；hosts文件中查询
    2. 向本地域名服务器查询：授权直接回答；DNS缓存回答
    3. 本地域名服务器向根服务器发起迭代查询

6.2 文件传送协议 FTP 文件传输

  1. 概述

    提供交互式访问；允许指明文件类型与格式；允许文件具有存取权限；屏蔽了计算机系统的细节，适合在异构网络中任意计算机之间传送文件。

    复杂->文件系统不同！

  2. FTP的基本原理

    1. 网络环境下复制文件的复杂性：计算机存储数据的格式不同；文件的目录结构和文件命名的规定不同；对于相同的文件存取功能，操作系统使用的命令不同；访问控制方法不同。

    2. 特点：

      1. FTP只提供文件传送的基本服务，使用TCP的可靠运输服务

      2. 减少/消除不同操作系统下处理文件的不兼容性

      3. 使用C/S方式，一个FTP服务器进程可同时为多个客户进程提供服务。

        一个主进程：负责接受新的请求

        若干从属进程：负责处理单个请求

    3. FTP两个连接和工作过程

      客户端使用任意分配的端口号（>1023），服务器使用熟知端口号21传输控制信息，使用熟知端口号20传输数据。

      服务器主程序打开21号端口等待客户进程发送连接请求，启动从属进程处理客户进程请求，此后回到等待状态继续接受其他客户进程请求

      主进程和从属进程的处理并发进行

      使用两个不同端口号的好处：使协议更加简单更容易实现，在传输文件时可以利用控制连接。

    4. 两个连接

      控制连接：在整个会话期间一直保持打开，FTP客户发出的传送请求通过控制连接发送给服务器端的控制进程，但控制连接不用来传送文件。

      数据连接：服务器端的控制进程在接收到FTP客户发送来的文件传输请求后就创建数据传送进程和数据连接，用来连接客户端和服务器端的数据传送进程。

      数据传送进程实际完成文件到的传送，在传送完毕后关闭数据传送连接并结束运行。

  3. 简单文件传送协议TFTP

    1. 概述

      很小且易于实现的文件传送协议；使用客户服务器方式和使用UDP数据报，因此TFTP需要有自己的差错改正措施；TFTP只支持文件传输而不支持交互；TFTP没有庞大的命令集，没有列目录的功能，也不能对用户进行身份鉴别。

    2. 特点

      每次传送的数据PDU有512B的数据，最后一次可以不足512B。

      数据PDU也称为文件块，每个块按序编号，从1开始。

      支持ASCII码或二进制传送。

      可对文件进行读或写。

      使用很简单的首部。

    3. 工作模式

      发送完一个文件块后就等待对方的确认，确认时应指明所确认的块编号，发完数据后在规定时间内收不到确认就要重发数据PDU，发送确认PDU的一方若在规定时间内收不到下一个文件块也要重发确认PDU，保证文件的传送不会因为某一个数据报的丢失而失败。

      开始工作时，TFTP客户进程发送一个读请求PDU或写请求PDU给TFTP服务器进程，其熟知端口号码为69。

      TFTP服务器进程要选择一个新的端口和TFTP客户进程进行通信。

      若文件长度恰好为512B的整数倍，则在文件发送完毕后还必须在最后发送一个只含首部而无数据的数据PDU。

      若文件长度不是512B的整数倍，则最后传送数据PDU的数据字段一定不满512B，这正好可作为文件结束的标志。

6.3 远程终端协议TELNET 远程登录

用户用TELNET就可在其所在地通过TCP连接注册到远地的另一个主机上（使用主机名或IP地址），TELNET能将用户的击键传到远地主机，同时也能将远地主机的输出通过TCP连接返回用户屏幕，这种服务是透明的，因为用户感觉到好像键盘和显示器是直接连在远地主机上。

TELNET采用客户服务器方式，在本地系统运行TELNET客户进程，而在远地主机则运行TELNET服务器进程。和FTP的情况相似，服务器中的主进程等待新的请求，并产生从属进程来处理每一个连接。

网络虚拟终端NVT格式：异构计算机字符编码方式不同，相互传送字符时转换成NVT格式的字符。客户软件把用户的击键和命令转换成NVT格式，并送交服务器。服务器软件把收到的数据和命令从NVT格式转换成远地系统所需的格式。

常见TELNET应用：连接一个远程网络设备以便获取信息或进行配置；连接一个大型计算机以使用其软硬件资源；连接一个在线论坛以便与其他用户进行交互或通信。

安全性：TELNET应用之间的信息交换是明码交换过程。

SSH与TELNET：

  1. 相同点

    远程登录其他台主机

    运输层采用TCP协议

  2. 不同点

    TELNET明文传送；SSH加密传送，且支持压缩

    TELNET服务默认端口号为23；SSH服务默认监听22号端口

    SSH使用对称加密算法实现数据安全传输

    对称加密算法的密钥通过非对称加密算法RSA进行交换

6.4 万维网WWW HTTP

  1. 概述

    1. 超媒体与超文本

      万维网是分布式超媒体系统，它是超文本系统的扩充。

      一个超文本由多个信息源链接而成。利用一个链接可使用户找到另一个文档，超文本是万维网的基础。

      超媒体与超文本的区别是文档内容不同。超文本文档仅包含文本信息，而超媒体文档害包含其他表示方式的信息如图形、图像、声音、动画甚至活动视频图像。

    2. 万维网的工作方式

      万维网以C/S方式工作。浏览器是用户计算机上的万维网客户程序，万维网文档所驻留的计算机则运行服务器程序，服务器程序向客户程序送回客户所需要的万维网文档。

      在一个客户程序主窗口上显示出的万维网文档称为页面。

    3. 需要解决的问题

      1. 标志分布在整个互联网上的万维网文档

        使用URL，每个文档在整个互联网的范围内有唯一的标识符URL。

      2. 实现超链链接的协议

        超文本传送协议HTTP，应用层协议，使用TCP连接进行可靠的传送。

      3. 万维网文档的显示及其超链的表示

        超文本标记语言HTML

      4. 用户找到所需信息

        搜索工具 搜索引擎

  2. 统一资源定位符URL

    1. URL的格式

      URL相当于一个文件名在网络范围内的扩展，是与互联网相连的机器上的任何可访问对象的一个指针。

      由以冒号隔开的两大部分组成，无大小写要求。

      <协议>://<主机>:<端口>/<路径>

      主机：从哪台目标主机上获取资源

      协议：采用什么协议获取目标主机的资源

      端口：从目标主机的哪个应用进程获取资源

      路径：资源在主机上存放的路径

  3. 超文本传送协议HTTP

    1. 操作过程

      HTTP是面向事务的应用层协议，是万维网上能够可靠地交换文件的重要基础。

      客户端发起请求、服务器回送响应。限制对HTTP协议的使用，无法实现在客户端没有发起请求的时候，服务器将消息推送给客户端。

      HTTP是一个无状态协议，同一个客户端的这次请求和上次请求没有对应关系。

    2. WWW的工作流程

      每个WWW服务器运行服务器进程，采用TCP协议监听TCP的80D端口，不断检测是否有浏览器向它发出连接建立请求。

      一旦监听到连接建立请求并建立了TCP链接之后，客户端浏览器就向万维网服务器发出浏览某个页面的请求，服务器接着就返回所请求的页面作为响应，最终释放TCP连接。

      在浏览器和服务器之间的请求和响应的交互必须按照规定的格式和遵循一定的规则HTTP。

      HTTP规定在HTTP客户和HTTP服务器之间的每次交互都由一个ASCII码串构成的请求和响应组成，HTTP报文通常使用TCP连接传送。

    3. HTTP的无连接性与无状态性

      无连接：限制每次连接只处理一个请求，服务器处理完客户的请求并收到客户的应答后即断开连接，从而节省传输时间。

      无连接性原因：早期客户端与服务器交换数据间歇性较大（传输具有突发性、瞬时性），网页浏览的联想性和发散性导致两次传送的数据关联性很低，大部分通道实际上会很空闲、无端占用资源。而现在网页很复杂每次访问图片都建立一次TCP连接很低效，由此提出Keep-Alive。

      无状态：协议对于事务处理没有记忆能力，服务器不知道客户端是什么状态。在向服务器发送HTTP请求之后，服务器根据请求发送数据，但发送完后不会记录任何信息。每个HTTP请求都是独立的，如果后续处理需要前面的信息则必须重传。

      无状态的优点和缺点：减轻了服务器负荷；每次请求可能会传输重复的内容。严重阻碍了客户端与服务器进行动态交互的web应用程序。由此提出cookie和session的概念。

    4. 在服务器上存放用户的信息

      万维网站点可以使用cookie来跟踪用户。

      cookie表示在HTTP服务器和客户之间传递的状态信息。

      使用cookie的网站服务器为用户产生一个唯一的识别码，利用此识别码网站能够跟踪该用户在该网站的活动。

      cookie会被附加在每个HTTP请求中，因此无形中增加了流量。

      cookie在HTTP中以明文形式传递，不够安全（可用HTTPS）。

      cookie大小限制在4KB左右，对于复杂的存储需求来说无法发满足。

    5. HTTP的事务

      一次HTTP操作称为一个事务，工作过程分为如下四步：

      1. 首先客户机与服务器建立连接，只要单击某个超链接，HTTP的工作即开始。
      2. 建立连接后，客户机发送一个请求给服务器。
      3. 服务器接到请求后，基于相应的响应信息。
      4. 客户端接收服务器所返回的信息通过浏览器显示在用户的显示屏上，然后客户机与服务器断开连接

    6. 浏览器访问某个网站的HTTP事务完整过程

      1. 客户发出http请求
      2. 服务器响应http请求
      3. 浏览器得到html代码
      4. 浏览器解析html代码并请求html代码中的资源
      5. 浏览器对页面进行渲染并呈现给用户

    7. 代理服务器 proxy server 又称为万维网高速缓存web cache 代表浏览器发出HTTP请求

      万维网高速缓存把最近的一些请求和响应暂存在本地磁盘中，当与暂时存放的请求相同的新请求到达时，万维网高速缓存就把暂存的响应发送出现而不需要按URL的地址去互联网访问该资源。

6.6 动态主机配置协议DHCP

  1. 协议配置：在协议软件中，给协议参数进行赋值的动作。

  2. 连接到互联网的计算机的协议软件需要配置的参数：IP地址、子网掩码、默认路由器IP地址、域名服务器IP地址

  3. DHCP提供了即插即用联网，允许一台计算机无需手工配置即可加入新的网络和获取IP地址；DHCP给运行服务器软件且位置固定的计算机指派一个永久地址，给运行客户端软件的计算机分配一个临时地址。

  4. DHCP的客户-服务器方式

    需要IP地址的主机在启动时就向DHCP服务器广播发送==发现报文DHCPDISCOVER==，此时该主机成为DHCP客户。

    本地网络上的所有主机都能收到此广播报文，但只有DHCP服务器才回答此广播报文。

    DHCP服务器先在其数据库中查找该计算机的配置信息，若找到则返回找到的信息；若找不到，则从==IP地址池==中取一个地址分配给该计算机。DHCP服务器的回答报文叫做==提供报文DHCPOFFER==。

  5. DHCP的工作方式

    DHCP使用客户-服务器方式，采用请求/应答方式工作。

    DHCP基于UDP工作，DHCP服务器运行在67号端口，DHCP客户端运行在68号端口。
